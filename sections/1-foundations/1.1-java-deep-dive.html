1.1Java Deep Dive

1.1.1 Key Concepts

Core

The JMM defines how threads interact through memory and what behaviors are allowed in concurrent execution. It specifies when changes made by one thread become visible to another.

  • Happens-Before Relationship - Guarantees that memory writes by one specific statement are visible to another. Thread.start() happens-before any action in the started thread.
  • Visibility - Without synchronization, Thread A's writes may never be seen by Thread B due to CPU caches and compiler reordering.
  • Atomicity - Read/write of reference variables and most primitives (except long/double) are atomic. Compound operations (i++) are NOT atomic.
  • Reordering - JIT compiler and CPU can reorder instructions for performance. Only constrained by happens-before rules.
// Double-Checked Locking - Requires volatile!
public class Singleton {
    private static volatile Singleton instance; // volatile prevents reordering

    public static Singleton getInstance() {
        if (instance == null) {                    // First check (no lock)
            synchronized (Singleton.class) {
                if (instance == null) {            // Second check (with lock)
                    instance = new Singleton();    // Without volatile, another thread
                }                                  // might see partially constructed object
            }
        }
        return instance;
    }
}

Trade-offs: volatile vs synchronized vs Atomic

synchronizedMultiple operations must be atomicMedium - thread blockingMutual exclusion + visibility + ordering

Interview Tip

When asked "why volatile in DCL?", explain: without volatile, the JIT can reorder instance = new Singleton() into (1) allocate memory, (2) assign reference, (3) call constructor. Another thread sees non-null reference to unconstructed object.

1.1.2 volatile - Visibility Without Locking

Critical

What it does: A volatile variable guarantees that any read always sees the most recent write by any thread. It inserts memory barriers (fences) that prevent CPU caches from serving stale values and prevent instruction reordering around the volatile access.

When to use:

  • Simple flags/signals between threads (e.g., shutdown flag, initialization complete)
  • One thread writes, multiple threads read (single-writer pattern)
  • Publishing an immutable object reference (the reference assignment is atomic)
  • Double-Checked Locking idiom (prevents reordering of object construction)

When NOT to use:

  • Compound operations like count++ (read-modify-write is NOT atomic even with volatile)
  • Check-then-act patterns like if (flag) { doSomething(); flag = false; }
  • When multiple variables must be updated together consistently
  • When you need mutual exclusion (only one thread should execute a section)

What goes wrong if you misuse it:

// BUG: volatile does NOT make count++ atomic!
private volatile int count = 0;

// Thread A and Thread B both call this:
public void increment() {
    count++;  // This is actually: temp = count; temp = temp + 1; count = temp;
    // Race condition: Both threads read count=5, both write count=6
    // Expected: 7, Actual: 6 (lost update!)
}

// FIX: Use AtomicInteger or synchronized
private AtomicInteger count = new AtomicInteger(0);
public void increment() {
    count.incrementAndGet(); // Atomic CAS operation - no lost updates
}
// CORRECT USE: volatile as a shutdown flag
public class WorkerThread extends Thread {
    private volatile boolean running = true; // visible across threads immediately

    public void run() {
        while (running) {   // Without volatile, JIT may cache 'running' in register
            doWork();       // and this loop runs forever even after shutdown() is called
        }
    }

    public void shutdown() {
        running = false;    // Immediately visible to the worker thread
    }
}

// CORRECT USE: volatile for safe publication of immutable object
private volatile Config config; // reference assignment is atomic

public void updateConfig(Config newConfig) {
    // newConfig is fully constructed (immutable) before assignment
    config = newConfig; // All threads immediately see the new config
}

public Config getConfig() {
    return config; // Always returns a fully-constructed Config, never a partial one
}

1.1.3 synchronized - Mutual Exclusion + Visibility

Critical

What it does: Acquires a monitor lock (intrinsic lock) on an object. Only one thread can hold the lock at a time. Provides three guarantees: (1) mutual exclusion - only one thread in the critical section, (2) visibility - all writes before unlock are visible after lock, (3) atomicity - the entire block appears as one operation to other threads.

When to use:

  • Compound operations that must be atomic (check-then-act, read-modify-write on multiple fields)
  • Guarding invariants that span multiple variables (e.g., transferring between two accounts)
  • When you need mutual exclusion (only one thread should modify state at a time)
  • Default choice when you need thread safety and don't have extreme performance requirements

When NOT to use:

  • Simple single-variable visibility (volatile is cheaper)
  • High-contention counters/accumulators (Atomic classes or LongAdder are faster)
  • When holding the lock for I/O or long-running operations (blocks all other threads)
  • When you need try-lock, timed-lock, or interruptible-lock (use ReentrantLock instead)

What goes wrong if you misuse it:

// BUG 1: Locking on different objects = no mutual exclusion!
public void transferWrong(Account from, Account to, int amount) {
    synchronized (from) {  // Lock on 'from'
        synchronized (to) { // Lock on 'to'
            from.balance -= amount;
            to.balance += amount;
        }
    }
    // DEADLOCK RISK: Thread1 locks A then waits for B
    //               Thread2 locks B then waits for A
}

// FIX: Always lock in consistent order (e.g., by account ID)
public void transferSafe(Account from, Account to, int amount) {
    Account first = from.id < to.id ? from : to;
    Account second = from.id < to.id ? to : from;
    synchronized (first) {
        synchronized (second) {
            from.balance -= amount;
            to.balance += amount;
        }
    }
}

// BUG 2: Synchronizing on wrong object
private final List<String> items = new ArrayList<>();
public void addItem(String item) {
    synchronized (item) {       // WRONG: locking on the parameter, not the shared state!
        items.add(item);        // Different 'item' strings = different locks = no protection
    }
}
// FIX: Lock on the shared mutable state
public void addItem(String item) {
    synchronized (items) {      // CORRECT: lock on the collection being protected
        items.add(item);
    }
}

// BUG 3: Holding lock during I/O (blocks all threads waiting for this lock)
public synchronized String fetchData(String url) {
    return httpClient.get(url);  // All other threads blocked for seconds during network call!
}
// FIX: Minimize lock scope
public String fetchData(String url) {
    String data = httpClient.get(url);  // No lock during I/O
    synchronized (this) {
        cache.put(url, data);           // Lock only for cache update
    }
    return data;
}

1.1.4 Atomic Classes - Lock-Free Thread Safety via CAS

Critical

What it does: Provides atomic operations on single variables using Compare-And-Swap (CAS) hardware instructions. No locks, no blocking - threads never wait for each other. If a CAS fails (another thread modified the value), the thread retries with the new value.

When to use:

  • Counters and accumulators under low-to-moderate contention
  • Lock-free algorithms (non-blocking stacks, queues)
  • Single-variable atomic update where you need atomicity but not mutual exclusion of a block
  • When you cannot afford thread blocking (real-time systems, latency-sensitive code)

When NOT to use:

  • High contention scenarios with many threads competing (CAS retries waste CPU cycles spinning)
  • When you need to update multiple variables atomically together (CAS is single-variable only)
  • When the computation in the retry loop is expensive
  • For high-contention counters, prefer LongAdder (shards the count across cells)

What goes wrong if you misuse it:

// BUG: Atomic individual operations but non-atomic combination
private AtomicInteger balance = new AtomicInteger(1000);
private AtomicInteger reserved = new AtomicInteger(0);

public boolean reserve(int amount) {
    if (balance.get() >= amount) {      // Step 1: Check
        balance.addAndGet(-amount);     // Step 2: Deduct
        reserved.addAndGet(amount);     // Step 3: Add to reserved
        return true;
    }
    // RACE: Between get() and addAndGet(), another thread may have deducted!
    // Result: balance goes negative - invariant violated
    return false;
}

// FIX: Use CAS loop for atomic check-and-update on single variable
public boolean reserveCAS(int amount) {
    while (true) {
        int current = balance.get();
        if (current < amount) return false;
        if (balance.compareAndSet(current, current - amount)) {
            reserved.addAndGet(amount);
            return true; // CAS succeeded - we atomically checked and deducted
        }
        // CAS failed - another thread modified balance, retry with fresh value
    }
}

// For multi-variable invariants, you MUST use synchronized or a single AtomicReference
// holding an immutable snapshot of both values:
record BalanceState(int available, int reserved) {}
private AtomicReference<BalanceState> state =
    new AtomicReference<>(new BalanceState(1000, 0));

public boolean reserveCorrect(int amount) {
    while (true) {
        BalanceState current = state.get();
        if (current.available() < amount) return false;
        BalanceState next = new BalanceState(
            current.available() - amount,
            current.reserved() + amount
        );
        if (state.compareAndSet(current, next)) return true;
        // Retry if another thread changed state concurrently
    }
}

1.1.5 CAS (Compare-And-Swap) - The Foundation of Lock-Free Programming

Critical

What is CAS? CAS is a single CPU instruction (e.g., CMPXCHG on x86) that atomically performs: "If the value at memory location X is still V (expected), then update it to N (new value) and return true. Otherwise, do nothing and return false." The entire check-and-update is one atomic hardware operation - no other thread can intervene between the check and the update.

// Pseudocode of what CAS does at the hardware level:
// This is ONE ATOMIC CPU instruction - cannot be interrupted
boolean compareAndSwap(memoryLocation, expectedValue, newValue) {
    if (memoryLocation.value == expectedValue) {
        memoryLocation.value = newValue;
        return true;   // Success: we updated the value
    }
    return false;      // Failure: someone else changed it, we did nothing
}

How Java exposes CAS:

// AtomicInteger internally uses Unsafe.compareAndSwapInt()
AtomicInteger counter = new AtomicInteger(0);

// compareAndSet(expected, update) - returns true if successful
counter.compareAndSet(0, 1);  // If counter is 0, set to 1 atomically

// Common CAS patterns in AtomicInteger:
counter.incrementAndGet();     // CAS loop: get current, try CAS(current, current+1)
counter.getAndUpdate(x -> x * 2);  // CAS loop with arbitrary function
counter.accumulateAndGet(5, Integer::max); // CAS loop: keep max of current and 5

CAS Loop Pattern (Optimistic Concurrency):

// Building a lock-free stack using CAS
public class LockFreeStack<T> {
    private AtomicReference<Node<T>> top = new AtomicReference<>(null);

    public void push(T value) {
        Node<T> newNode = new Node<>(value);
        while (true) {                          // CAS retry loop
            Node<T> currentTop = top.get();     // Read current state
            newNode.next = currentTop;           // Prepare new state
            if (top.compareAndSet(currentTop, newNode)) {
                return; // CAS succeeded - our push is committed
            }
            // CAS failed: another thread pushed/popped. Retry with fresh top.
        }
    }

    public T pop() {
        while (true) {
            Node<T> currentTop = top.get();
            if (currentTop == null) return null; // Stack is empty
            Node<T> newTop = currentTop.next;
            if (top.compareAndSet(currentTop, newTop)) {
                return currentTop.value; // Successfully popped
            }
            // Retry - another thread modified the stack
        }
    }

    private static class Node<T> {
        final T value;
        Node<T> next;
        Node(T value) { this.value = value; }
    }
}

ABA Problem with CAS:

// ABA Problem: CAS checks value equality, not identity of modifications
// Scenario:
// 1. Thread A reads value = "A"
// 2. Thread B changes value to "B", then back to "A"
// 3. Thread A's CAS succeeds (value is still "A") but the state changed underneath!

// This is dangerous in linked structures (popped node reused, CAS sees same pointer)

// FIX: AtomicStampedReference - adds a version stamp to detect intermediate changes
AtomicStampedReference<String> ref = new AtomicStampedReference<>("A", 0);

// Read value AND stamp together
int[] stampHolder = new int[1];
String current = ref.get(stampHolder);  // current="A", stampHolder[0]=0

// CAS checks BOTH value AND stamp - detects ABA
boolean success = ref.compareAndSet(
    "A",              // expected value
    "C",              // new value
    stampHolder[0],   // expected stamp (0)
    stampHolder[0] + 1 // new stamp (1) - always increment
);
// If another thread changed A->B->A, the stamp would be 2, not 0 -> CAS fails correctly!

When CAS degrades - High Contention:

// Under HIGH contention (many threads), CAS loops burn CPU retrying
// Example: 64 threads all incrementing same AtomicInteger
// Most CAS attempts fail -> threads spin -> wasted CPU cycles -> poor scalability

// Solution: LongAdder (Java 8+) - shards the counter across CPU cache lines
import java.util.concurrent.atomic.LongAdder;

LongAdder adder = new LongAdder();       // Each thread updates its own "cell"
adder.increment();                        // Low contention per cell
long total = adder.sum();                 // Sum all cells (slightly stale under writes)

// LongAdder vs AtomicLong under contention:
// 4 threads:   LongAdder ~1.5x faster than AtomicLong
// 16 threads:  LongAdder ~4x faster
// 64 threads:  LongAdder ~10-15x faster (AtomicLong barely scales)
// Use AtomicLong when you need exact real-time value (e.g., sequence generator)
// Use LongAdder when you need fast aggregation (e.g., metrics counters)

Interview Tip

When discussing CAS in interviews, always mention: (1) It's a single CPU instruction - truly atomic at hardware level, (2) The retry loop pattern for building higher-level atomic operations, (3) ABA problem and how AtomicStampedReference solves it, (4) Why it degrades under high contention (cache-line bouncing, failed retries), and (5) LongAdder as the solution for high-contention counters. Show you understand the full spectrum from hardware to design trade-offs.

1.1.7 G1GC Internals

Performance

G1 divides heap into equal-sized regions (~1-32MB). Tracks garbage-first: collects regions with most garbage first for maximum reclamation with minimum pause.

// Production G1GC Tuning for 16GB heap microservice
-XX:+UseG1GC
-Xms16g -Xmx16g                          // Fixed heap prevents resize pauses
-XX:MaxGCPauseMillis=100                  // Target 100ms pause
-XX:G1HeapRegionSize=16m                  // Larger regions for big heap
-XX:InitiatingHeapOccupancyPercent=35     // Start concurrent cycle early
-XX:G1MixedGCCountTarget=4               // Spread mixed GC work
-XX:+ParallelRefProcEnabled               // Parallel reference processing
-Xlog:gc*:file=gc.log:time,level,tags     // Structured GC logging

1.1.8 ZGC - Sub-millisecond Pauses

Performance

ZGC uses colored pointers (metadata in pointer bits) and load barriers to achieve concurrent relocation without stop-the-world pauses proportional to heap size.

// ZGC for latency-critical payment service
-XX:+UseZGC
-XX:+ZGenerational                         // Java 21+: generational ZGC
-Xms32g -Xmx32g
-XX:SoftMaxHeapSize=28g                    // Prefer staying under this
-XX:ZCollectionInterval=5                  // Force GC every 5s if idle

Trade-off: G1GC vs ZGC

  • Choose G1GC when: General microservice, 100-200ms pauses acceptable, want predictable throughput, heap < 32GB
  • Choose ZGC when: Latency SLA < 10ms p99, large heaps (64GB+), trading/real-time systems, can sacrifice 5-15% throughput
  • Choose Parallel GC when: Batch jobs, offline processing, throughput matters more than latency

1.1.9 Sealed Classes - Controlled Inheritance

LTS
// Sealed classes restrict which classes can extend them
// Enables exhaustive pattern matching - compiler knows all subtypes
public sealed interface PaymentResult
    permits PaymentSuccess, PaymentFailure, PaymentPending {
}

public record PaymentSuccess(String transactionId, BigDecimal amount)
    implements PaymentResult {}

public record PaymentFailure(String errorCode, String reason)
    implements PaymentResult {}

public record PaymentPending(String trackingId, Duration estimatedTime)
    implements PaymentResult {}

// Compiler ensures exhaustive handling - no default needed
public String handleResult(PaymentResult result) {
    return switch (result) {
        case PaymentSuccess s -> "Paid: " + s.transactionId();
        case PaymentFailure f -> "Failed: " + f.reason();
        case PaymentPending p -> "Pending: " + p.trackingId();
    };
}

1.1.11 Virtual Threads (Project Loom)

Latest LTS

Virtual threads are lightweight threads managed by the JVM, not OS. Cost: ~1KB stack (vs 1MB for platform threads). Can run millions concurrently.

// Before: Thread pool limits concurrency to ~200 platform threads
ExecutorService traditional = Executors.newFixedThreadPool(200);

// After: Virtual threads - one per request, millions possible
ExecutorService virtual = Executors.newVirtualThreadPerTaskExecutor();

// Spring Boot 3.2+ - just enable in config:
// spring.threads.virtual.enabled=true

// Production HTTP client with virtual threads
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    List<Future<Response>> futures = urls.stream()
        .map(url -> executor.submit(() -> httpClient.send(
            HttpRequest.newBuilder(URI.create(url)).build(),
            HttpResponse.BodyHandlers.ofString()
        )))
        .toList();

    // All requests run concurrently on virtual threads
    // 10,000 concurrent HTTP calls with minimal memory
    List<Response> responses = futures.stream()
        .map(f -> { try { return f.get(); } catch (Exception e) { throw new RuntimeException(e); }})
        .toList();
}

1.1.12 Structured Concurrency (Preview)

// Structured concurrency: child tasks are scoped to parent lifetime
// If one fails, others are cancelled automatically
Response fetchOrder(String orderId) throws Exception {
    try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
        Subtask<Order> order = scope.fork(() -> orderService.get(orderId));
        Subtask<Payment> payment = scope.fork(() -> paymentService.get(orderId));
        Subtask<Shipping> shipping = scope.fork(() -> shippingService.get(orderId));

    }
    // All subtasks guaranteed complete/cancelled when scope exits
}

private static final ScopedValue<UserContext> CURRENT_USER = ScopedValue.newInstance();

void handleRequest(HttpRequest request) {
    UserContext ctx = authenticate(request);

1.1.17 CompletableFuture Composition

Critical
// Parallel aggregation pattern - common in microservices
CompletableFuture<OrderSummary> buildSummary(String orderId) {
    var orderFuture = CompletableFuture.supplyAsync(
        () -> orderService.getOrder(orderId), ioExecutor);
    var paymentFuture = CompletableFuture.supplyAsync(
        () -> paymentService.getPayment(orderId), ioExecutor);
    var inventoryFuture = CompletableFuture.supplyAsync(
        () -> inventoryService.check(orderId), ioExecutor);

    return orderFuture.thenCombine(paymentFuture, (order, payment) ->
        new OrderWithPayment(order, payment)

}

1.1.18 Lock Comparison

Critical
Lock TypeUse CasePerformanceFairness
synchronizedSimple mutual exclusionGood (biased locking)Unfair
ReentrantLockNeed tryLock, timeout, conditionGoodConfigurable
ReadWriteLockRead-heavy, rare writesGood for readsConfigurable
StampedLockOptimistic reads, highest throughputBest for read-heavyUnfair
// StampedLock - optimistic read pattern (no blocking for readers)
private final StampedLock lock = new StampedLock();
private double x, y;

double distanceFromOrigin() {
    long stamp = lock.tryOptimisticRead(); // Non-blocking!
    double currentX = x, currentY = y;     // Read fields
    if (!lock.validate(stamp)) {           // Check if write occurred
        stamp = lock.readLock();           // Fallback to read lock
        try { currentX = x; currentY = y; }
        finally { lock.unlockRead(stamp); }
    }
    return Math.sqrt(currentX * currentX + currentY * currentY);
}

Trade-off: When to Use What

  • synchronized: Default choice. Simple, JVM optimizes well (biased/thin/fat locking). Use unless you need specific features.
  • ReentrantLock: Need tryLock() with timeout (deadlock avoidance), multiple Conditions, or interruptible waiting.
  • StampedLock: Read-heavy workloads (95%+ reads). 2-3x faster than ReadWriteLock for reads. NOT reentrant - careful with nested calls.
