1.2Spring Boot & Reactive Programming

1.2.1 Auto-Configuration Mechanism

Core

Spring Boot reads META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports to discover configuration classes. Each uses @Conditional annotations to activate only when classpath/property conditions are met.

// Custom auto-configuration for a caching library
@AutoConfiguration
@ConditionalOnClass(CacheManager.class)           // Only if class on classpath
@ConditionalOnProperty(prefix = "app.cache", name = "enabled", havingValue = "true")
@EnableConfigurationProperties(CacheProperties.class)
public class CustomCacheAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean  // User can override with their own bean
    public CacheManager cacheManager(CacheProperties props) {
        return new RedisCacheManager(props.getTtl(), props.getMaxSize());
    }
}

// Order matters: load after Spring's own cache config
// In META-INF/spring/AutoConfiguration.imports:
// com.mylib.CustomCacheAutoConfiguration

1.2.3 N+1 Problem & Solutions

Performance
// PROBLEM: N+1 queries - fetches 1 order + N queries for items
@Entity
public class Order {
    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    private List<OrderItem> items; // Each access triggers separate query
}

// SOLUTION 1: JOIN FETCH in JPQL
@Query("SELECT o FROM Order o JOIN FETCH o.items WHERE o.customerId = :id")
List<Order> findByCustomerIdWithItems(@Param("id") String customerId);

// SOLUTION 2: @EntityGraph (declarative)
@EntityGraph(attributePaths = {"items", "items.product"})
@Query("SELECT o FROM Order o WHERE o.status = :status")
List<Order> findByStatusWithGraph(@Param("status") OrderStatus status);

// SOLUTION 3: Batch fetching (global config)
// application.yml: spring.jpa.properties.hibernate.default_batch_fetch_size=25
// Fetches items in batches of 25 instead of one-by-one

1.2.4 HikariCP Connection Pool Tuning

Performance
# Production HikariCP config for 8-core machine
spring:
  datasource:
    hikari:
      maximum-pool-size: 20        # Rule: connections = (2 * CPU cores) + effective_spindle_count
      minimum-idle: 5               # Keep some warm connections
      connection-timeout: 3000      # Fail fast - 3 seconds
      idle-timeout: 600000          # 10 min idle before eviction
      max-lifetime: 1800000         # 30 min max - before DB kills it
      leak-detection-threshold: 30000  # Log warning if connection held > 30s
      pool-name: OrderServicePool
      # Critical: statement caching
      data-source-properties:
        cachePrepStmts: true
        prepStmtCacheSize: 250
        prepStmtCacheSqlLimit: 2048

Trade-off: Eager vs Lazy Loading

  • Lazy (default for @OneToMany): Loads on access. Risk: LazyInitializationException outside transaction. Risk: N+1 if accessed in loop.
  • Eager (default for @ManyToOne): Always loaded. Risk: Loading entire object graph. Cartesian product with multiple eager collections.
  • Best Practice: Keep everything LAZY + use JOIN FETCH / @EntityGraph for specific use cases. Never use EAGER in entities.

Spring Security - JWT & OAuth2

Security
// Modern Spring Security 6.x config (lambda DSL)
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .csrf(csrf -> csrf.disable())  // Disable for stateless APIs
            .sessionManagement(session ->
                session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .requestMatchers("/actuator/health").permitAll()
                .anyRequest().authenticated()
            )
            .oauth2ResourceServer(oauth2 ->
                oauth2.jwt(jwt -> jwt.jwtAuthenticationConverter(jwtConverter()))
            )
            .exceptionHandling(ex -> ex
                .authenticationEntryPoint(new CustomAuthEntryPoint())
            )
            .build();
    }

    // Extract roles from JWT claims
    private JwtAuthenticationConverter jwtConverter() {
        JwtGrantedAuthoritiesConverter conv = new JwtGrantedAuthoritiesConverter();
        conv.setAuthoritiesClaimName("roles");
        conv.setAuthorityPrefix("ROLE_");
        JwtAuthenticationConverter jwtConv = new JwtAuthenticationConverter();
        jwtConv.setJwtGrantedAuthoritiesConverter(conv);
        return jwtConv;
    }
}

1.2.5 Reactive Streams Core Types

Advanced
// Mono = 0 or 1 element | Flux = 0 to N elements
// Key operators for production:

// 1. Parallel processing with concurrency control
Flux.fromIterable(orderIds)
    .flatMap(id -> orderService.enrich(id), 10)  // Max 10 concurrent
    .onErrorContinue((err, obj) ->               // Skip failures, continue
        log.warn("Failed to enrich {}: {}", obj, err.getMessage()))
    .collectList()
    .block();

// 2. Retry with exponential backoff
webClient.get()
    .uri("/api/inventory/{id}", productId)
    .retrieve()
    .bodyToMono(Inventory.class)
    .retryWhen(Retry.backoff(3, Duration.ofMillis(100))
        .maxBackoff(Duration.ofSeconds(2))
        .filter(ex -> ex instanceof WebClientResponseException.ServiceUnavailable))
    .timeout(Duration.ofSeconds(5))
    .onErrorReturn(Inventory.empty());

// 3. Caching with TTL
Mono<UserProfile> cachedProfile = userService.getProfile(userId)
    .cache(Duration.ofMinutes(5));  // Cache result for 5 minutes

1.2.6 Backpressure Strategies

Advanced
// Producer faster than consumer - must handle backpressure
Flux.create(sink -> {
    kafkaConsumer.subscribe(records -> sink.next(records));
}, FluxSink.OverflowStrategy.BUFFER)     // Options: BUFFER, DROP, LATEST, ERROR
    .onBackpressureBuffer(1000,            // Buffer up to 1000
        dropped -> metrics.increment("dropped.events"),  // Overflow handler
        BufferOverflowStrategy.DROP_OLDEST)
    .publishOn(Schedulers.boundedElastic()) // Process on separate scheduler
    .subscribe(this::processRecord);

Trade-off: Spring MVC (Virtual Threads) vs WebFlux

  • Spring MVC + Virtual Threads (Java 21+): Simple blocking code, familiar debugging, works with all JDBC drivers, good for request-response patterns
  • WebFlux: True streaming (SSE, WebSocket), built-in backpressure, non-blocking from top to bottom, better for event-driven architectures
  • Recommendation for 2024+: Default to MVC + Virtual Threads. Use WebFlux only when you need streaming semantics or are already invested in reactive ecosystem (R2DBC, reactive Redis/Kafka)

1.2.8 Resilience4j Circuit Breaker

Architecture
// Circuit breaker config
resilience4j:
  circuitbreaker:
    instances:
      paymentService:
        slidingWindowSize: 10
        failureRateThreshold: 50          # Open after 50% failures
        waitDurationInOpenState: 10s      # Wait before half-open
        permittedNumberOfCallsInHalfOpenState: 3
        slowCallRateThreshold: 80
        slowCallDurationThreshold: 2s

// Usage with annotations
@CircuitBreaker(name = "paymentService", fallbackMethod = "paymentFallback")
@TimeLimiter(name = "paymentService")
@Retry(name = "paymentService")
public CompletableFuture<PaymentResponse> processPayment(PaymentRequest req) {
    return CompletableFuture.supplyAsync(() -> paymentClient.charge(req));
}

public CompletableFuture<PaymentResponse> paymentFallback(PaymentRequest req, Throwable t) {
    // Queue for retry, return pending status
    retryQueue.publish(req);
    return CompletableFuture.completedFuture(PaymentResponse.pending(req.orderId()));
}

Production Insight

Always order annotations: @Retry(outermost) -> @CircuitBreaker -> @TimeLimiter(innermost). Retry wraps the circuit breaker - retries only happen when circuit is CLOSED. If circuit is OPEN, fail immediately without retrying.
