Extended Patterns (Priority 2)

Important supporting concepts that complement the core patterns above.

Fundamentals

CAP Theorem

A distributed system can provide at most 2 of 3 guarantees: Consistency, Availability, Partition tolerance. Since partitions are inevitable, you choose CP or AP.

When relevant: Any distributed database choice (CP: MongoDB, HBase | AP: Cassandra, DynamoDB)

Trade-off: CP sacrifices availability during partitions; AP sacrifices consistency (stale reads)

See detailed section →
Fundamentals

PACELC Theorem

Extension of CAP: if Partition, choose A or C; Else (normal operation), choose Latency or Consistency. Covers the common case when there's no partition.

When relevant: Explains why Cassandra (PA/EL) is fast but eventually consistent, while Spanner (PC/EC) is consistent but slower

Trade-off: Systems that choose consistency (PC/EC) always pay latency cost, even without partitions

See detailed section →
Observability

Three Pillars of Observability

Logs (what happened), Metrics (how much/how fast), Traces (request path across services). Together they provide full system visibility.

When relevant: Every production system needs all three; debugging distributed failures requires correlated traces

Trade-off: Full observability is expensive (storage, processing); balance detail vs cost with sampling

See detailed section →
Observability

SLI / SLO / SLA

SLI = measurable indicator (p99 latency). SLO = target for the SLI (p99 < 200ms). SLA = contractual commitment with consequences if SLO is missed.

When relevant: Defining service reliability targets, error budgets, alerting thresholds

Trade-off: Tighter SLOs = more engineering investment in reliability; loose SLOs = higher error budget for feature velocity

See detailed section →
Distributed

Distributed Locking

Coordinate access to shared resources across nodes using locks backed by a distributed store (Redis SETNX, ZooKeeper ephemeral nodes, etcd leases).

When relevant: Preventing double-processing, leader election, distributed cron, resource reservation

Trade-off: Adds latency and a dependency; lock holder crashing requires TTL/fencing tokens to avoid deadlocks

See detailed section →
Reliability

Chaos Engineering

Intentionally inject failures (kill nodes, add latency, corrupt data) in production to discover weaknesses before they cause real outages.

When relevant: Validating resilience patterns actually work under real conditions (Netflix Chaos Monkey)

Trade-off: Risk of actual outage during experiments; requires mature monitoring and blast radius control

See detailed section →
Consistency

Outbox Pattern

Write events to an "outbox" table in the same database transaction as the business data. A separate process reads the outbox and publishes to the message broker.

When relevant: Reliable event publishing without distributed transactions (dual-write problem)

Trade-off: Guaranteed delivery but adds latency (async publish) and requires outbox polling/CDC

See detailed section →
Architecture

Service Discovery

Mechanism for services to find each other's network locations dynamically, without hardcoded addresses. Client-side (Eureka) or server-side (Kubernetes DNS).

When relevant: Microservices with dynamic scaling (IPs change), container orchestration

Trade-off: Adds infrastructure complexity; stale registry = routing to dead instances

See detailed section →
Caching

Cache Stampede Prevention

When a hot cache key expires, thousands of requests simultaneously hit the database. Prevent with request coalescing (singleflight), probabilistic early refresh, or stale-while-revalidate.

When relevant: High-traffic systems with popular cache keys (product pages, user profiles)

Trade-off: Coalescing adds lock coordination; early refresh wastes some DB reads; stale serves outdated data briefly

See detailed section →
Communication

Backpressure

Mechanism to prevent a fast producer from overwhelming a slow consumer. Strategies: drop, buffer, throttle producer, sample, or scale consumers.

When relevant: Streaming pipelines, reactive systems, any producer-consumer with rate mismatch

Trade-off: Dropping loses data; buffering risks OOM; throttling slows producers; scaling costs money

See detailed section →
Communication

Long Polling / SSE

Long Polling: server holds request until data available. SSE: persistent one-way server-to-client stream over HTTP. Both simpler than WebSocket for push-only scenarios.

When relevant: Notifications, live feeds, dashboards (SSE); compatibility-constrained environments (Long Polling)

Trade-off: SSE is unidirectional only; Long Polling has connection overhead per message

See detailed section →
Reliability

Failover Patterns

Automatically redirect traffic to standby when primary fails. Active-Passive (hot standby, seconds), Active-Active (both serve, near-zero), Pilot Light (minimal infra, minutes).

When relevant: High availability requirements, multi-region deployments, disaster recovery

Trade-off: Active-Active = zero downtime but 2x cost; Pilot Light = cheap but slow recovery

See detailed section →
Distributed

Leader Election

Elect one node as coordinator in a cluster. If leader fails, remaining nodes hold new election. Implemented via Raft consensus, ZooKeeper ephemeral nodes, or etcd leases.

When relevant: Distributed schedulers, single-writer databases, coordinator selection

Trade-off: Simplifies coordination but leader is SPOF; election period = brief unavailability

See detailed section →
Distributed

Split Brain

Network partition divides cluster into groups, each electing its own leader — causing conflicting writes. Prevented by quorum requirement (majority needed to accept writes).

When relevant: Any replicated system with leader election, distributed databases

Trade-off: Quorum prevents split brain but minority partition becomes unavailable (sacrifices A for C)

See detailed section →
Architecture

ETL Pipeline

Extract data from sources, Transform (clean, aggregate, join), Load into destination (warehouse/lake). Modern variant ELT loads raw data first, transforms in-place.

When relevant: Analytics pipelines, data warehousing, business intelligence, ML feature stores

Trade-off: ETL = cleaner destination but slow; ELT = faster ingest but warehouse must handle transforms
