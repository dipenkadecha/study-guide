2.6Concurrency-Focused Data Structures

2.6.1 Producer-Consumer with BlockingQueue

Production
// Production pattern: bounded queue with backpressure
BlockingQueue<Event> queue = new ArrayBlockingQueue<>(10000);

// Producer thread
executor.submit(() -> {
    while (running) {
        Event event = receiveFromKafka();
        boolean offered = queue.offer(event, 100, TimeUnit.MILLISECONDS);
        if (!offered) {
            metrics.increment("queue.full.dropped");
            deadLetterQueue.send(event);  // Don't lose data
        }
    }
});

// Consumer threads (multiple)
for (int i = 0; i < numConsumers; i++) {
    executor.submit(() -> {
        while (running) {
            Event event = queue.poll(500, TimeUnit.MILLISECONDS);
            if (event != null) process(event);
        }
    });
}

2.6.2 LMAX Disruptor - Ultra-Low Latency

Production
// Disruptor: lock-free ring buffer, ~100ns latency (vs ~1000ns BlockingQueue)
// Used by: LMAX exchange (6M orders/sec), Apache Storm, Apache Log4j2
// Key insight: pre-allocated ring buffer, no GC, mechanical sympathy (cache lines)

// When to use Disruptor vs BlockingQueue:
// Disruptor: latency < 1μs needed, single producer, high throughput (>1M/sec)
// BlockingQueue: simpler API, multiple producers OK, latency < 1ms acceptable

2.6.3 CopyOnWriteArrayList

Production
// Snapshot-on-write: every write creates a new copy of the array
// Perfect for: event listener lists, configuration caches, read-heavy registries
List<EventListener> listeners = new CopyOnWriteArrayList<>();

// Reads: no locking, iterate over snapshot - O(1) to start
// Writes: O(n) copy - acceptable when writes are rare (< 1% of operations)
// Thread-safe iteration - never throws ConcurrentModificationException
