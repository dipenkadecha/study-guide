4.5Database Design

4.5.1 Database Selection Guide

Database
TypeExamplesData ModelBest ForAvoid When
RelationalPostgreSQL, MySQLTables, SQL, ACIDComplex queries, transactions, joinsMassive horizontal scale needed
Key-ValueRedis, DynamoDBKey -> ValueCaching, sessions, leaderboardsComplex queries, relationships
DocumentMongoDB, FirestoreJSON documentsFlexible schema, content managementMany-to-many relationships
Column-FamilyCassandra, HBaseWide columnsTime-series, write-heavy, large scaleComplex queries, joins
GraphNeo4j, NeptuneNodes + edgesSocial networks, fraud detectionSimple CRUD, large scans
Time-SeriesInfluxDB, TimescaleDBTimestamp-keyedMetrics, IoT, monitoringGeneral-purpose queries

Trade-off: ACID vs BASE

  • ACID (SQL): Atomicity, Consistency, Isolation, Durability. Strong guarantees but limits horizontal scaling. Use for: payments, inventory, booking.
  • BASE (NoSQL): Basically Available, Soft-state, Eventually consistent. Better scalability/availability but weaker guarantees. Use for: social feeds, analytics, logs.
  • Hybrid approach: Use ACID for critical path (payment) + BASE for eventual (notifications, analytics). Most production systems mix both.

Production Example

Discord: Started with PostgreSQL, hit scaling limits at 1 billion messages. Migrated to Cassandra for messages (write-heavy, partitioned by channel_id + time bucket). Kept PostgreSQL for user accounts (strong consistency needed for auth).

Key Tip

Always justify your database choice: "I'm choosing PostgreSQL for the user service because we need ACID transactions for account operations. For the activity feed, I'd use Cassandra — write-heavy, partitioned by user_id, eventual consistency is acceptable for social feeds." Never say "just use MySQL" without reasoning.

4.5.2 Indexing Internals

Database
// B+Tree Index (PostgreSQL, MySQL InnoDB default)
// Leaf nodes contain all keys in sorted order + pointers to rows
// Good for: range queries, ORDER BY, equality
CREATE INDEX idx_orders_status_date ON orders(status, created_at DESC);
// Composite index: leftmost prefix rule - can use (status), (status, created_at)
// Cannot use just (created_at) with this index!

// Hash Index: O(1) equality only, no range queries
// GIN Index: full-text search, array contains, JSONB queries
CREATE INDEX idx_products_tags ON products USING GIN(tags);

// Partial Index: index subset of rows (saves space + speeds writes)
CREATE INDEX idx_active_orders ON orders(customer_id) WHERE status = 'ACTIVE';

// Covering Index: includes all columns needed (index-only scan)
CREATE INDEX idx_orders_cover ON orders(customer_id) INCLUDE (total, status);

Trade-off: B-Tree vs LSM-Tree

B-Tree (PostgreSQL, MySQL): fast reads (O(log n) point lookups), slower writes (random I/O for updates). LSM-Tree (Cassandra, RocksDB): fast writes (sequential I/O, append-only), slower reads (must check multiple levels). Choose B-Tree for read-heavy OLTP; LSM-Tree for write-heavy ingestion.

Production Example

Slack: Added a composite index on (channel_id, timestamp) to their messages table. Query "last 50 messages in channel X" went from 800ms full table scan to 2ms index seek. A single index change eliminated their #1 performance bottleneck at 10M+ daily active users.

Key Tip

When discussing database design, always mention: "I'd add indexes on columns used in WHERE clauses and JOIN conditions. For the messages table: composite index on (channel_id, created_at DESC) supports our primary query pattern — latest messages per channel."

4.5.3 Sharding Strategies

Database
// Hash-based sharding: hash(key) % num_shards
// Pro: Even distribution. Con: Resharding requires data migration.

// Range-based sharding: key ranges per shard (e.g., A-M shard1, N-Z shard2)
// Pro: Range queries on shard key. Con: Hot spots if range is popular.

// Consistent Hashing: minimize redistribution on shard add/remove
// Production choice: DynamoDB uses consistent hashing internally

// Shard key selection is CRITICAL:
// Good: customer_id (even distribution, queries isolated to one shard)
// Bad: created_date (all recent writes hit one shard - hot partition!)
// Bad: country_code (uneven - US shard gets 60% of traffic)

Production Example

Instagram: Shards PostgreSQL by user_id. Each shard holds ~10M users. Shard key chosen so one user's feed/posts/likes stay on same shard (locality). Cross-shard queries (global search) go to separate Elasticsearch cluster.

Trade-off: Sharding Strategies

Hash-based: even distribution but cross-shard queries are expensive and range queries impossible. Range-based: efficient range queries and data locality but risks hot shards (recent dates get all writes). Directory-based: flexible mapping but directory is SPOF. Most systems start hash-based and add application-level routing for cross-shard needs.

Key Tip

Only introduce sharding when you've shown the scale requires it. Say: "At 1TB and 50K QPS, a single PostgreSQL instance won't suffice. I'd shard by user_id using consistent hashing — keeps one user's data co-located, and we can add shards without full re-partition." Always mention the cross-shard query challenge and how you'd handle it (scatter-gather or separate search index).

4.5.5 Write-Ahead Log Pattern

Critical

What it does: Before writing data to the main storage, write the operation to a sequential, append-only log file. If the system crashes mid-operation, replay the log on restart to recover incomplete operations. The log is the source of truth until the data is safely persisted.

How it works:

// Simplified WAL flow:
// 1. Client sends: INSERT INTO orders (id, amount) VALUES (42, 99.99)
// 2. Database writes to WAL: [LSN: 1001] INSERT orders (42, 99.99)
// 3. Database acknowledges write to client (durable - it's in the WAL)
// 4. Later: background process flushes the actual data pages to disk
// 5. On crash: replay WAL from last checkpoint to recover unflushed writes

// WAL entry structure (PostgreSQL-style):
// [LSN] [Transaction ID] [Operation Type] [Table] [Before Image] [After Image]
// LSN = Log Sequence Number (monotonically increasing)

When to use:

  • Every reliable database uses WAL internally (PostgreSQL, MySQL InnoDB, Cassandra commit log)
  • Database replication — replicas replay the primary's WAL to stay in sync
  • Change Data Capture (CDC) — applications read the WAL to react to data changes (Debezium)
  • Event sourcing implementations — the event log IS effectively a WAL
  • Distributed consensus — Raft and Paxos use replicated logs

When NOT to use:

  • In-memory caches where durability isn't needed (pure Redis cache without persistence)
  • Ephemeral/disposable data (session stores that can be regenerated)

Trade-offs:

AdvantageDisadvantage
Crash recovery — no data lossDouble-write overhead (log + actual storage)
Sequential writes are fast (append-only)Log grows continuously, needs compaction/checkpointing
Enables replication and CDCRecovery time depends on log size since last checkpoint
Atomic transactions across multiple operationsAdds I/O latency (fsync on every commit for durability)

Key Tip

When discussing database internals or replication, always mention WAL. It's the foundation of: (1) crash recovery, (2) replication (streaming WAL to replicas), (3) point-in-time recovery (replay WAL to any timestamp), and (4) CDC (Debezium reads PostgreSQL's WAL). Say: "The WAL guarantees durability — once the write is in the log, it survives crashes even if the data pages haven't been flushed yet."

Production Example

PostgreSQL streaming replication: Primary streams WAL records to replicas in real-time. Debezium reads PostgreSQL's WAL for Change Data Capture — every INSERT/UPDATE/DELETE becomes a Kafka event for downstream services without touching the application code.

Trade-off: WAL fsync Frequency

fsync every commit: maximum durability (zero data loss) but adds ~2ms latency per write. fsync batched (every 100ms): better throughput but up to 100ms of data loss on crash. PostgreSQL defaults to fsync per commit (synchronous_commit=on); can be relaxed for non-critical data.

4.5.6 Transactions & Isolation Levels

Critical

What it does: A transaction groups multiple operations into an atomic unit — either all succeed or all rollback. Isolation levels control how concurrent transactions see each other's uncommitted changes.

Isolation LevelDirty ReadNon-Repeatable ReadPhantom ReadPerformanceUse Case
Read UncommittedPossiblePossiblePossibleFastestAnalytics on approximate data
Read CommittedPreventedPossiblePossibleFastDefault in PostgreSQL, most OLTP apps
Repeatable ReadPreventedPreventedPossible (InnoDB prevents)ModerateDefault in MySQL InnoDB, financial reports
SerializablePreventedPreventedPreventedSlowestMoney transfers, inventory (correctness critical)
// Anomaly examples:
// Dirty Read: Transaction B reads data that Transaction A wrote but hasn't committed yet.
//   If A rolls back, B used invalid data.

// Non-Repeatable Read: Transaction A reads row X, Transaction B updates X and commits.
//   A reads X again — gets different value within same transaction.

// Phantom Read: Transaction A queries WHERE age > 25 (gets 10 rows).
//   Transaction B inserts a new row with age=30 and commits.
//   A re-runs same query — now gets 11 rows (phantom appeared).

// PostgreSQL Serializable Snapshot Isolation (SSI):
// Uses optimistic concurrency — allows transactions to proceed but
// aborts one if a serialization conflict is detected at commit time.
// No locks held during execution = better throughput than traditional 2PL.

ACID Guarantee Breakdown:

PropertyWhat It MeansImplementation
AtomicityAll or nothing — partial failures rolled backWAL (undo log)
ConsistencyDB moves from one valid state to anotherConstraints, triggers
IsolationConcurrent transactions don't interfereLocks or MVCC
DurabilityCommitted data survives crashesWAL (redo log), fsync

Trade-off: Isolation Level Selection

Higher isolation = fewer anomalies but more locking/aborts = lower throughput. Read Committed (PostgreSQL default) is the sweet spot for most apps — prevents dirty reads without heavy locking. Use Serializable only for financial operations where correctness > performance.

Production Example

Stripe: Payment processing uses Serializable isolation for balance operations — a double-charge would be catastrophic. Non-critical reads (dashboard analytics) use Read Committed for better performance. Different isolation levels for different operations within the same application.

Key Tip

When designing payment or inventory systems, explicitly state: "I'd use Serializable isolation for the balance deduction to prevent race conditions — two concurrent purchases can't both read balance=100 and both succeed." Shows you understand concurrency at the database level.

4.5.7 Schema Design: Normalization vs Denormalization

Database

Normalization: Eliminate data redundancy by splitting into related tables (1NF → 2NF → 3NF → BCNF). Each fact stored once.

Denormalization: Intentionally add redundancy to avoid expensive JOINs at read time. Trade storage for speed.

AspectNormalizedDenormalized
Data redundancyNone — single source of truthDuplicated across tables
Write performanceFast (update one place)Slow (update multiple copies)
Read performanceSlow (many JOINs)Fast (single table scan)
StorageMinimalHigher (duplicate data)
ConsistencyEasy (one copy)Risk of inconsistency across copies
Best forOLTP, write-heavy, transactionalOLAP, read-heavy, analytics, caching
// Normalized (3NF): Orders table references customer_id and product_id
// SELECT o.id, c.name, p.title, o.quantity
// FROM orders o
// JOIN customers c ON o.customer_id = c.id
// JOIN products p ON o.product_id = p.id
// WHERE o.date > '2024-01-01'
// Problem: 3-table JOIN, expensive at scale

// Denormalized: Orders table includes customer_name, product_title directly
// SELECT id, customer_name, product_title, quantity
// FROM orders_denormalized
// WHERE date > '2024-01-01'
// Problem: If customer changes name, must update all their order rows

// Materialized View (compromise): Pre-computed JOIN refreshed periodically
// CREATE MATERIALIZED VIEW order_summary AS
// SELECT o.id, c.name, p.title, o.quantity, o.date
// FROM orders o JOIN customers c ON ... JOIN products p ON ...;
// REFRESH MATERIALIZED VIEW order_summary;  -- run on schedule

Key Tip

In system design: normalize for the write path (transactional core), denormalize for the read path (query/search/dashboard). This is essentially what CQRS does — normalized write model, denormalized read model. Mention Materialized Views as the middle ground. For NoSQL (DynamoDB, Cassandra), denormalization is the default because JOINs don't exist — you design your schema around your access patterns.

Trade-off: When to Denormalize

Denormalize when: read queries are slow due to JOINs, data changes infrequently, and you accept write complexity. Keep normalized when: data changes frequently (avoid update anomalies), storage is a concern, or consistency is critical. The modern pattern: normalized for writes (OLTP), denormalized for reads (OLAP/cache).

Production Example

Twitter: Tweets table is denormalized — includes author_name directly (avoids JOIN to users table on every timeline render). When a user changes their display name, a background job updates all their tweet records. Acceptable trade-off: name changes are rare, timeline reads happen billions of times daily.
