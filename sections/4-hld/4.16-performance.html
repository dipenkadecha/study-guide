4.16Performance Optimization

4.16.1 Caching Strategies

Performance

Caching is the highest-leverage performance optimization — it removes work entirely rather than making it faster.

→ The caching patterns (cache-aside, write-through, write-behind, read-through), eviction policies, and stampede prevention are covered in full on the Caching page (4.6).

4.16.2 Index Optimization

Performance
// Database Index Types and When to Use:

// 1. B-Tree Index (default): range queries, equality, sorting
CREATE INDEX idx_orders_date ON orders(created_at);
-- Good for: WHERE created_at BETWEEN x AND y, ORDER BY created_at

// 2. Hash Index: equality lookups only (O(1))
-- Good for: WHERE user_id = 'abc123' (exact match only)

// 3. Composite Index: multi-column queries (leftmost prefix rule)
CREATE INDEX idx_user_status_date ON orders(user_id, status, created_at);
-- Supports: WHERE user_id = ? AND status = ? AND created_at > ?
-- Also: WHERE user_id = ? AND status = ? (prefix)
-- NOT: WHERE status = ? (skips leftmost column)

// 4. Covering Index: all columns in query are in index (no table lookup)
CREATE INDEX idx_covering ON orders(user_id, status) INCLUDE (total_amount);
-- SELECT total_amount FROM orders WHERE user_id = ? AND status = ?
-- Index-only scan: never touches the table heap

// 5. Partial Index: index subset of rows (smaller, faster)
CREATE INDEX idx_active_premium ON users(email) WHERE status = 'active' AND tier = 'premium';

Trade-off

More indexes = faster reads but slower writes (each INSERT/UPDATE must update all relevant indexes). A table with 10 indexes can have 5-10x slower write throughput. Strategy: index only columns in WHERE/JOIN/ORDER BY clauses. For write-heavy tables, keep indexes minimal and use covering indexes to eliminate table lookups for hot queries.

Production Example

LinkedIn: Reduced query latency by 95% by adding a covering index that included all columns needed by their most frequent query — eliminated the table lookup (index-only scan). Before: 50ms with 2 disk seeks. After: 2ms from index cache only.

Key Tip

When designing schemas: 'For the users table, I'd add a unique index on (email) for login lookups, a composite index on (status, created_at) for admin filtering, and ensure our primary query uses a covering index to avoid heap lookups. I'd monitor unused indexes periodically and drop them to reduce write overhead.'

4.16.4 Connection Pooling

Performance

Reusing pre-established DB connections avoids the 50–100ms TCP+TLS+auth cost on every query — a major latency win under load.

→ HikariCP sizing, HTTP connection pooling, and PgBouncer multiplexing are covered in full on the Caching page (4.6.3 Connection Pooling).

4.16.5 Compression

Performance

DatabaseColumn compression, TOAST (PostgreSQL)VariesLarge text/JSON fields, time-series data

Trade-off: Performance vs Complexity

  • Premature optimization: Adding caching/indexes before understanding access patterns wastes effort and adds complexity
  • Measure first: Use APM (Application Performance Monitoring) to identify actual bottlenecks before optimizing
  • Cache invalidation: "The two hardest problems in CS: cache invalidation, naming things, and off-by-one errors." Every cache adds a consistency challenge.

Production Example

Google: Uses Brotli compression for all web assets — 20-26% smaller than gzip at comparable CPU cost. For internal data: Zstandard (zstd) compresses Bigtable column families — 3-5x compression ratio with decompression at 1.5 GB/s (fast enough to be I/O-bound, not CPU-bound). Net effect: more data fits in memory cache.

Key Tip

Mention compression at appropriate layers: 'API responses use gzip/Brotli (Accept-Encoding). Database uses page-level compression (PostgreSQL TOAST, RocksDB LZ4). Network: gRPC has built-in compression for internal RPCs. Kafka: producer-side compression (LZ4) reduces network bandwidth by 70% with minimal CPU overhead.'
