4.6Caching

4.6.1 Caching Patterns

Performance
// Cache-Aside pattern (Lazy Loading) - most common
public Order getOrder(String orderId) {
    // 1. Check cache
    Order cached = redisTemplate.opsForValue().get("order:" + orderId);
    if (cached != null) return cached;

    // 2. Cache miss - load from DB
    Order order = orderRepository.findById(orderId).orElseThrow();

    // 3. Populate cache with TTL
    redisTemplate.opsForValue().set("order:" + orderId, order, Duration.ofMinutes(15));
    return order;
}

// Cache Stampede prevention (thundering herd)
// Problem: cache expires, 1000 concurrent requests all hit DB
// Solution 1: Distributed lock (only one thread refreshes)
public Order getOrderWithLock(String orderId) {
    Order cached = cache.get("order:" + orderId);
    if (cached != null) return cached;

    String lockKey = "lock:order:" + orderId;
    if (redisLock.tryLock(lockKey, Duration.ofSeconds(5))) {
        try {
            cached = cache.get("order:" + orderId); // Double-check
            if (cached != null) return cached;
            Order fresh = orderRepository.findById(orderId).orElseThrow();
            cache.set("order:" + orderId, fresh, Duration.ofMinutes(15));
            return fresh;
        } finally { redisLock.unlock(lockKey); }
    }
    return cache.get("order:" + orderId); // Wait for other thread's result
}

Production Example

Facebook (Meta): Memcached cluster with 1000+ servers caching user profiles, friend lists, and news feed data. Cache-aside pattern: on miss, read from MySQL, populate cache. Handles 10B+ cache requests/second with p99 < 1ms.

Trade-off: Cache-Aside vs Read-Through

Cache-Aside: application controls loading logic (flexible, explicit) but cache misses are expensive (app must handle). Read-Through: cache auto-loads on miss (simpler app code) but less control over loading strategy. Write-Through ensures consistency but doubles write latency. Write-Behind is fast but risks data loss if cache crashes before flush.

Interview Tip

Always specify your caching strategy in HLD: "We use Cache-Aside with Redis for user profiles — TTL of 5 minutes. On write, we invalidate the cache key (not update it) to avoid race conditions between concurrent reads and writes." Mention TTL + invalidation strategy together.

4.6.2 Eviction Policies

Performance
PolicyEvictsBest ForOverhead
LRULeast Recently UsedGeneral purpose, recency mattersO(1) with LinkedHashMap
LFULeast Frequently UsedPopularity matters (CDN)O(log n) or O(1) with O(n) space
W-TinyLFU (Caffeine)Admission filter + LFUBest overall hit rateO(1), compact sketch
TTLExpired entriesTime-sensitive dataO(1)

Production Example

Caffeine (Java): Uses W-TinyLFU eviction — combines recency (LRU window) and frequency (count-min sketch). Achieves near-optimal hit rates. Used by Spring Boot's default cache, Neo4j, Apache Druid. Outperforms Guava cache by 30-50% hit rate in production benchmarks.

Trade-off: LRU vs LFU vs W-TinyLFU

LRU: evicts least recently accessed (simple, good for temporal locality). LFU: evicts least frequently accessed (good for stable hot set) but slow to adapt to new popular items. W-TinyLFU (Caffeine): combines both — admission window (LRU) + main cache (LFU with aging). Dominates in benchmarks but more complex implementation.

Interview Tip

Default to LRU in interviews (it's the most common). If asked to optimize: "For our product catalog cache where some items are always popular, I'd use W-TinyLFU (Caffeine in Java, or Redis with approximated LFU). It prevents cache pollution from one-time scans that would evict genuinely hot items."

4.6.3 Connection Pooling

Performance
// HikariCP - fastest Java connection pool
// Formula: pool_size = (2 * CPU_cores) + effective_spindle_count
// For SSD (spindle=0): pool_size = 2 * cores = 16 for 8-core machine
// More connections != better: context switching overhead dominates

// HTTP Connection Pooling (RestTemplate/WebClient)
@Bean
public WebClient webClient() {
    HttpClient httpClient = HttpClient.create()
        .option(ChannelOption.CONNECT_TIMEOUT_MILLIS, 3000)
        .responseTimeout(Duration.ofSeconds(5))
        .pool(ConnectionProvider.builder("custom")
            .maxConnections(100)           // Max total connections
            .maxIdleTime(Duration.ofSeconds(30))
            .maxLifeTime(Duration.ofMinutes(5))
            .pendingAcquireMaxCount(1000)  // Queue when pool exhausted
            .build());
    return WebClient.builder().clientConnector(new ReactorClientHttpConnector(httpClient)).build();
}

Trade-off: Redis vs Memcached vs Caffeine

FeatureRedisMemcachedCaffeine (local)

ClusteringYes (Redis Cluster)Client-side shardingNo (single JVM)

Best practice: Multi-level caching: L1 (Caffeine, in-JVM, 100ns) -> L2 (Redis, distributed, 0.5ms) -> DB (10-50ms)

Production Example

Heroku: Each dyno gets max 20 database connections. With 50 dynos, that's 1000 connections to PostgreSQL (near its limit). PgBouncer in transaction-pooling mode reduces actual DB connections to 100 while serving 1000 application connections — 10x multiplexing ratio.

Interview Tip

Mention connection pooling whenever you discuss databases at scale: "Each service instance maintains a HikariCP pool of 10-20 connections. With 100 instances, that's 1000-2000 DB connections — if that exceeds PostgreSQL's limit (~500 recommended), we add PgBouncer for connection multiplexing."

4.6.4 Read-Through Cache Pattern

Pattern

What it does: The cache itself is responsible for loading data from the database on a miss. The application only talks to the cache, never directly to the database. On a cache miss, the cache loads from DB, stores the result, and returns it.

Comparison of all cache loading patterns:

PatternRead FlowWrite FlowBest For
Cache-Aside (Lazy Loading)App checks cache → miss → app reads DB → app writes to cacheApp writes to DB, invalidates cacheGeneral purpose, most common
Read-ThroughApp reads cache → miss → cache reads DB → cache stores & returnsApp writes to DB, cache auto-loads on next readSimplify app code, cache handles loading
Write-ThroughApp reads cache (always hit after first write)App writes to cache → cache writes to DB synchronouslyRead-after-write consistency critical
Write-Behind (Write-Back)App reads cacheApp writes to cache → cache async flushes to DB in batchesWrite-heavy, brief inconsistency OK

4.6.5 Cache Stampede Prevention

Critical

Problem: When a popular cache entry expires, thousands of simultaneous requests hit the database for the same data — a "thundering herd" or cache stampede.

Solutions:

TechniqueHow It WorksTrade-off
Request Coalescing (Singleflight)First request acquires a lock and fetches from DB. All other concurrent requests for the same key wait for the first to complete, then share the result.Adds lock coordination; all waiters blocked on single fetch
Probabilistic Early ExpirationEach request has a small random chance of refreshing the cache BEFORE it expires. Popular keys get refreshed by natural traffic.Occasional redundant DB reads; doesn't guarantee no stampede
Background RefreshA background job refreshes popular cache entries before they expire. The cache never actually expires for hot keys.Stale data during refresh window; extra infrastructure
Stale-While-RevalidateServe stale data immediately while asynchronously refreshing the cache in the background.Clients briefly see stale data; simpler than locking
// Singleflight pattern (Go-style, applicable in Java with Caffeine or custom)
// Only ONE thread fetches from DB, others wait and share the result

// Java with Caffeine (built-in coalescing):
LoadingCache<String, User> cache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(Duration.ofMinutes(5))
    .build(key -> userRepository.findById(key)); // Only called ONCE per key miss

// Manual singleflight with ConcurrentHashMap:
ConcurrentMap<String, CompletableFuture<User>> inFlight = new ConcurrentHashMap<>();

public User getUser(String id) {
    User cached = cache.get(id);
    if (cached != null) return cached;
    
    // computeIfAbsent ensures only ONE future is created per key
    CompletableFuture<User> future = inFlight.computeIfAbsent(id, k ->
        CompletableFuture.supplyAsync(() -> {
            User user = db.findById(k);
            cache.put(k, user);
            return user;
        })
    );
    
    try {
        return future.get(); // All concurrent callers share this single DB fetch
    } finally {
        inFlight.remove(id); // Clean up after completion
    }
}

Interview Tip

When discussing caching, interviewers often follow up with "what happens when the cache key expires and 10,000 requests hit simultaneously?" This is the cache stampede question. Mention request coalescing (singleflight) as the primary solution, then note that Caffeine and Guava LoadingCache handle this automatically. For distributed caches (Redis), mention distributed locks or probabilistic early refresh.

Trade-off: Stampede Prevention Approaches

Locking (singleflight): guaranteed single fetch but adds coordination latency + lock contention. Probabilistic early refresh: no locks but occasional redundant DB reads. Stale-while-revalidate: instant response (stale) but clients briefly see outdated data. Best practice: use Caffeine/Guava LoadingCache (built-in coalescing) for local cache, distributed lock for Redis.

Production Example

Reddit: Popular post cache keys expire simultaneously → thousands of requests hit PostgreSQL for the same post. Solution: probabilistic early expiration — random 10% of requests refresh the cache 30 seconds before TTL. Popular keys stay warm naturally from organic traffic.
