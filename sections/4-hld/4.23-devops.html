4.23DevOps & Deployment Patterns

4.23.1 GitOps

DevOps
// GitOps: Git is single source of truth for infrastructure & application state
// Flow: Code change -> PR -> merge -> ArgoCD detects -> syncs to cluster
// Benefits: audit trail (git history), rollback (git revert), declarative

// ArgoCD watches a GitOps repo containing Kubernetes manifests
// Any drift between git and cluster is auto-corrected
// No kubectl apply from laptops - all changes through PR review

Trade-off

GitOps (declarative, git as source of truth): full audit trail, easy rollback (git revert), consistent environments. But: secrets management is tricky (can't store secrets in git), and drift detection adds complexity. Traditional CI/CD (imperative push): simpler for small teams but no audit trail of WHAT is deployed WHERE without extra tooling.

Production Example

Weaveworks/Intuit: ArgoCD watches git repositories for Kubernetes manifest changes. When a developer merges a PR that changes a Deployment replica count, ArgoCD detects the drift from desired state and automatically syncs the cluster. Rollback = revert the git commit. Entire cluster state is reproducible from a single git repository.

Interview Tip

For deployment questions: 'We use GitOps with ArgoCD — Kubernetes manifests stored in git, ArgoCD syncs cluster state to match. Deployment = merge PR to main branch. Rollback = git revert. Benefits: complete audit trail (who changed what, when, why via PR), reproducible environments (spin up staging from same git repo), and developer-friendly (no kubectl access needed).'

4.23.2 Feature Flags

DevOps
// Decouple deployment from release - deploy code without activating it
// Use cases: gradual rollout, A/B testing, kill switch, trunk-based development
if (featureFlags.isEnabled("new-checkout-flow", userId)) {
    return newCheckoutService.process(cart);
} else {
    return legacyCheckoutService.process(cart);
}
// Tools: LaunchDarkly, Unleash, Flagsmith, custom Redis-backed
// Critical: clean up old flags! Tech debt accumulates fast.

Production Example

Facebook: GateKeeper system controls feature rollout to 3B+ users. New features start at 0.1% (internal), scale to 1% (dogfood), then 10%, 50%, 100%. If metrics degrade at any stage, one click disables the feature globally in seconds without a deploy. Engineers ship code daily — GateKeeper decouples deploy from release.

Trade-off

Feature flags enable safe deployments (deploy dark, enable gradually) but accumulate as technical debt if not cleaned up. Each flag adds conditional complexity, increases test matrix (flag on/off), and risk of flag conflicts. Best practice: treat flags as temporary — remove within 2 sprints of full rollout. Permanent flags only for ops toggles (maintenance mode) or A/B experiments.

Interview Tip

Mention feature flags for risk mitigation: 'New recommendation algorithm deployed behind a feature flag. Enable for 1% of users, monitor engagement metrics for 24h. If positive, roll to 10%, 50%, 100% over a week. If metrics degrade at any stage, kill switch (disable flag) gives instant rollback without deployment. Flags are cleaned up after full rollout.'

4.23.3 Immutable Infrastructure

DevOps
// Never modify running servers. Replace them entirely.
// Old way: SSH into server, apt-get update, restart (snowflake servers)
// New way: Build new AMI/image with changes, deploy new instances, destroy old

// Benefits: reproducible, no configuration drift, easy rollback (switch back)
// Implementation: Packer builds AMI -> Terraform deploys -> old instances terminated
// Containers embody this: never docker exec to fix - rebuild image and redeploy

Trade-off: Feature Flags vs Branch-based Development

  • Feature Flags: Trunk-based, deploy often, toggle features. Risk: flag complexity, dead code if not cleaned.
  • Feature Branches: Isolation, full testing before merge. Risk: long-lived branches, merge conflicts, integration pain.
  • Best practice: Use flags for major features (gradual rollout), short-lived branches (< 2 days) for smaller changes.

Production Example

Netflix: Every deployment creates new AMIs (Amazon Machine Images). Servers are never patched in place — a new image is built, tested, and deployed. The old ASG (Auto Scaling Group) is scaled down as the new one scales up (red/black deployment). If issues arise, simply scale the old ASG back up. No configuration drift — every server is identical to its image.

Interview Tip

Connect to deployment strategy: 'We use immutable infrastructure — containers are built once, promoted through environments (dev → staging → prod) without modification. No SSH into production, no config changes on running instances. If a fix is needed, build a new image and deploy. This eliminates configuration drift — every production instance is identical and reproducible from the Dockerfile.'

4.23.4 Production CI/CD Pipeline Stages

Production
// Modern CI/CD for a Java microservice:
// 1. Commit Stage (2-5 min): compile, unit tests, SAST (SonarQube), dependency check
// 2. Build Stage (3-5 min): Docker image build, push to registry (ECR)
// 3. Integration Stage (5-10 min): Testcontainers integration tests
// 4. Security Stage: container scan (Trivy), secret detection, OWASP check
// 5. Deploy Staging: full environment, contract tests, performance tests
// 6. Deploy Prod: canary 5% -> metrics check -> auto-rollback or promote -> 100%

// Key metrics to gate promotion:
// Error rate < 0.1%, p99 latency < SLO, no new error types
// If breached -> automatic rollback to previous version

Production Example

Google: Monorepo with 1 billion files, 86TB of source code. Every commit triggers CI across affected targets. 150K+ builds/day on distributed build system (Bazel). Deploys to production multiple times daily via automated canary analysis — if error rates spike, automatic rollback within 2 minutes.

Trade-off

Comprehensive pipeline (lint → unit test → integration test → security scan → performance test → canary) gives confidence but is slow (30-60 min). Fast pipeline (lint → unit test → deploy) enables rapid iteration but risky. Balance: fast pipeline for feature branches (lint + unit tests, ~5 min), comprehensive pipeline for main/release branches (all stages, 30 min). Never block developers with slow feedback loops.

Interview Tip

Describe your ideal pipeline: 'PR triggers: lint + unit tests + security scan (5 min). Merge to main triggers: full integration tests + build image + deploy to staging + smoke tests (15 min). Release: canary 5% for 30 min (automated metric comparison) → progressive rollout. On failure at any stage: block + notify. Key metric: lead time (commit to production) < 1 hour.'

4.23.5 Blue-Green vs Canary vs Rolling

Production
StrategyRollbackRiskInfra CostBest For
Blue-GreenInstant (LB switch)Low2x during deployCritical services, instant rollback needed
CanaryFast (seconds)LowestSmall extraHigh-traffic, subtle bugs
RollingSlow (reverse roll)MediumNoneK8s default, stateless services
RecreateRedeploy (slow)HighNoneSchema changes, incompatible versions
// Progressive canary with Argo Rollouts (GitOps: ArgoCD syncs this on image bump)
apiVersion: argoproj.io/v1alpha1
kind: Rollout
spec:
  strategy:
    canary:
      steps:
      - setWeight: 5       # send 5% of traffic to the new version
      - pause: {duration: 5m}
      - analysis:          # automated gate: is success-rate healthy?
          templates:
          - templateName: success-rate
      - setWeight: 50
      - pause: {duration: 10m}
      - setWeight: 100     # promote to 100% if metrics stayed green

Production Example

Amazon: Canary deployments across 100M+ customer requests. New version gets 1% traffic for 30 minutes. Automated analysis compares error rates, latency percentiles, and business metrics against baseline. If any metric regresses beyond threshold, automatic rollback fires — zero human intervention for 99% of bad deploys.

Interview Tip

When asked about deployment, always mention: "We use canary with automated metric analysis. If error rate exceeds threshold, Argo Rollouts automatically rolls back. This caught a memory leak in production last quarter before it affected more than 2% of users."

Trade-off

Blue-Green: instant rollback (switch back to old), simple, but requires 2x infrastructure during transition. Canary: gradual risk exposure (1% → 5% → 100%), resource-efficient (only a few new pods), but rollback is slower (need to detect issue, route back). Rolling update: resource-efficient (replace pods one by one), zero extra infra, but rollback means re-rolling forward with old version (slow).
