4.20Microservices Patterns

4.20.1 Service Mesh (Istio/Linkerd)

Architecture
// Service mesh: infrastructure layer for service-to-service communication
// Sidecar proxy (Envoy) handles: mTLS, retries, circuit breaking, observability
// Application code is unaware - zero code changes

// Benefits: consistent security (mTLS everywhere), traffic management (canary),
// observability (distributed tracing auto-injected), no library dependencies

// Trade-off: Added latency (~1-3ms per hop), operational complexity,
// resource overhead (sidecar per pod). Worth it at 20+ services.

Production Example

Lyft (Envoy): Created Envoy proxy — runs as sidecar next to every service. Handles mTLS, circuit breaking, retries, observability without application code changes. Now part of Istio. Lyft routes 3M+ requests/second through Envoy sidecars — services just talk to localhost, Envoy handles the rest.

Trade-off

Sidecar overhead: each pod gets an Envoy proxy (+100MB RAM, +2ms latency per hop). For 500 services with 5 replicas each = 2500 sidecars consuming 250GB RAM cluster-wide. Benefits: mTLS everywhere, traffic management, observability — all without code changes. Skip service mesh if <20 services or latency-critical (trading systems). Consider ambient mesh (Istio) for lower overhead.

Trade-off: Service Mesh vs Library-based (Spring Cloud)

  • Service Mesh (Istio): Language-agnostic, no code changes, consistent policies. Higher infra complexity, latency overhead.
  • Library (Spring Cloud): Simpler ops, no sidecar overhead. Java-only, library version management, code coupling.
  • Decision: Polyglot/large orgs -> Service Mesh. Java-only/small team -> Spring Cloud. Can combine both.

Key Tip

Mention service mesh for cross-cutting concerns: "Rather than implementing retry/timeout/circuit-breaker in every service, we use Istio service mesh. The Envoy sidecar handles: mTLS (zero-trust), retries with budget (max 20% of requests retried), circuit breaking (5xx threshold), and emits golden metrics (latency, errors, saturation) automatically. App code stays focused on business logic."

4.20.2 Strangler Fig Migration

Architecture
// Incrementally migrate monolith to microservices without big-bang rewrite
// Steps:
// 1. Place proxy/facade in front of monolith
// 2. New features built as microservices behind proxy
// 3. Gradually route traffic from monolith to new services
// 4. Eventually monolith shrinks to nothing

// Anti-corruption Layer: adapter between old and new systems
// Prevents legacy domain model from leaking into new services
public class LegacyOrderAdapter implements OrderPort {
    private final LegacyOrderSystem legacy;
    public Order getOrder(String id) {
        LegacyOrderDTO dto = legacy.fetchOrder(id);
        return mapToDomainModel(dto); // Translate legacy to clean domain
    }
}

Production Example

Amazon: Migrated from monolithic bookstore (Obidos) to microservices over 8+ years. New features built as services, old monolith routes gradually redirected via API Gateway. No big-bang rewrite — the monolith "strangled" as traffic shifted service by service until nothing remained.

Trade-off

Strangler Fig is low-risk (old system runs until new one proves itself) but slow (months-to-years for full migration) and complex (routing logic, data sync between old and new). Big-bang rewrite is faster but high-risk (if it fails, massive wasted effort). Strangler Fig is almost always preferred — reduces risk at the cost of longer timeline and temporary complexity of running both systems.

Key Tip

For migration questions: "I'd use Strangler Fig — route new feature traffic to the new service while existing traffic stays on the monolith. An API gateway routes by path: /v2/users → new service, /users → legacy. Gradually migrate endpoints. Data: new service reads from both old DB and new DB during transition. Once all traffic migrated, decommission legacy."

4.20.3 BFF (Backend for Frontend)

Architecture

What it does: a separate API backend layer per client type (web, mobile, internal tools, third-party). Each BFF aggregates and shapes data from the same backend microservices for its specific frontend's needs.

// Without BFF: one generic API serves all clients
//   Mobile app → Generic API → 10 microservices
//   Web app    → Generic API → 10 microservices
//   Problem: Mobile gets too much data, Web needs extra calls for rich views

// With BFF: each client type has its own backend
//   Mobile app → Mobile BFF → microservices (lightweight, minimal data, offline-first)
//   Web app    → Web BFF    → microservices (rich, aggregated data, SSR, WebSocket)
//   Admin tool → Admin BFF  → microservices (full access, different auth)

// Mobile BFF: aggregate 3 service calls into 1 optimized response
// GET /mobile/dashboard  -> userService + orderService + recommendationService
//   Returns: single small, flat JSON payload (no deep nesting)

When to use:

  • Significantly different client needs (mobile bandwidth vs desktop richness)
  • Different teams own different frontends (mobile team controls their BFF)
  • Third-party clients need different auth, rate limits, and data shapes than internal ones

When NOT to use:

  • Single client type (just have one API layer)
  • Clients have similar needs (unnecessary duplication)
  • Small team (maintaining multiple BFFs is overhead) — GraphQL can sometimes replace BFF

Trade-off: BFF vs GraphQL vs Generic API

ApproachProsCons
BFF per clientOptimized per client, team autonomy, simple client codeCode duplication across BFFs, more services to maintain
GraphQLSingle API, clients fetch exactly what they need, no over/under-fetchingComplex server implementation, N+1 query risk, caching harder
Generic REST APISimple, one API to maintainOver-fetching for mobile, under-fetching for web (multiple round trips)

Production Example

Netflix: Separate BFFs for TV app (simple navigation, large images), mobile app (bandwidth-constrained, smaller payloads), and web app (interactive, more data). Each BFF aggregates the same backend microservices but returns device-optimized responses — TV BFF returns pre-formatted layouts; mobile BFF returns compressed, paginated data.

Key Tip

Use BFF when clients have very different needs: "Mobile app needs minimal payload (battery, bandwidth). Web app needs rich data for interactive dashboards. Rather than one generic API that over-serves mobile or under-serves web, each platform gets a BFF that aggregates and shapes data from the same backend services for its specific needs."

4.20.4 Service Discovery

Patterns
// Client-side discovery: client queries service registry, picks instance
// Server-side discovery: load balancer queries registry, routes request

// Client-side (Netflix Eureka, Consul):
// Pro: fewer hops, client controls LB. Con: client needs discovery library.
// Server-side (AWS ALB, Kubernetes Service):
// Pro: clients are simple. Con: extra hop, LB is bottleneck.

// Kubernetes built-in: Service object = stable DNS name + load balancing
// Pod IPs change on restart, Service IP stays constant
// kube-dns resolves "order-service.namespace.svc.cluster.local" -> pod IPs

Trade-off

Client-side (Eureka) vs Server-side (K8s DNS/AWS ALB): Client-side gives more control (custom load balancing, circuit breaking) but adds library dependency to every service. Server-side is transparent (services just use DNS) but less flexible (standard round-robin only). In Kubernetes: use K8s Service (server-side) as default; add Istio for advanced routing if needed.

Production Example

Netflix (Eureka): Each service instance registers itself with Eureka on startup, sends heartbeats every 30s. Client library caches the registry locally — continues working even if Eureka goes down. Services fetch registry updates every 30s. Deregistration on graceful shutdown; 90s timeout for crashed instances. Self-preservation mode prevents mass deregistration during network partitions.

Key Tip

In Kubernetes designs, service discovery is mostly solved: "K8s Services provide DNS-based discovery (my-service.namespace.svc.cluster.local resolves to pod IPs). For cross-cluster or non-K8s: Consul for service registry with health checks. Key point: services register on startup, deregister on shutdown, and health checks remove unhealthy instances automatically."

4.20.5 Database-per-Service Pattern

Patterns
// Each microservice owns its data - no shared database!
// Benefits: independent schema evolution, independent scaling, technology freedom
// Challenge: cross-service queries (no JOINs across services)

// Solutions for cross-service data:
// 1. API Composition: aggregate data from multiple service APIs
// 2. CQRS: maintain materialized views via events
// 3. Saga: coordinate multi-service transactions

// API Composition (Aggregator pattern):
@Service
public class OrderSummaryAggregator {
    public OrderSummary getFullOrder(String orderId) {
        Order order = orderClient.getOrder(orderId);           // Service 1
        Payment payment = paymentClient.getByOrder(orderId);   // Service 2
        Address address = userClient.getAddress(order.userId()); // Service 3
        return new OrderSummary(order, payment, address);       // Combine
    }
    // Trade-off: multiple network calls, eventual consistency, partial failures
}

Production Example

Netflix: Each microservice owns its data store — user service uses Cassandra (high write throughput), recommendation service uses MySQL + ML feature store, search uses Elasticsearch, session service uses Redis. No shared databases — services communicate only via APIs and events.

Trade-off

Data isolation (each service owns its data) eliminates coupling but makes cross-service queries impossible (no JOINs). Requires: API calls for cross-service data, eventual consistency between services, Saga pattern for distributed transactions. Alternative: shared database with schema-per-service (logical isolation, physical sharing) — easier queries but risks coupling through shared tables.

Key Tip

Explain the motivation: "Each microservice owns its database — User Service has PostgreSQL, Product Service has MongoDB, Search Service has Elasticsearch. This enables: independent deployments (schema changes don't affect others), technology diversity (right DB for each use case), independent scaling. Trade-off: cross-service queries require API composition (BFF aggregates) or materialized views via CDC."

4.20.6 Anti-corruption Layer (ACL)

Patterns
// When integrating with legacy/external systems, ACL prevents their domain
// model from polluting your clean domain model

// Without ACL: legacy field names, formats, and quirks leak everywhere
// With ACL: adapter translates between external model and your domain

public class PaymentGatewayACL implements PaymentPort {
    private final LegacyPaymentAPI legacy; // Horrible API with weird naming

    public PaymentResult charge(Money amount, PaymentMethod method) {
        // Translate clean domain to legacy format
        LegacyRequest req = new LegacyRequest();
        req.setAMT(amount.cents());          // Legacy uses cents as int
        req.setPMT_TYP(mapToLegacyCode(method)); // Legacy uses integer codes
        LegacyResponse resp = legacy.processTransaction(req);
        // Translate back to clean domain
        return new PaymentResult(resp.getTXN_ID(), mapStatus(resp.getSTS()));
    }
}

Trade-off: Shared Database vs Database-per-Service

ApproachConsistencyCouplingComplexityWhen
Shared DBStrong (ACID joins)High (schema lock-in)LowEarly-stage, small team, tightly related services
DB-per-serviceEventualLow (independent)High (sagas, events)Independent teams, different scaling needs
Shared schema, separate tablesMediumMediumMediumTransitional step from monolith

Production Example

Shopify: When integrating with legacy ERP systems for enterprise merchants, they use an ACL that translates between Shopify's modern domain model (Product, Variant, Inventory) and the ERP's legacy model (SKU, BOM, WarehouseSlot). The ACL prevents ERP concepts from leaking into Shopify's clean domain — isolates legacy complexity.

Key Tip

Use ACL during migrations or integrations: "Between our new Order Service and the legacy billing system, I'd place an Anti-corruption Layer. It translates our domain events (OrderPlaced) into the legacy system's expected format (SOAP XML with their specific field names). If the legacy system changes or is replaced, only the ACL adapter needs updating — our domain stays clean."
