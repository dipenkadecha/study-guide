SessionTraditional web appsCookieNo (server-side state)

AuthorizationModelBest For

ReBACRelationship-based (Google Zanzibar)Social graphs, document sharing

Production Example

Google: OAuth 2.0 + OpenID Connect for "Sign in with Google" across 4M+ third-party apps. Internally uses ALTS (Application Layer Transport Security) for service-to-service mTLS — every RPC between Google services is authenticated and encrypted without developer involvement.

Trade-off: JWT vs Session Tokens

JWT: stateless (no server-side session store), scalable, but can't be revoked until expiry (use short TTL + refresh tokens). Session tokens: revocable instantly (delete from store), but require centralized session store (Redis) — adds latency and SPOF. Most production systems use short-lived JWTs (15min) + refresh tokens stored server-side.

Key Tip

Specify your auth architecture: "API Gateway validates JWT signatures (stateless, no DB call). JWTs expire in 15 minutes. Refresh tokens are stored in Redis with user_id — enables instant revocation (logout, password change). Internal service-to-service uses mTLS (mutual TLS) — no tokens needed, identity verified by certificate."

4.12.4 Zero Trust Architecture

Security
// Zero Trust Principles:
// 1. Never trust, always verify (even internal traffic)
// 2. Least privilege access (minimal permissions per service)
// 3. Assume breach (encrypt everything, segment networks)

// Implementation:
// - mTLS between all services (Istio/Linkerd service mesh)
// - Short-lived tokens (rotate every 15 min, not days)
// - Network segmentation (namespace isolation in K8s)
// - Secrets management (Vault with dynamic secrets)
// - Continuous verification (re-auth on sensitive operations)

Production Example

Google BeyondCorp: Eliminated VPN entirely. Every request (internal or external) must prove identity + device health + context. An engineer accessing an internal tool from a coffee shop gets the same auth challenge as from the office. Access decisions made per-request, not per-network-location.

Trade-off: Zero Trust vs Perimeter Security

Zero Trust: every request authenticated/authorized regardless of network location — more secure but adds latency (auth check on every call) and complexity (policy engine, identity infrastructure). Perimeter: simple (trust everything inside the VPN) but catastrophic if breached (lateral movement is trivial). Industry is moving to Zero Trust after high-profile breaches (SolarWinds).

Key Tip

For security-sensitive designs: "We follow Zero Trust — no implicit trust based on network location. Every service-to-service call requires mTLS + JWT with fine-grained permissions. Even internal services behind the firewall authenticate every request. Access decisions are policy-based (OPA/Rego) evaluating: who + what + where + when."
