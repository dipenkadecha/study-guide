4.7Data Processing

4.7.1 Processing Paradigms

Data
TypeLatencyToolsUse Case
BatchMinutes to hoursSpark, MapReduce, Flink (batch)ETL, reports, ML training
StreamMilliseconds to secondsKafka Streams, Flink, Spark StreamingReal-time analytics, fraud detection
Lambda ArchitectureBothBatch layer + Speed layer + Serving layerAccuracy + low latency (complex)
Kappa ArchitectureStream onlySingle stream processing pipelineSimpler, replay from log
// Kafka Streams - real-time event processing in Java
StreamsBuilder builder = new StreamsBuilder();
KStream<String, OrderEvent> orders = builder.stream("orders");

// Real-time aggregation: revenue per product per hour
KTable<Windowed<String>, Long> hourlyRevenue = orders
    .groupBy((key, order) -> order.productId())
    .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofHours(1)))
    .aggregate(
        () -> 0L,
        (key, order, total) -> total + order.amount(),
        Materialized.with(Serdes.String(), Serdes.Long())
    );

// Output to topic for downstream consumers
hourlyRevenue.toStream()
    .map((windowed, total) -> KeyValue.pair(windowed.key(), total))
    .to("hourly-revenue");

Trade-off: Batch vs Stream Processing

  • Batch: Higher throughput, complete data (full recomputation), complex joins easy, higher latency. Use for: daily reports, ML training, data warehouse ETL.
  • Stream: Lower latency, incomplete data (approximations), complex state management, exactly-once harder. Use for: fraud detection, real-time dashboards, alerting.
  • Modern trend: Unified (Apache Flink handles both). Use stream-first architecture, batch as special case of bounded stream.

Production Example

Spotify Wrapped: Batch processing (Apache Spark) computes yearly listening statistics for 400M+ users. Runs in December, processes petabytes of listening history accumulated all year. Results cached and served when users open the app — classic batch-then-serve pattern.

Key Tip

Always clarify latency requirements: "Does the user need results in real-time (stream processing) or is nightly batch acceptable?" This single question determines your architecture. Real-time → Flink/Kafka Streams. Batch → Spark/Airflow. Hybrid → Lambda Architecture (both).

4.7.2 Lambda Architecture Pattern

Architecture

What it does: Runs batch and stream processing in parallel. The batch layer produces accurate results with high latency. The speed layer (stream) produces approximate results with low latency. A serving layer merges both views for queries.

// Lambda Architecture layers:
//
// ┌─────────────────────────────────────────────────┐
// │              Raw Data (Immutable)                 │
// └────────────┬───────────────────┬─────────────────┘
//              │                   │
//    ┌─────────▼────────┐  ┌──────▼──────────┐
//    │   Batch Layer     │  │  Speed Layer     │
//    │ (MapReduce/Spark) │  │ (Flink/Kafka)    │
//    │ Complete, accurate│  │ Real-time, approx│
//    │ High latency      │  │ Low latency      │
//    └─────────┬─────────┘  └──────┬───────────┘
//              │                    │
//    ┌─────────▼────────────────────▼──────────┐
//    │           Serving Layer                   │
//    │  Merges batch views + real-time views     │
//    │  Serves queries with best available data  │
//    └──────────────────────────────────────────┘

// Example: Analytics Dashboard
// - Batch layer: Nightly Spark job computes accurate daily/weekly/monthly metrics
// - Speed layer: Kafka Streams computes real-time counters since last batch run
// - Serving layer: Query = batch_result + speed_result_since_last_batch

When to use:

  • Systems needing both real-time responsiveness AND batch-level accuracy
  • Analytics platforms showing real-time trends and daily aggregates
  • Fraud detection: real-time scoring + nightly model retraining
  • Search engines: real-time index updates + periodic full reindexing

When NOT to use:

  • Simple systems where batch OR stream alone suffices
  • When the Kappa Architecture (stream-only with replay) can handle your needs

Trade-off: Lambda vs Kappa Architecture

AspectLambda (Batch + Stream)Kappa (Stream Only)
ComplexityHigh — two parallel pipelines to maintainLower — single stream pipeline
Code duplicationLogic often duplicated in batch and streamSingle codebase
AccuracyBatch layer guarantees correctnessRelies on stream reprocessing for corrections
ReprocessingBatch naturally reprocesses everythingReplay events from start (can be slow for years of data)
Use caseNeed guaranteed accurate aggregates + real-timeReal-time is sufficient, can tolerate replay for corrections

Production Example

LinkedIn: Real-time analytics use Lambda Architecture — batch layer (Hadoop) computes accurate daily metrics, speed layer (Kafka Streams + Samza) provides real-time counters for active sessions. Serving layer merges both for dashboard that shows "today's views" (stream) alongside "last 30 days" (batch).

Key Tip

Mention Lambda Architecture for analytics systems: "We run batch (Spark, nightly) for accurate historical metrics and stream (Flink) for real-time dashboards. The serving layer merges both views." If the interviewer asks about complexity, suggest Kappa as the simpler alternative when reprocessing from Kafka replay is acceptable.

4.7.3 OLTP vs OLAP

Critical

What they are: Two fundamentally different workload types that drive database and architecture choices.

AspectOLTP (Online Transaction Processing)OLAP (Online Analytical Processing)
PurposeRun the business (transactions)Analyze the business (insights)
OperationsINSERT, UPDATE, DELETE (single rows)SELECT with aggregations (millions of rows)
QueriesSimple, known in advance, indexedComplex, ad-hoc, full scans
LatencyMilliseconds (user-facing)Seconds to minutes (analyst-facing)
SchemaNormalized (3NF), many tablesDenormalized (Star/Snowflake schema)
ConcurrencyThousands of concurrent usersTens of concurrent analysts
Storage engineRow-oriented (PostgreSQL, MySQL)Column-oriented (ClickHouse, Redshift, BigQuery)
Data freshnessReal-time (current state)Near-real-time to batch (historical)
ExamplesOrder placement, user signup, paymentRevenue dashboard, funnel analysis, reporting
// Why column-oriented storage is fast for OLAP:
// Row store: reads entire row (100 columns) to get 3 columns → wasted I/O
// Column store: reads only the 3 columns needed → minimal I/O

// Example: "What is total revenue by country for 2024?"
// Row store: scan ALL columns of ALL rows, extract country + revenue
// Column store: scan ONLY country column + revenue column (highly compressible)

// Star Schema for OLAP:
//   fact_orders (order_id, customer_key, product_key, date_key, amount)
//        ↓              ↓              ↓            ↓
//   dim_customer   dim_product    dim_date
//   (name, country) (title, cat)  (year, quarter, month)

Trade-off: Row Store vs Column Store

Row stores (PostgreSQL): fast for point lookups and transactional writes (entire row in one disk read). Column stores (ClickHouse, Redshift): fast for aggregations over millions of rows (reads only needed columns, better compression). Never use a column store for OLTP or a row store for heavy analytics.

Production Example

Uber: OLTP (MySQL) handles ride transactions in real-time. OLAP (Apache Pinot) powers the internal analytics dashboard — queries like "average ride duration by city this week" scan billions of rows in seconds using columnar storage + pre-aggregated indexes.

Key Tip

When designing systems with both transactional AND analytical needs: "OLTP database (PostgreSQL) handles user-facing operations. CDC streams changes to the OLAP warehouse (ClickHouse/BigQuery) for analytics. This separation lets each optimize for its workload without interference."

4.7.4 ETL (Extract, Transform, Load)

Data

What it does: Move data from operational systems into analytical systems. Extract from sources, Transform (clean, aggregate, join), Load into destination (warehouse/lake).

PatternFlowWhen to UseTools
ETLExtract → Transform → LoadData needs cleaning before loading, warehouse has strict schemaApache Spark, Airflow, dbt, Informatica
ELTExtract → Load → TransformWarehouse is powerful enough to transform (BigQuery, Snowflake), schema-on-readFivetran + dbt, Snowflake, BigQuery
Streaming ETLContinuous Extract → Transform → LoadNear-real-time analytics neededKafka Connect + Flink/Kafka Streams
// Modern ELT pipeline example:
// 1. Extract: Fivetran/Airbyte pulls data from PostgreSQL, Stripe, Salesforce
// 2. Load: Raw data dumped into Snowflake staging tables (schema-on-read)
// 3. Transform: dbt models clean, join, and aggregate into analytics tables
//
// dbt model example (SQL-based transform):
// models/marts/revenue_by_country.sql
// SELECT
//   c.country,
//   DATE_TRUNC('month', o.created_at) as month,
//   SUM(o.amount) as total_revenue,
//   COUNT(DISTINCT o.customer_id) as unique_customers
// FROM {{ ref('stg_orders') }} o
// JOIN {{ ref('stg_customers') }} c ON o.customer_id = c.id
// GROUP BY 1, 2

Trade-off: ETL vs ELT

ETL: transform before loading (data arrives clean, warehouse stays tidy) but pipeline is complex and slow to iterate. ELT: load raw data first, transform in-place (faster iteration, schema-on-read) but warehouse accumulates raw junk. Modern trend favors ELT with tools like dbt for in-warehouse transformations.

Production Example

Airbnb: Uses Apache Airflow (which they created) to orchestrate 50,000+ daily ETL jobs. Data flows from production PostgreSQL → S3 (raw) → Spark transforms → Hive/Presto tables → Superset dashboards. DAG-based scheduling ensures jobs run in dependency order with automatic retries.

Key Tip

When asked about data pipelines: "We use ELT — Fivetran syncs raw data to Snowflake, then dbt models transform it into analytics-ready tables. This decouples ingestion from transformation, letting analysts iterate on transforms without re-running expensive extracts."

4.7.5 Data Lake vs Data Warehouse vs Lakehouse

Storage
AspectData WarehouseData LakeData Lakehouse
Data typeStructured onlyStructured + Semi-structured + UnstructuredAll types (unified)
SchemaSchema-on-write (defined upfront)Schema-on-read (raw storage)Schema enforcement optional
Storage formatProprietary (columnar)Open files (Parquet, JSON, CSV)Open table formats (Delta Lake, Iceberg, Hudi)
Query performanceFast (optimized, indexed)Slow without optimizationFast (indexing + caching on open formats)
CostExpensive (compute + storage coupled)Cheap (object storage like S3)Moderate (cheap storage + compute on demand)
ACID transactionsYesNo (append-only)Yes (Delta Lake/Iceberg add transactions)
Use caseBI dashboards, SQL analyticsML training data, raw logs, data scienceUnified analytics + ML on one platform
ExamplesSnowflake, BigQuery, RedshiftS3 + Athena, HDFS, Azure Data LakeDatabricks (Delta Lake), Apache Iceberg on S3

Key Tip

When discussing data architecture, mention the modern trend: "Companies are moving from separate data lakes and warehouses to lakehouses. A lakehouse gives you the cheap, scalable storage of a data lake (S3/Parquet) with the ACID transactions and query performance of a warehouse (via Delta Lake or Iceberg). This eliminates the need to copy data between systems." This shows awareness of current industry direction.

Trade-off: Separate vs Unified Architecture

Separate lake + warehouse: mature tooling but data copied between systems (expensive, stale). Unified lakehouse: single copy of data, no movement, but newer technology (Delta Lake/Iceberg maturity). If starting fresh in 2024+, default to lakehouse. Legacy systems may keep both during migration.

Production Example

Netflix: Runs a Lakehouse on Apache Iceberg + S3. All data (structured analytics, semi-structured logs, unstructured ML features) lives in one S3-backed system. Both Spark batch jobs and Trino interactive queries access the same tables — no ETL between lake and warehouse needed.
