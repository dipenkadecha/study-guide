4.10Scalability Techniques

4.10.1 Horizontal vs Vertical Scaling

Foundation
AspectHorizontal (Scale Out)Vertical (Scale Up)
ApproachAdd more machines/nodesAdd more CPU/RAM to existing machine
Cost ModelLinear, commodity hardwareExponential, specialized hardware
ComplexityHigher (distributed coordination)Lower (single machine)
LimitVirtually unlimitedHardware ceiling
Fault ToleranceBuilt-in (redundancy)Single point of failure
ExamplesKubernetes pods, Cassandra nodes, Kafka brokersLarger RDS instance, more RAM

Production Example

Instagram: Started on a single Django server (vertical scaling). At 25M users, switched to horizontal scaling — 12 Django app servers behind HAProxy. Now runs 1000+ instances on Kubernetes. Database remained vertically scaled (PostgreSQL on largest available instance) until sharding was introduced at 1B posts.

Trade-off: Scale Up vs Scale Out

Vertical (bigger machine): simple, no distributed complexity, but has a ceiling (largest available instance) and is a SPOF. Horizontal (more machines): theoretically unlimited, fault-tolerant, but introduces distributed systems complexity (consistency, partitioning, network failures). Start vertical, go horizontal when you hit the ceiling or need fault tolerance.

Key Tip

Show pragmatism: "For the first version, I'd vertically scale the database (RDS r6g.16xlarge handles most loads up to 100K QPS). When we outgrow that, we shard horizontally." Interviewers appreciate knowing WHEN to add complexity rather than defaulting to the most complex architecture from the start.

4.10.2 Auto-Scaling Strategies

Foundation
// Reactive Auto-Scaling: scale based on current metrics
// - CPU utilization > 70% -> add pods
// - Request queue depth > 1000 -> add consumers

// Predictive Auto-Scaling: scale based on patterns
// - Historical traffic patterns (peak at 9am, scale at 8:45am)
// - Scheduled scaling for known events (Black Friday)

// Key metrics for scaling decisions:
// 1. CPU/Memory utilization (resource-based)
// 2. Request rate / concurrent connections (traffic-based)
// 3. Queue depth / processing lag (queue-based)
// 4. Custom business metrics (orders/second)
→ A full Kubernetes HPA manifest (CPU + custom metrics + scale-up/down stabilization windows) is on the Cloud & Infrastructure page (4.14.1 Kubernetes Scaling).

Production Example

Airbnb: Uses Kubernetes HPA with custom metrics (requests per pod + booking queue depth). During New Year's Eve search spikes, scales from 200 to 2000 pods in under 2 minutes. Scales down overnight to save 60% compute cost. KEDA handles event-driven scaling for async booking processors.

Trade-off: Reactive vs Predictive Scaling

Reactive (scale on current metrics): simple but has cold-start lag — traffic spike hits before new instances are ready. Predictive (scale on forecasted load): pre-provisions capacity but wastes resources on wrong predictions. Best practice: predictive for known patterns (daily peaks) + reactive for unexpected spikes, with aggressive scale-up and conservative scale-down.

Key Tip

Specify scaling triggers: "HPA scales on CPU (target 70%) for compute-bound services and on custom metrics (queue depth, p99 latency) for I/O-bound services. Scale-up threshold is aggressive (1 minute), scale-down is conservative (5 minutes) to avoid flapping during bursty traffic."

4.10.3 Stateless vs Stateful Services

Foundation
AspectStatelessStateful
DefinitionNo server-side session; all state in request or external storeServer maintains client state between requests
ScalingTrivial (any instance handles any request)Complex (sticky sessions or state replication)
Load BalancingRound-robin, randomSession affinity, consistent hashing
RecoveryInstant (just spin up new instance)Requires state recovery from replicas/logs
ExamplesREST APIs, Lambda functionsWebSocket servers, game servers, databases
// Making services stateless - externalize state:
// 1. Session -> Redis (Spring Session + Redis)
// 2. File uploads -> S3/object storage
// 3. Cache -> Distributed cache (Redis, Memcached)
// 4. Configuration -> Config server (Spring Cloud Config)

// Benefits: easier deployment, zero-downtime scaling, simplified failover

4.10.4 Partitioning Strategies

Foundation
StrategyDescriptionProsCons
Range-BasedPartition by key ranges (A-M, N-Z)Range queries efficient, simpleHot spots if distribution uneven
Hash-BasedHash key to determine partitionEven distributionRange queries require scatter-gather
Consistent HashingHash ring with virtual nodesMinimal redistribution on add/removeMore complex implementation
GeographicPartition by region/locationData locality, compliance (GDPR)Cross-region queries expensive
Time-BasedPartition by time periodEasy archival, hot/cold separationRecent partition may be hot

Production Example

Kafka: Topics partitioned by message key (e.g., user_id). All events for one user land on the same partition — guarantees ordering per user. Adding partitions increases parallelism (more consumer instances), but requires careful key selection to avoid hot partitions (celebrity users).

Key Tip

Always explain your partition key choice: "I partition by user_id — keeps one user's data co-located for efficient queries. The hash distribution ensures even load across shards. For the 0.01% of power users with disproportionate data, we'd use a composite key (user_id + date bucket) to prevent hot partitions."
