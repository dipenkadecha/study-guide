4.18Common System Design Problems

4.18.1 The 5-Question Pattern Recognition Framework

Strategy

When an interviewer says "Design X," don't jump into components. Ask yourself these 5 questions in order — they generate the skeleton of any system design answer by identifying which patterns you need:

#QuestionIdentifiesPatterns to Consider
1How does data flow?Communication patternsRequest-Response (sync APIs), Message Queues (async), Pub/Sub (fan-out), WebSockets (real-time), Webhooks (push notifications), SSE (live feeds)
2How is data stored?Storage patternsPrimary-Replica (read scaling), Sharding (write scaling), WAL (durability), Event Sourcing (audit), CQRS (different read/write models)
3How is data accessed fast?Caching patternsCache-Aside (general), Write-Through (consistency), Write-Behind (write-heavy), CDN (static), Multi-level (L1+L2)
4How does it survive failures?Reliability patternsCircuit Breaker, Retry + Backoff, Bulkhead, Timeout, Idempotency, DLQ, Graceful Degradation
5How does it grow?Scaling patternsHorizontal Scaling, Sharding, CDN, Auto-scaling, Connection Pooling, Consistent Hashing

4.18.3 URL Shortener (Bitly)

Classic

Capacity Estimation (do this first — interviewers expect it)

Assume: 100M new URLs/day, read:write = 100:1, links kept 5 years.

Writes/sec  = 100M / 86,400s        ~= 1,160 writes/sec  (~1.2K)
Reads/sec   = 1,160 * 100           ~= 116K reads/sec     (peak ~2x = ~230K)

Storage/record: shortCode(7) + longURL(~500B) + metadata(~100B) ~= ~600B
New data/day = 100M * 600B          ~= 60 GB/day
5-year store = 60GB * 365 * 5       ~= ~110 TB  (with replication x3 -> ~330 TB)

Total URLs in 5y = 100M * 365 * 5   ~= 182B  ->  fits easily in 62^7 (3.5T)

Cache (80/20): cache the hot 20% of one day's reads.
Working set ~= 20% * 100M * 600B    ~= ~12 GB hot set -> a few Redis nodes
Read bandwidth = 116K * 600B        ~= ~70 MB/s out

What the numbers tell you: it's read-heavy (→ cache-aside + read replicas), writes are modest (a single primary can take ~1.2K/s), and storage is large-but-cheap (→ a distributed KV store like Cassandra). Always convert per-day to per-second, then size cache from the read QPS and the 80/20 rule.

  • ID Generation: Base62 encoding of auto-increment ID or random. 7 chars = 62^7 = 3.5 trillion URLs.
  • Storage: Cassandra (write-heavy, simple key-value lookup, distributed). Key=shortURL, Value=longURL+metadata.
  • Caching: Redis for hot URLs (80/20 rule - 20% of URLs get 80% of traffic). Cache-aside pattern.
  • Redirect: 301 (permanent, browser caches) vs 302 (temporary, allows analytics tracking). Use 302 for analytics.

Trade-off: ID Generation

  • Counter-based (Snowflake/ZooKeeper): Sequential, no collision. Predictable (security concern). Requires coordination.
  • Hash (MD5/SHA256 truncated): No coordination. Collision possible (check+retry). Unpredictable.
  • Pre-generated pool: Pre-compute IDs, store in DB. No collision, no coordination at runtime. Complexity in pool management.

Production Example

Bitly: Handles 28 billion clicks/month. Uses Base62 encoding (a-zA-Z0-9) for short codes. Pre-generates IDs in batches (counter ranges) to avoid distributed coordination. Read:write ratio is 100:1, so aggressive caching of popular short links in Redis achieves sub-millisecond redirect latency.

Interview Tip

Start with API design (shorten + redirect), then discuss: "Key challenge is unique ID generation at scale. I'd use a counter-based approach with range allocation (each server gets 1M IDs) rather than random to avoid collisions. 7 chars in Base62 = 3.5 trillion combinations — sufficient for decades."

4.18.4 Chat System (WhatsApp/Slack)

Real-time
  • Protocol: WebSocket for persistent bidirectional connection. Fallback: long polling for unreliable networks.
  • Message ordering: Per-conversation ordering using Kafka partition key = conversation_id.
  • Delivery guarantees: Sent -> Delivered (recipient's device ACKs) -> Read (user opens chat).
  • Storage: Cassandra with partition key = (chat_id), clustering key = (message_timestamp). Efficient range queries per chat.
  • Presence: Heartbeat every 30s to Redis. Absent for 60s = offline. Fan-out presence to friends list.
  • Group messages: Fan-out on write (small groups) vs fan-out on read (large groups/channels).

Trade-off: Fan-out on Write vs Read

  • Fan-out on Write (push): Write message to each recipient's inbox. Fast reads, expensive writes. Good for: small groups, 1:1 chat.
  • Fan-out on Read (pull): Store once, recipients query on load. Cheap writes, slow reads. Good for: large channels (1000+ members), celebrities.
  • Hybrid: WhatsApp uses push for small groups, pull for broadcast lists.

Production Example

WhatsApp: 2 billion users, 100 billion messages/day. Uses Erlang for connection management (each server handles 2M concurrent connections). Messages stored in Mnesia (write-optimized). End-to-end encrypted — server can't read message content. Delivery receipts (single tick, double tick, blue tick) are separate lightweight messages.

Interview Tip

Key design points: "WebSocket for persistent connections. Messages stored in Cassandra (partition by chat_id, cluster by timestamp). Offline users get messages queued — delivered when they reconnect. Group messages use fan-out-on-write (max 256 members for WhatsApp groups). For presence: heartbeat every 30s, shown as online/last seen."

4.18.5 News Feed (Instagram/Twitter)

Scale
  • Post creation: User posts -> store in Posts DB -> fan-out to followers' feed caches
  • Feed retrieval: User opens app -> read pre-computed feed from Redis/cache -> rank/filter
  • Celebrity problem: User with 100M followers = 100M writes on each post (too slow)
  • Solution: Hybrid: fan-out on write for normal users (<5K followers), fan-out on read for celebrities
// Feed fanout service (simplified)
public void onNewPost(Post post) {
    List<String> followers = followService.getFollowers(post.authorId());

    if (followers.size() > CELEBRITY_THRESHOLD) { // > 5000 followers
        // Celebrity: don't fan out, merge at read time
        celebrityPostsCache.add(post);
    } else {
        // Normal user: push to each follower's feed cache
        for (String followerId : followers) {
            feedCache.prepend("feed:" + followerId, post.id());
            feedCache.trim("feed:" + followerId, MAX_FEED_SIZE); // Keep last 1000
        }
    }
}

// Feed retrieval: merge cached feed + celebrity posts + rank
public List<Post> getFeed(String userId) {
    List<String> cachedPostIds = feedCache.getRange("feed:" + userId, 0, 50);
    List<Post> posts = postService.getByIds(cachedPostIds);

    // Merge celebrity posts user follows
    List<String> celebrities = followService.getCelebritiesFollowed(userId);
    posts.addAll(celebrityPostsCache.getRecent(celebrities));

    return rankingService.rank(posts, userId); // ML ranking
}

Trade-off

Fan-out-on-write (precompute feeds, fast reads, expensive writes for users with many followers) vs Fan-out-on-read (compute at read time, cheaper writes, slower reads). Hybrid: fan-out-on-write for normal users, fan-out-on-read for celebrities (>1M followers). This avoids writing to millions of feed caches when a celebrity posts.

Production Example

Instagram: Uses hybrid fan-out. Regular users' posts are pushed to all followers' feed caches (Redis). For accounts with 10M+ followers, posts are pulled at read time and merged. Feed ranking model considers: relationship (interactions), timeliness, content type, and predicted engagement.

Interview Tip

Always mention the hybrid approach: "For the 99% of users with <10K followers, fan-out-on-write gives instant feed reads. For the top 1% celebrities, fan-out-on-read avoids writing to millions of caches per post. At read time, we merge the precomputed feed with celebrity posts and rank by engagement score."

4.18.6 File Storage (Dropbox/Google Drive)

Storage
  • Chunking: Split files into 4MB chunks. Only upload changed chunks (deduplication + bandwidth saving).
  • Metadata DB: PostgreSQL - file tree, permissions, versions. Separate from file content.
  • Block storage: S3 for actual chunks. Content-addressable (chunk hash = key).
  • Sync: Client computes diff, uploads only changed chunks. Server notifies other clients via WebSocket.
  • Conflict resolution: Last-writer-wins with conflict copies (Dropbox approach). Or OT/CRDT for real-time collab (Google Docs).

Trade-off: Chunking Strategy

  • Fixed-size chunks (4MB): Simple, good for large files. Problem: small edit = re-upload entire chunk.
  • Content-defined chunking (CDC/Rabin fingerprint): Chunks break at content-defined boundaries. Insert in middle only affects 1-2 chunks. Better dedup. More complex.
  • Dropbox uses CDC for efficient sync of large files with small modifications.

Production Example

Dropbox: Files split into 4MB chunks, content-addressed by SHA-256. Deduplication across all users — if 100 users upload the same PDF, stored once. Block-level sync: changing 1 byte in a 1GB file only uploads one 4MB chunk. Metadata in MySQL (sharded by namespace_id), blocks in a custom storage system (Magic Pocket on bare metal).

Interview Tip

Key design decisions: "Chunk files into fixed-size blocks (4MB). Each block content-addressed (SHA-256 hash). Client computes hashes locally, uploads only new/changed blocks — enables efficient sync. Metadata service (what files exist, their block lists) separated from block storage (actual bytes). Conflict resolution: last-writer-wins with conflict copy for simultaneous edits."

4.18.7 Video Streaming (Netflix)

Scale
  • Transcoding: Encode each video in multiple resolutions (240p-4K) and codecs (H.264, VP9, AV1). Segment into 2-10 second chunks for adaptive streaming.
  • Adaptive Bitrate (ABR): Client measures bandwidth, requests appropriate quality segment. Seamless quality switching mid-playback.
  • CDN: Netflix Open Connect - ISP-embedded servers. 95% of traffic served from CDN edge. Predictive pre-positioning based on trending content.
  • Storage: S3 for master copies. CDN edge for popular content. Cold storage for rarely-watched titles.

Trade-off: Transcoding Strategy

  • Pre-transcode all formats: Faster playback start, huge storage cost (1 movie = 100+ files). Netflix approach for popular content.
  • Just-in-time transcoding: Transcode on request. Saves storage, higher latency. Good for long-tail content.
  • Netflix hybrid: Pre-transcode popular titles in all formats. JIT for old/rare content. Per-title encoding (optimize bitrate ladder per video complexity).

Production Example

Netflix: Encodes each title in 1200+ variants (resolution x bitrate x codec x audio). Open Connect CDN (custom hardware in 1000+ ISP locations) serves 95% of traffic from within the user's ISP network. Adaptive bitrate streaming (DASH) measures bandwidth every 2s and switches quality seamlessly. Pre-positions popular titles overnight to edge locations.

Interview Tip

Cover the pipeline: "Upload → async transcoding pipeline (FFmpeg, multiple resolutions/codecs) → store chunks in S3 → distribute to CDN edge. Playback: manifest file lists available qualities, client picks based on bandwidth. Adaptive streaming adjusts mid-playback. Key metric: rebuffer rate (should be <0.5%). Pre-position predicted popular content at edge."

4.18.8 Ride Sharing (Uber)

Real-time
  • Location indexing: Geohash or H3 (Uber's hexagonal grid). Drivers update location every 4s. Store in Redis with geospatial commands (GEOADD, GEORADIUS).
  • Matching algorithm: Find nearby available drivers (GEORADIUS 2km), rank by ETA, send request with timeout. If declined/timeout, next driver.
  • Surge pricing: Supply/demand ratio per geohash cell. Low supply + high demand = multiplier. Recalculated every 1-2 minutes.
  • ETA calculation: Graph-based routing (Dijkstra/A*) with real-time traffic data. Pre-computed for common routes, live for others.

Trade-off: Geospatial Indexing

  • Geohash: String prefix-based (nearby = same prefix). Simple, works with Redis. Edge cases at boundaries (neighbors can have different prefixes).
  • Quadtree: Adaptive resolution (dense areas = deeper tree). Better for non-uniform distribution. More complex implementation.
  • H3 (Uber): Hexagonal grid, uniform neighbors, no edge distortion. Best for ride-sharing. Library available.

Production Example

Uber: Geospatial matching uses Google S2 cells (Earth divided into ~400M cells at level 12). Driver locations updated every 4 seconds via persistent connection. Matching algorithm considers: distance, ETA (routing), driver rating, trip acceptance probability. Surge pricing uses real-time supply/demand ratio per S2 cell — computed every 2 minutes.

Interview Tip

Focus on matching: "Drivers send GPS coordinates every 4s. We index by geohash (S2 cells) in Redis. When rider requests: query nearby cells, compute ETA for top-20 candidates (routing service), pick optimal match. Key challenge: consistency during matching — use distributed locks per driver to prevent double-dispatch."

4.18.9 Typeahead / Autocomplete (Google Search)

Low Latency
  • Data collection: User searches -> Kafka -> aggregation pipeline -> update suggestion DB periodically
  • Serving: Trie stored in memory across multiple nodes. Prefix -> top-K suggestions (pre-computed).
  • Latency target: < 100ms response (user types between keystrokes ~200ms)
// Trie with pre-computed top-K per node
class TrieNode {
    Map<Character, TrieNode> children;
    List<String> topSuggestions; // Pre-computed top 10 for this prefix
    // Updated offline (every 15 min from analytics pipeline)
}

// Query flow:
// 1. Client sends prefix after 2+ chars (debounced 100ms)
// 2. Route to correct shard (first 2 chars determine shard)
// 3. Trie lookup: traverse prefix -> return topSuggestions at final node
// 4. Response time: O(prefix_length) = ~O(1) for short prefixes

// Ranking factors for suggestions:
// - Search frequency (most popular)
// - Recency (trending queries boosted)
// - Personalization (user's past searches)
// - Geography (location-specific suggestions)

Trade-off: Trie vs Elasticsearch vs Redis

  • In-memory Trie: Fastest (<1ms), pre-computed results. Memory intensive. Best for: prefix-only, high QPS.
  • Elasticsearch: Flexible (fuzzy matching, typo tolerance). Higher latency (~10ms). Best for: full-text, complex scoring.
  • Redis Sorted Sets: Simple, distributed. ZRANGEBYLEX for prefix. Moderate latency. Best for: small dataset, simple ranking.

Production Example

Google: Trie built from billions of queries. Updated incrementally (new trending queries propagated in minutes). Personalized layer overlays user's search history. Ranking considers: query frequency, freshness, user history, language, and location. Infrastructure: entire trie sharded across 100s of machines, each prefix maps to specific shards.

Interview Tip

Explain the data structure: "Trie (prefix tree) where each node stores top-K suggestions. Pre-computed and cached at each level. Trie sharded by prefix (a-m on shard 1, n-z on shard 2). Updates: batch rebuild nightly for frequency changes, real-time path for trending queries. Response must be <100ms — serve from memory, never disk."

4.18.10 Ticket Booking (BookMyShow/Ticketmaster)

Concurrency
  • Problem: 100K users trying to book same 50K seats simultaneously. Must prevent overselling.
  • Seat locking: Temporary hold (10 min) when user selects seat. Release if payment not completed.
  • Distributed lock vs DB lock: Redis lock for seat hold (fast, auto-expire). DB constraint as final safety net.
// Seat booking flow with distributed locking
public BookingResult bookSeat(String eventId, String seatId, String userId) {
    String lockKey = "seat_lock:" + eventId + ":" + seatId;

    // 1. Try to acquire seat lock (10 min TTL)
    boolean locked = redis.set(lockKey, userId, SetParams.nx().ex(600));
    if (!locked) {
        String holder = redis.get(lockKey);
        if (userId.equals(holder)) return BookingResult.alreadyHeld();
        return BookingResult.unavailable(); // Another user holds it
    }

    try {
        // 2. Verify seat still available in DB (race condition safety)
        Seat seat = seatRepo.findByIdForUpdate(seatId); // SELECT FOR UPDATE
        if (seat.status() != AVAILABLE) {
            redis.del(lockKey);
            return BookingResult.unavailable();
        }

        // 3. Mark as held, start payment timer
        seat.setStatus(HELD);
        seat.setHeldBy(userId);
        seat.setHeldUntil(Instant.now().plusSeconds(600));
        seatRepo.save(seat);

        return BookingResult.held(seatId, Duration.ofMinutes(10));
    } catch (Exception e) {
        redis.del(lockKey); // Release on failure
        throw e;
    }
}

// Background job: release expired holds every minute
@Scheduled(fixedRate = 60000)
public void releaseExpiredHolds() {
    seatRepo.releaseExpired(Instant.now()); // UPDATE seats SET status='AVAILABLE' WHERE held_until < NOW()
}

Trade-off: Booking Strategies

  • Pessimistic locking (SELECT FOR UPDATE): Strong consistency, no overselling. Blocks other transactions. Use for: high-value events, limited seats.
  • Optimistic locking (version column): Non-blocking, retry on conflict. Higher throughput. Use for: general seating, acceptable retry rate.
  • Queue-based (virtual waiting room): Serialize access, fair ordering. Higher latency. Use for: extremely high demand (Taylor Swift tickets).

Production Example

Ticketmaster: Handles 1.5M+ concurrent users for popular events. Uses queue system (virtual waiting room) to control load — 100K users see "selecting seats" at a time. Seat holds: pessimistic locking with 10-minute timeout. Payment processed within hold window. If timeout expires, seat released back to available pool automatically.

Interview Tip

Key challenge is inventory consistency: "Seats use pessimistic locking — SELECT FOR UPDATE. When user selects seat, it's held for 10 minutes (timer starts). Concurrent requests for same seat get "unavailable." If payment fails or timer expires, seat releases. For hot events: virtual queue (CloudFlare Waiting Room) prevents thundering herd. Show seat map from a cache with near-real-time availability."

4.18.11 Design a Web Crawler

Practice

Requirements: Crawl billions of web pages, extract content, handle politeness (robots.txt), detect duplicates, scale horizontally.

ComponentPurposeKey Pattern
URL FrontierPriority queue of URLs to crawlPriority Queue + Consistent Hashing (partition by domain)
DNS ResolverResolve domains (cache heavily)Cache-Aside (DNS cache)
Fetcher WorkersDownload pages in parallelHorizontal Scaling + Rate Limiting (per domain)
Content DedupDetect duplicate pagesBloom Filter (seen URLs) + SimHash (content fingerprint)
Content StoreStore raw HTML + extracted textObject Storage (S3) + NoSQL metadata

Key trade-off: BFS (breadth-first) discovers more domains quickly but misses deep pages; priority-based crawling focuses on important pages but may never reach long-tail content. Politeness delays (1-2s per domain) limit throughput per domain — parallelize across many domains instead.

Trade-off

BFS vs DFS crawling: BFS (level-by-level) discovers important pages first (homepage links) and distributes load evenly. DFS goes deep into one domain (risk of getting stuck in infinite traps). Hybrid: BFS across domains + priority queue weighted by page importance (PageRank) and freshness (time since last crawl).

Production Example

Google: Crawls 100+ billion pages. Uses a URL frontier (priority queue) with politeness constraints (max 1 request/second per domain). DNS pre-fetching and connection pooling for throughput. Duplicate detection via SimHash (content fingerprint) — avoids recrawling mirror sites. Crawl budget allocated proportionally to site quality.

Interview Tip

Structure your answer: "1) Seed URLs → URL frontier (priority queue). 2) Fetcher pool respects politeness (robots.txt, rate limit per domain). 3) Content parser extracts links + text. 4) URL dedup via Bloom filter (seen before?). 5) Content dedup via SimHash. Scaling: partition frontier by domain hash across workers. Key metrics: pages/second, coverage, freshness."

4.18.12 Design a Payment System

Practice

Requirements: Process payments reliably, prevent double-charging, handle refunds, support multiple payment methods, PCI compliance.

ComponentPurposeKey Pattern
Payment ServiceOrchestrate payment flowSaga (orchestration) for multi-step process
Idempotency StorePrevent duplicate chargesIdempotency Key (client-generated UUID per request)
LedgerImmutable record of all transactionsEvent Sourcing + WAL (append-only, auditable)
Payment GatewayInterface to Stripe/Visa/PayPalCircuit Breaker + Retry with Backoff
ReconciliationVerify internal records match bank statementsBatch Processing (nightly comparison)

Key trade-off: Synchronous processing gives instant feedback but risks timeout failures mid-payment. Async (queue-based) is more reliable but user waits for confirmation. Solution: accept payment intent synchronously, process asynchronously, notify via webhook/push.

Trade-off

Synchronous vs Asynchronous payment processing: Sync gives immediate confirmation but blocks user (and timeout = unknown state). Async (webhook callback) is more resilient but user must wait/poll. Best practice: optimistic response ("payment processing") + async confirmation. For refunds: always async — batch processing reduces costs.

Production Example

Stripe: Uses double-entry bookkeeping — every money movement creates two entries (debit + credit). Idempotency keys on every API call prevent duplicate charges. Payment state machine: created → processing → succeeded/failed. Reconciliation job runs hourly comparing internal ledger with bank statements — discrepancies trigger alerts.

Interview Tip

Critical points: "Idempotency key on every payment request (UUID from client). State machine: pending → authorized → captured → settled. Use distributed transactions (Saga) for multi-step flows (charge + update order + send notification). Every money movement is double-entry in the ledger. Reconciliation job compares our records vs payment gateway daily."

4.18.13 Design a Notification System

Practice

Requirements: Send push/email/SMS notifications at scale, support user preferences, handle templates, ensure delivery, respect rate limits.

ComponentPurposeKey Pattern
Notification ServiceAPI to trigger notificationsAPI Gateway + Rate Limiting
Priority QueueRoute by urgency (OTP > marketing)Priority Message Queue (critical > normal > bulk)
Channel WorkersSend via push/email/SMS providersFan-out + Circuit Breaker (per provider)
Template EngineRender personalized messagesCache-Aside (compiled templates)
Preference StoreUser opt-in/out, quiet hoursRead-through Cache + Database
DLQ + RetryHandle delivery failuresDead Letter Queue + Exponential Backoff

Key trade-off: Push notifications are real-time but unreliable (device offline). Email is reliable but slow. SMS is reliable and fast but expensive. Solution: use push as primary, fall back to email/SMS for critical messages, deduplicate across channels.

Trade-off

Push vs Pull: Push (server sends to device) for time-sensitive notifications (messages, alerts). Pull (client polls) for non-urgent (digest emails). Push risks overwhelming devices (notification fatigue). Best practice: priority levels — P0 (push immediately), P1 (batch per 15 min), P2 (daily digest). Users control per-channel preferences.

Production Example

Facebook: Sends 10+ billion push notifications daily. Uses a priority queue with rate limiting per user (max 5 pushes/hour to prevent fatigue). Multi-channel: push notification → if unread after 4 hours → email → if still unread after 24 hours → SMS. ML model predicts optimal delivery time per user to maximize engagement.

Interview Tip

Design template: "Event triggers notification → message queue (Kafka) → priority router → channel-specific workers (push/SMS/email). Each channel has its own adapter (APNS for iOS, FCM for Android, SendGrid for email). Key features: user preferences (opt-out per channel), rate limiting (max N/hour), template engine (i18n), delivery tracking (sent/delivered/read)."

4.18.14 Design a Collaborative Editor (Google Docs)

Practice

Requirements: Real-time multi-user editing, conflict resolution, cursor presence, version history, offline support.

ComponentPurposeKey Pattern
Document ServerManage document state and operationsOT (Operational Transform) or CRDT
Real-time SyncPush changes to all editors instantlyWebSocket + Pub/Sub (per document channel)
Presence ServiceShow who's editing, cursor positionsHeartbeat + Pub/Sub (ephemeral state)
Version StoreFull edit history, point-in-time recoveryEvent Sourcing (every keystroke is an event)
Conflict ResolutionMerge concurrent edits without data lossCRDT (Yjs, Automerge) or OT (Google Docs original)

Key trade-off: OT requires a central server to order operations (simpler but single point of failure). CRDTs work peer-to-peer with no central coordination (resilient but larger state overhead and limited operation types). Google Docs uses OT; newer tools (Figma, Notion) increasingly use CRDTs.

Trade-off

OT (Operational Transform) vs CRDTs: OT requires central server to order operations — simpler, proven (Google Docs), but server is bottleneck. CRDTs are decentralized (work offline, peer-to-peer) but more complex and higher metadata overhead. OT for centralized real-time (Google Docs). CRDTs for offline-first or P2P (Figma canvas, local-first apps).

Production Example

Google Docs: Uses OT with a central server maintaining document version. Each keystroke becomes an operation (insert char at position X). Server transforms concurrent operations to maintain consistency. Cursor positions of all collaborators broadcast via presence channel (WebSocket). Undo stack is per-user (local operations only).

Interview Tip

Explain the core mechanism: "Each edit is an operation (insert/delete at position). When two users edit simultaneously, the server resolves conflicts via Operational Transform — adjusts positions based on concurrent edits. WebSocket for real-time sync (<50ms). Cursor presence via lightweight broadcast. Document stored as operation log — replay to reconstruct any version."

4.18.15 Design an E-Commerce Platform (Amazon)

Practice

Requirements: Product catalog, search, cart, checkout, inventory management, order tracking, recommendations.

ComponentPurposeKey Pattern
Product CatalogStore millions of products with attributesDatabase-per-Service (NoSQL for flexibility) + CDN (images)
SearchFull-text search with filters and facetsElasticsearch + Cache-Aside (popular queries)
Cart ServicePersistent shopping cart across sessionsSession affinity + Redis (fast reads) + eventual consistency
InventoryTrack stock levels, prevent oversellingOptimistic Locking (version check on decrement) + Event-driven sync
Order ServiceProcess checkout, coordinate payment + shippingSaga (orchestration): reserve inventory → charge payment → confirm order
Recommendation EnginePersonalized product suggestionsBatch Processing (collaborative filtering) + Cache-Aside

Key trade-off: Strong consistency on inventory prevents overselling but adds latency and reduces throughput. Solution: optimistic approach — allow checkout, validate inventory asynchronously, compensate (refund + notify) on rare conflicts. Amazon accepts ~0.1% oversell rate as acceptable vs blocking millions of checkouts for strict consistency.

Trade-off

Monolith vs Microservices for e-commerce: Start monolithic (faster development, simpler transactions for cart+inventory+payment). Split when: team grows >50 engineers, deploy conflicts increase, one component needs independent scaling. First split candidates: search (different scaling pattern), payments (compliance isolation), recommendations (ML pipeline).

Production Example

Amazon: Hundreds of microservices. Product page alone involves 100+ service calls (rendered in parallel). Cart service is eventually consistent — adding to cart always succeeds (optimistic), inventory check happens at checkout. "Add to Cart" is the most important button in e-commerce — it must never fail or show loading.

Interview Tip

Cover the key flows: "Product catalog (Elasticsearch for search + DynamoDB for product details). Cart service (Redis for session cart, persisted to DB on login). Checkout flow: reserve inventory (distributed lock, 10-min hold) → payment (idempotent charge) → order created (event published) → inventory deducted → fulfillment triggered. Handle: abandoned carts (release inventory), payment failures (compensating transaction), overselling (backorder queue)."

4.18.16 Design a Distributed Task Scheduler

Practice

Requirements: Schedule and execute tasks at specified times, handle recurring jobs, guarantee exactly-once execution, scale to millions of scheduled tasks.

ComponentPurposeKey Pattern
Scheduler ServiceAccept task submissions, store scheduleAPI Gateway + Sharding (partition by task hash)
Time Wheel / Priority QueueEfficiently find tasks due for executionHierarchical Time Wheel or Min-Heap sorted by trigger time
Worker PoolExecute tasks when triggeredHorizontal Scaling + Consumer Group (Kafka partitions)
Leader ElectionOne scheduler node scans and dispatchesLeader Election (ZooKeeper/etcd) to avoid duplicate dispatch
IdempotencyEnsure task runs exactly once even on retryIdempotency Key + Distributed Lock (fencing token)
Dead Letter QueueHandle permanently failing tasksDLQ + Alerting + Manual retry

Key trade-off: Pull-based (workers poll for due tasks) is simple but has latency jitter. Push-based (scheduler dispatches to workers) is precise but scheduler becomes a bottleneck. Hybrid: scheduler pushes to a Kafka topic partitioned by time window, workers consume from their partition.

Trade-off

Push vs Pull model: Push (scheduler assigns tasks to workers) gives better load balancing but scheduler is bottleneck/SPOF. Pull (workers pull from queue when ready) is simpler, self-balancing, but priority/ordering is harder. Hybrid: scheduler enqueues prioritized tasks, workers pull from priority queues. Exactly-once execution via distributed lock + fencing token per task.

Production Example

Uber (Peloton/Mesos): Schedules millions of tasks daily across 10,000+ machines. Tasks have priorities (P0 ride-matching > P1 ETL > P2 batch). Resource-aware scheduling: each task declares CPU/memory needs, scheduler bin-packs onto machines. Dead task detection via heartbeat (no heartbeat in 30s = reschedule). At-least-once with idempotent tasks.

Interview Tip

Structure: "Task submitted → persisted to DB (durable) → scheduler picks tasks by priority + scheduled_time → assigns to available worker (via queue). Worker sends heartbeat every 10s. If missed 3x, task re-queued. Completed tasks update state. Key challenges: exactly-once execution (idempotency key + fencing), clock skew for scheduled tasks (centralized time source), fair scheduling across tenants (weighted queues)."
