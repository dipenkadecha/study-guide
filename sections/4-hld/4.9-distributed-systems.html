4.9.1 Consensus Algorithms

Advanced
AlgorithmApproachLeaderUse Case
RaftLeader-based log replicationYes (elected)etcd, Consul, CockroachDB
PaxosProposal/accept phasesOptionalGoogle Spanner, Chubby
GossipEpidemic broadcastNo (peer-to-peer)Cassandra membership, SWIM
// Raft simplified: Leader Election + Log Replication
// 1. All nodes start as Followers
// 2. If no heartbeat from Leader in timeout -> become Candidate
// 3. Candidate requests votes from peers (majority wins)
// 4. Leader sends heartbeats + replicates log entries
// 5. Entry committed when majority of nodes have it
// Key: only committed entries are applied to state machine

Production Example

etcd (Kubernetes): Uses Raft consensus for cluster state. Every kubectl apply writes to etcd — requires majority (3 of 5 nodes) to acknowledge before confirming. If 2 nodes die, cluster remains operational. If 3 die, cluster becomes read-only (no quorum for writes).

Trade-off: Raft vs Paxos

Raft: easier to understand and implement (clear leader, sequential log), used by etcd, Consul. Paxos: more flexible (multi-decree, leaderless variants) but notoriously hard to implement correctly. In practice, choose Raft for new systems — correctness is easier to verify. Paxos for academic/research contexts or Google-scale (Chubby uses Multi-Paxos).

Key Tip

When discussing coordination services: "We use etcd (Raft-based) for leader election and distributed configuration. Raft guarantees that as long as a majority of nodes (3 of 5) are alive, the cluster remains operational and consistent." Mention the write penalty: every write requires majority acknowledgment.

4.9.2 Distributed Locking

Advanced
// Redis Distributed Lock (Redlock algorithm by Antirez)
// Simple version (single Redis):
public boolean acquireLock(String resource, String lockId, Duration ttl) {
    Boolean acquired = redisTemplate.opsForValue()
        .setIfAbsent("lock:" + resource, lockId, ttl);
    return Boolean.TRUE.equals(acquired);
}

public void releaseLock(String resource, String lockId) {
    // Lua script for atomic check-and-delete (prevent releasing someone else's lock)
    String script = "if redis.call('get',KEYS[1]) == ARGV[1] then " +
                    "return redis.call('del',KEYS[1]) else return 0 end";
    redisTemplate.execute(new DefaultRedisScript<>(script, Long.class),
        List.of("lock:" + resource), lockId);
}

// Production: use Redisson (Java) for robust implementation with:
// - Watchdog (auto-extend TTL while holding lock)
// - Fair locking (FIFO ordering)
// - RedLock (multiple Redis instances for fault tolerance)

Production Example

Redlock (Redis): Stripe uses Redis-based distributed locks for idempotent payment processing. Lock acquired across 5 Redis instances — if client gets lock on 3+, it proceeds. Fencing tokens prevent stale lock holders from making writes after their lock expired.

Trade-off: Redis Lock vs ZooKeeper Lock

Redis (Redlock): fast, simple, but relies on clock synchronization (unsafe if clocks drift). ZooKeeper/etcd: slower but uses sequential nodes + ephemeral sessions (no clock dependency). For critical sections (payment deduction), use ZooKeeper. For best-effort dedup (prevent duplicate emails), Redis is fine.

Key Tip

Always mention fencing tokens with distributed locks: "The lock includes a monotonically increasing fencing token. If a lock holder is slow and its lock expires, the new holder gets a higher token. Storage rejects writes with stale tokens — prevents the 'zombie process' problem where an expired lock holder still writes."

→ The fencing-token mechanism (with a step-by-step GC-pause example) is detailed on the Concurrency & Locking page (4.22.2 Distributed Lock with Fencing).

4.9.4 Distributed Rate Limiter

Advanced
// Distributed rate limiting with Redis - Sliding Window Counter
// Challenge: centralized counter across multiple app instances
public boolean isAllowed(String clientId, int maxRequests, Duration window) {
    String key = "ratelimit:" + clientId;
    long now = System.currentTimeMillis();
    long windowStart = now - window.toMillis();

    // Lua script for atomic operation (avoid race conditions)
    String luaScript = """
        redis.call('ZREMRANGEBYSCORE', KEYS[1], 0, ARGV[1])
        local count = redis.call('ZCARD', KEYS[1])
        if count < tonumber(ARGV[2]) then
            redis.call('ZADD', KEYS[1], ARGV[3], ARGV[3])
            redis.call('EXPIRE', KEYS[1], ARGV[4])
            return 1
        end
        return 0
    """;
    Long result = redisTemplate.execute(new DefaultRedisScript<>(luaScript, Long.class),
        List.of(key),
        String.valueOf(windowStart), String.valueOf(maxRequests),
        String.valueOf(now), String.valueOf(window.toSeconds()));
    return result == 1;
}
// Trade-off: Redis ZSET gives precise sliding window but O(log n) per request
// Alternative: Redis INCR + EXPIRE for fixed window (O(1) but boundary burst issue)

Production Example

Cloudflare: Rate limits 45M requests/second globally using distributed counters. Each edge node keeps a local counter synced via gossip protocol. Allows brief over-limit bursts (eventual consistency) rather than blocking legitimate traffic with strict global coordination.

Trade-off: ZooKeeper vs etcd vs Redis for Distributed Coordination

SystemConsensusUse ForLimitation
ZooKeeperZABLeader election, config, namingComplex ops model, JVM-based
etcdRaftK8s config, service discovery, locksSmaller datasets (<8GB)
RedisNone (single/sentinel)Caching, locks, rate limitingNot CP by default, data loss on failover

4.9.5 Vector Clocks

Advanced

What it does: Tracks causality in a distributed system by maintaining a vector of logical timestamps, one per node. When events are causally related, vector clocks identify the ordering. When events are concurrent (neither caused the other), vector clocks identify the conflict for resolution.

// Vector Clock Example with 3 nodes (A, B, C):
// Each node maintains a vector: [A_count, B_count, C_count]

// Initial state: A=[0,0,0], B=[0,0,0], C=[0,0,0]

// Node A writes "x=1": A increments own counter
//   A=[1,0,0]

// Node A sends to B: B merges (takes max of each position) + increments own
//   B = max([1,0,0], [0,0,0]) + increment B = [1,1,0]

// Node C writes "x=2" independently: C increments own counter  
//   C=[0,0,1]

// CONFLICT DETECTED: B=[1,1,0] vs C=[0,0,1]
// Neither dominates the other (B has higher A,B but C has higher C)
// These are CONCURRENT writes - application must resolve the conflict!

// Comparison rules:
// V1 dominates V2 (V1 happened-after V2) if ALL positions in V1 >= V2
// If neither dominates, events are concurrent (conflict!)

// Resolution strategies:
// 1. Last-Writer-Wins (LWW) - use physical timestamp to pick winner (data loss!)
// 2. Application-level merge - e.g., for shopping cart, take union of items
// 3. CRDTs - data structures that auto-resolve without coordination

When to use:

  • Conflict detection in eventually consistent systems (DynamoDB, Riak)
  • Multi-leader replication where concurrent writes can happen
  • Collaborative editing (detecting conflicting edits)

Trade-offs:

AspectVector ClocksLamport TimestampsPhysical Timestamps (LWW)
Detects concurrencyYes (main advantage)No (only total order)No
SizeO(N) per entry (N = nodes)O(1) single integerO(1) single timestamp
Clock sync neededNo (logical)No (logical)Yes (NTP drift = silent data loss)
Conflict resolutionApplication decidesN/A (no conflict info)Arbitrary winner (last write)
Used byRiak (deprecated), researchMany distributed DBs internallyCassandra, DynamoDB (simplified)

Key Tip

In practice, most production systems use simplified approaches: DynamoDB uses Last-Writer-Wins with server timestamps, Cassandra uses LWW per column. Full vector clocks are rarely used due to space overhead. But knowing them shows deep understanding of distributed causality. Mention: "Vector clocks solve the problem of detecting concurrent writes, but most systems trade correctness for simplicity with LWW."

4.9.6 Leader Election

Critical

What it does: In a cluster, one node is elected as the "leader" to coordinate work (writes, scheduling, decisions). Other nodes are followers. If the leader fails, a new one is elected.

// Leader Election approaches:
// 1. Bully Algorithm: highest-ID node wins (simple but slow recovery)
// 2. Raft consensus: nodes vote, majority wins, term-based (etcd, Consul)
// 3. External coordinator: ZooKeeper ephemeral nodes, etcd lease

// ZooKeeper leader election:
// - Each candidate creates an ephemeral sequential znode: /election/node_00001
// - Node with lowest sequence number becomes leader
// - If leader dies, ephemeral node auto-deleted → next lowest becomes leader

// Raft leader election (simplified):
// 1. All nodes start as followers with random election timeouts (150-300ms)
// 2. First to timeout becomes candidate, increments term, votes for self
// 3. Sends RequestVote to all peers
// 4. Wins if gets majority (N/2 + 1) votes
// 5. Leader sends periodic heartbeats to maintain authority
// 6. If heartbeat stops → followers timeout → new election

Trade-off: Single Leader vs Multi-Leader

Single leader: simple coordination, no conflicts, but leader is SPOF and bottleneck. Multi-leader: higher availability and write throughput but requires conflict resolution (LWW, vector clocks, CRDTs). Use single leader for coordination (schedulers, lock managers); multi-leader for geo-distributed writes (DynamoDB global tables).

Production Example

Kubernetes: The kube-controller-manager uses leader election via etcd lease. Only ONE controller-manager runs active at a time — others are standby. If the leader crashes, lease expires in 15s, and a new leader is elected automatically. Prevents duplicate scheduling of pods.

Key Tip

Use leader election for any "exactly-one" constraint: "The scheduler service uses leader election (etcd lease) — only one instance dispatches jobs at a time. If the leader dies, a new one takes over within the lease TTL (10-15s). This prevents duplicate job execution without distributed locking per job."

4.9.7 Heartbeat & Failure Detection

Critical

What it does: Nodes send periodic "I'm alive" signals to detect failures. If heartbeats stop, the node is suspected dead.

ApproachHow It WorksFalse Positive RiskUsed By
Fixed timeoutNo heartbeat in T seconds = deadHigh (network delays trigger false detection)Simple systems
Phi AccrualStatistical model of heartbeat intervals, outputs suspicion level (phi)Low (adapts to network conditions)Cassandra, Akka
SWIM ProtocolRandom probe + indirect probe via peers before declaring deadVery low (peer confirmation required)Serf, Memberlist
// Heartbeat configuration trade-off:
// Short interval (100ms): Fast detection but high network overhead + more false positives
// Long interval (30s): Low overhead but slow failure detection

// Typical production values:
// - Heartbeat interval: 1-5 seconds
// - Failure threshold: 3 missed heartbeats (3-15s detection time)
// - etcd: election timeout = 1000ms, heartbeat = 100ms

Trade-off: Fast Detection vs False Positives

Short heartbeat interval (100ms) + low threshold (3 misses = 300ms detection): fast failure detection but network jitter causes false positives (healthy nodes marked dead). Long interval (5s) + high threshold (3 misses = 15s): fewer false positives but slow to detect actual failures. Adaptive approaches (Phi Accrual) auto-tune based on observed latency distribution.

Production Example

Cassandra: Uses Phi Accrual failure detector — adapts threshold based on historical heartbeat intervals between specific node pairs. If Node A typically responds in 50ms but hasn't in 500ms, suspicion is high. If Node B (cross-region) typically takes 200ms, same 500ms delay has lower suspicion. Reduces false positives in heterogeneous networks.

Key Tip

When designing health checks: "Each node sends heartbeats every 1s. If 3 consecutive heartbeats are missed (3s), the node is suspected dead and traffic is redirected. We use a Phi Accrual detector to adapt to network conditions rather than a fixed timeout — reduces false positives during GC pauses or network congestion."

4.9.8 Split Brain Problem

Critical

What it is: A network partition divides the cluster into two groups, each believing the other is dead. Both halves may elect their own leader, leading to conflicting writes and data divergence.

// Split Brain scenario:
// Cluster: [A, B, C, D, E] — A is leader
// Network partition: [A, B] | [C, D, E]
//
// Partition 1: A thinks it's still leader (has B)
// Partition 2: C, D, E can't reach A → elect C as new leader
// Result: TWO leaders accepting writes → DATA DIVERGENCE!

// Prevention strategies:
// 1. Quorum requirement: Leader must have majority (N/2 + 1) to accept writes
//    - [A, B] = 2 nodes, need 3 for majority → A steps down (CORRECT)
//    - [C, D, E] = 3 nodes, has majority → C becomes leader (CORRECT)
//
// 2. Fencing tokens: New leader gets monotonically increasing token
//    - Old leader's writes rejected by storage (token too old)
//
// 3. STONITH (Shoot The Other Node In The Head):
//    - When partition detected, power off the other node physically
//    - Used in traditional HA (Pacemaker/Corosync)

Trade-off: Quorum vs Availability During Partition

Requiring quorum (majority) prevents split brain but makes the minority partition completely unavailable. Without quorum, both partitions serve traffic but risk data divergence. Trade-off: safety (data correctness) vs liveness (continued operation). CP systems choose safety; AP systems choose liveness and reconcile later.

Production Example

Elasticsearch: Historically suffered split brain when network partitions caused two master nodes. Fixed by requiring minimum_master_nodes = (N/2)+1 (quorum). In ES 7+, this is automatic via the voting configuration. A 3-node cluster tolerates 1 node failure; 2 failures = cluster goes read-only (no quorum for writes).

Key Tip

When asked "how do you prevent split brain?": "Quorum requirement — a leader must maintain majority acknowledgment to accept writes. If a network partition splits our 5-node cluster into [2] and [3], only the [3] partition elects a new leader. The [2] partition stops accepting writes, serving stale reads until reconnection."

4.9.9 Hinted Handoff

Critical

What it does: When a node is temporarily unavailable, another node stores the writes intended for it as "hints." When the original node comes back, hints are replayed to bring it up to date.

// Hinted Handoff flow (Cassandra/DynamoDB-style):
// 1. Write arrives for Node C (which is down)
// 2. Coordinator stores write locally as a "hint":
//    { target: "Node C", data: {key: "user:123", value: ...}, timestamp: T }
// 3. Coordinator still acknowledges write to client (if quorum met by other replicas)
// 4. Background: periodically check if Node C is back
// 5. Node C comes back → replay all stored hints → Node C is caught up
// 6. Delete hints after successful replay

// Trade-offs:
// + Improves availability during brief outages
// + Node recovers faster (doesn't need full anti-entropy repair)
// - Hints consume storage on the hinting node
// - Only works for SHORT outages (hints expire after timeout, e.g., 3 hours)
// - NOT a replacement for full repair (read repair + anti-entropy still needed)

Trade-off: Hint Duration vs Storage Cost

Long hint retention (24h+): node recovers fully from short outages but hinting nodes accumulate significant storage. Short retention (1h): less storage overhead but longer outages require full anti-entropy repair (slower recovery). Cassandra defaults to 3 hours — balances recovery speed vs storage cost for typical maintenance windows.

Production Example

Cassandra: During rolling upgrades, each node goes down for 2-3 minutes. Hinted handoff stores writes destined for the upgrading node on its peers. When it comes back, hints replay in seconds — the node catches up without triggering expensive full repair. Enables zero-downtime upgrades across 100+ node clusters.

Key Tip

Mention hinted handoff for eventually consistent systems: "If a replica is temporarily down, coordinator stores the write as a hint. On recovery, hints replay to bring the replica current. This handles brief outages (maintenance, restarts) without requiring a full anti-entropy repair — which would scan the entire dataset."

4.9.10 Read Repair

Critical

What it does: During a read, if replicas return different versions of the data, the coordinator identifies the most recent version and pushes it to the stale replicas. Reads "repair" inconsistencies opportunistically.

// Read Repair flow:
// 1. Client reads key "user:123" with consistency level ALL (read from all 3 replicas)
// 2. Replica A: version 5 (latest)
//    Replica B: version 5 (latest)
//    Replica C: version 3 (STALE - was down during writes 4 and 5)
// 3. Coordinator returns version 5 to client
// 4. Coordinator sends version 5 to Replica C in background → C is now up to date

// Types of repair:
// 1. Read Repair (foreground): triggered by client reads, only repairs read keys
// 2. Anti-Entropy Repair (background): Merkle tree comparison, repairs ALL keys
//    - Cassandra: nodetool repair (scheduled, e.g., weekly)
//    - Compares hash trees between replicas to find divergent ranges

// When read repair happens depends on read consistency:
// - ONE: coordinator reads from 1 replica, may do digest read from others in background
// - QUORUM: reads from majority, repairs minority if they disagree
// - ALL: reads from all replicas, guaranteed to detect and repair inconsistency

Key Tip

These patterns form a complete failure recovery story: Heartbeats detect failure → Leader election picks a new coordinator → Quorum prevents split brain → Hinted handoff handles short outages → Read repair fixes staleness over time → Anti-entropy repair catches everything else. Mention this full chain when discussing "how does your system handle node failures?" in interviews.
