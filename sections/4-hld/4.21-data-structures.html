4.21System Design Data Structures

4.21.1 Bloom Filter

Advanced
// Probabilistic set membership: "definitely NOT in set" or "probably in set"
// False positives possible, false negatives impossible
// Space: ~10 bits per element for 1% FP rate (vs 64 bits for HashSet)
// Use cases: avoid expensive DB lookups, spam filtering, cache miss reduction

// Example: Check if username exists before querying DB
BloomFilter<String> usernames = BloomFilter.create(Funnels.stringFunnel(), 10_000_000, 0.01);
if (!usernames.mightContain(username)) {
    return "Available"; // Definitely not taken - skip DB query
}
// Might be taken - check DB to confirm (1% false positive)

Production Example

Google Chrome: Safe Browsing uses a Bloom filter to check if a URL is malicious. The filter (stored locally) contains millions of known-bad URLs in just a few MB. On every navigation, Chrome checks the Bloom filter — if positive, it makes a server call to confirm. False positives (~1%) cause a brief server check; false negatives are impossible.

Trade-off

False positives (says 'maybe present' when absent) but NEVER false negatives (says 'definitely absent' is always correct). Larger bit array = fewer false positives but more memory. At 1% false positive rate: ~10 bits per element. Cannot delete elements (use Counting Bloom Filter for deletion support, 4x memory). Trade-off: memory efficiency vs false positive rate.

Interview Tip

Mention Bloom Filters for any 'is this element in a large set?' question: 'Before querying the database for a URL (expensive disk I/O), check a Bloom Filter in memory. If absent → definitely not in DB (skip query). If present → might be in DB (query to confirm). Eliminates 99% of unnecessary disk reads for a cache/shortener/crawler dedup service.'

4.21.2 Count-Min Sketch & HyperLogLog

Advanced
// Count-Min Sketch: estimate frequency of items in a stream
// Use: top-K heavy hitters, rate limiting per IP, trending hashtags
// Space: O(1) fixed size regardless of stream length. May overcount, never undercount.

// HyperLogLog: estimate cardinality (unique count) with ~0.8% error
// Uses only 12KB memory to count billions of unique items!
// Use: unique visitors, distinct IP counts, unique search queries
// Redis: PFADD users "user123" / PFCOUNT users -> approximate unique count

Production Example

Redis: HyperLogLog counts unique visitors with 0.81% error using only 12KB of memory regardless of cardinality. Reddit uses it to count unique post viewers — tracking "4.2M people viewed this post" without storing 4.2M user IDs. Standard error of 0.81% means the displayed count may be off by ~34K, which is acceptable for display purposes.

Trade-off

Both trade accuracy for massive memory savings. HyperLogLog: counts unique elements (cardinality) using ~12KB regardless of dataset size, ±0.81% error. Count-Min Sketch: estimates frequency (how often each element appears) with bounded overcount. Neither supports exact answers or element deletion. Perfect when approximate answers are acceptable (analytics, rate limiting).

Interview Tip

Use HyperLogLog for 'count unique X': 'For unique visitors per page, exact counting would need O(n) memory (store every user_id). Redis HyperLogLog gives ±0.81% accuracy using only 12KB per page — even for billions of visitors. PFADD for each visit, PFCOUNT for the unique count. Perfect for analytics dashboards where approximate is acceptable.'

4.21.3 Geospatial Indexing

Advanced
// Geohash: encode lat/lng into string. Same prefix = nearby
// "9q8yyk" (San Francisco) - longer string = more precise
// Redis: GEOADD locations 13.361389 38.115556 "restaurant1"
//        GEORADIUS locations 15 37 200 km -> find all within 200km

// Geohash edge problem: adjacent points can have totally different prefixes
// Solution: also query the 8 neighboring cells, then post-filter by distance

// Quadtree: recursively divide 2D space into 4 quadrants
// Adaptive: dense areas (cities) get a deeper tree; node splits when a cell's
// point count exceeds a threshold (e.g. 100). Sparse areas stay shallow (saves memory)
// Use: spatial databases, collision detection, Uber driver locations

// R-tree: balanced tree of bounding rectangles - best for "all X within a box"
// PostGIS uses R-trees for its spatial indexes

// H3 (Uber): hexagonal grid system, 16 resolution levels (0=continent, 15=1m^2)
// Uniform neighbors (each hex has exactly 6) -> no geohash edge distortion
// Efficient ops: k-ring (all hexes within k steps), polyfill (hexes covering a polygon)

Trade-off

Geohash (1D string, easy to index, prefix matching for proximity) vs R-Tree (2D spatial index, exact range/box queries) vs S2/H3 Cells (hierarchical, variable resolution). Geohash: simple but edge cases at cell boundaries (nearby points in different prefixes). S2: no edge issues (sphere geometry) but more complex. PostGIS (R-Tree) for SQL databases; Redis GEO (geohash) for in-memory lookups. Resolution trade-off: smaller cells = more precise proximity but more cells to scan per radius query; larger cells query faster but produce false "nearby" positives.

Production Example

Uber: Uses Google S2 cells (level 12, ~3km² each) for driver matching — queries the rider's S2 cell + adjacent cells for available drivers; higher levels drive city-wide surge pricing, lower levels precise matching. Google Maps uses the same S2 library for POI indexing, route segments, and geofencing — querying level 4 for country-level and level 12 for local search.

Interview Tip

For 'find nearby X' problems: 'Store locations with geohash prefix. To find restaurants within 5km: compute geohash of user's location at appropriate precision (6 chars ≈ 1.2km), query all entries with same prefix + 8 neighboring prefixes. Redis GEORADIUS does this natively. For more precision, post-filter results by exact Haversine distance.'

4.21.4 MVCC (Multi-Version Concurrency Control)

Advanced
// MVCC: readers don't block writers, writers don't block readers.
// Each transaction sees a snapshot of the data as of its start time;
// old row versions are kept and later reclaimed (PostgreSQL VACUUM).
// Use: high-concurrency OLTP databases (PostgreSQL, MySQL InnoDB, Oracle).
→ Full MVCC internals (xmin/xmax, isolation levels, bloat & VACUUM) are on the Concurrency & Locking page (4.22.3 MVCC Deep Dive). Fencing tokens for safe distributed locks are covered there too (4.22.2 Distributed Lock with Fencing).

4.21.5 Merkle Trees

Internals
// Merkle Tree: binary tree of hashes for efficient data integrity verification
// Leaf = hash of data block. Parent = hash(left_child + right_child). Root = tree hash.
// Use cases: detect data differences between replicas, blockchain, git, Cassandra anti-entropy

// To find which blocks differ between two replicas:
// 1. Compare root hashes (O(1)) - if same, all data identical
// 2. If different, compare children - recurse down tree
// 3. Only transfer the differing leaf blocks
// Efficiency: O(log n) comparisons to find differences vs O(n) full scan

// Cassandra uses Merkle trees for repair:
// Each node builds Merkle tree of its data range
// Compares trees with replicas -> only syncs differing partitions

Trade-off

Efficient difference detection (O(log n) to find divergent data ranges between replicas) but expensive to maintain (every data change updates the tree from leaf to root). Works best when: data changes infrequently relative to sync checks, and datasets are large enough that full comparison is impractical.

Production Example

Cassandra (Anti-Entropy Repair): Builds Merkle trees over each token range per table. Two replicas compare root hashes — if they match, data is identical (no transfer needed). If different, they traverse down to find the specific divergent ranges and only sync those rows. Reduces repair data transfer from 'entire dataset' to 'only differences.'

Interview Tip

Mention Merkle Trees for data sync/verification: 'To detect data inconsistencies between replicas, we build Merkle trees over partition ranges. Comparing root hashes tells us instantly if replicas are in sync. If not, we traverse the tree to identify exactly which data ranges differ — only those ranges need repair. Used by Cassandra, DynamoDB, and git (for commits).'

4.21.6 Skip List

Internals
// Skip List: probabilistic sorted data structure, O(log n) search/insert/delete
// Like a linked list with express lanes (multiple levels of forward pointers)
// Used in: Redis sorted sets (ZSET), LevelDB/RocksDB memtable, ConcurrentSkipListMap

// Why skip list over balanced BST?
// Simpler implementation, better cache locality for range scans,
// lock-free concurrent versions easier to implement (ConcurrentSkipListMap)
// Trade-off: ~50% more memory than BST, probabilistic O(log n) not guaranteed

Trade-off

Skip List vs Balanced BST (Red-Black Tree): Skip List is simpler to implement, naturally supports concurrent access (lock-free with CAS), and has the same O(log n) performance. BST gives deterministic worst-case; Skip List is probabilistic (but practically equivalent). Redis uses Skip Lists for sorted sets because they're simpler and support range queries efficiently.

Production Example

Redis Sorted Sets (ZSET): Implemented as Skip List + hash table. The Skip List enables O(log n) insertion and range queries (ZRANGEBYSCORE). Used for: leaderboards (rank players by score), rate limiting (sliding window of timestamps), and priority queues. Redis chose Skip List over BST for simplicity of implementation and lock-free concurrent access.

Interview Tip

Mention Skip Lists for ordered data with fast range queries: 'For the leaderboard, I'd use Redis Sorted Set (backed by a Skip List). ZADD for score updates (O(log n)), ZREVRANGE for top-K players (O(log n + K)). Skip Lists support concurrent writers without complex rebalancing — unlike AVL/Red-Black trees. Perfect for real-time ranking systems.'

4.21.7 Probabilistic Structures - When to Use

Internals
// Decision framework:
// Q: "Do I need exact answers?"
// If NO and dataset is huge -> probabilistic structure saves 10-100x memory

// Bloom Filter: "Is X in this set?" (maybe yes / definitely no)
//   - Use: dedup URLs in web crawler, check if key exists before DB lookup
//   - Redis: BF.ADD / BF.EXISTS (RedisBloom module)

// HyperLogLog: "How many unique items?" (~0.8% error, 12KB)
//   - Use: unique visitors per page, distinct queries per day
//   - Redis: PFADD page:home user123 / PFCOUNT page:home

// Count-Min Sketch: "How often does X appear?" (over-estimates possible)
//   - Use: top-K trending items, frequency cap in ads, heavy hitter detection
//   - Redis: CMS.INCRBY / CMS.QUERY (RedisBloom module)

// TopK: "What are the K most frequent items?" (approximate)
//   - Use: trending hashtags, most searched queries
//   - Redis: TOPK.ADD / TOPK.LIST
StructureSpaceAccuracyUse Case
HashSetO(n)ExactSmall sets, precision needed
Bloom FilterO(1) ~10 bits/elemFP possibleHuge sets, can tolerate FP
HashMap (counts)O(n)ExactSmall cardinality
HyperLogLog12KB fixed~0.8% errorBillions of unique items
Count-Min SketchO(1) fixedOver-estimatesFrequency in streams

Trade-off

All probabilistic data structures trade exactness for massive space savings. Bloom Filter: membership testing (12 bits/element vs full storage). HyperLogLog: cardinality (12KB vs storing all elements). Count-Min Sketch: frequency estimation (fixed size vs exact counters). T-Digest: percentiles (few KB vs storing all values). Use when: dataset is huge AND approximate answers are acceptable.

Production Example

Google Chrome: Uses Bloom Filter to check URLs against a malware database. The full database is on Google's servers, but a Bloom Filter fits in the browser. If URL passes Bloom Filter ('definitely not malicious'), no network call needed. If Bloom Filter says 'maybe malicious' (1% false positive rate), Chrome queries the full database server-side for exact check.

Interview Tip

Know when to suggest probabilistic structures: 'Exact unique user count across 1B events? HyperLogLog (12KB, 0.81% error). Check if URL was seen in a crawler with 10B URLs? Bloom Filter (1.2GB vs 100GB for hash set). Estimate top-K trending hashtags in a stream? Count-Min Sketch with min-heap. The interviewer wants to see you optimize for scale — exact isn't always necessary.'
