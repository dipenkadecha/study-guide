4.4Load Balancing & Traffic Management

4.4.1 L4 vs L7 Load Balancing

Infrastructure
LayerOperates OnSpeedFeaturesExamples
L4 (Transport)TCP/UDP packetsVery fast (hardware)IP/port routing onlyAWS NLB, HAProxy (TCP mode)
L7 (Application)HTTP requestsModerateURL routing, headers, SSL terminationAWS ALB, Nginx, Envoy

Production Example

Cloudflare: L4 (TCP) load balancing for raw throughput (gaming, IoT). L7 (HTTP) for smart routing — inspects headers to route /api/* to backend clusters and /static/* to CDN. Processes 45M+ HTTP requests/second at L7.

Trade-off: L4 vs L7 Load Balancing

L4 (TCP): faster (no packet inspection), handles any protocol, but can't route based on content. L7 (HTTP): smarter routing (URL path, headers, cookies), can terminate TLS, enable caching, but adds latency from packet inspection. Use L4 for raw throughput (gaming, streaming); L7 for web apps needing content-based routing.

Key Tip

In system design, always specify L7: "We use an L7 load balancer (ALB/Nginx) that routes /api/* to backend service pods and /static/* to a CDN origin. Health checks at L7 verify HTTP 200 responses, not just TCP connectivity — catches application-level failures."

4.4.3 CDN: Push vs Pull

Infrastructure
// PUSH CDN: You upload content to CDN. Good for static content you control.
// Pro: Content always available immediately. Con: Must manage uploads.
// Example: Upload video to CloudFront after encoding.

// PULL CDN: CDN fetches from origin on first request, caches.
// Pro: No upload management. Con: First request slow (cache miss).
// Example: Web assets, API responses with Cache-Control headers.

// Cache Invalidation patterns:
// 1. TTL-based: Cache-Control: max-age=3600
// 2. Version in URL: /static/bundle.a3f2b1.js (immutable + new URL on change)
// 3. Purge API: POST /purge (CloudFront invalidation)
// Best practice: immutable + versioned URLs for assets, short TTL for API responses

Trade-off: CDN Cache Invalidation

  • Short TTL (seconds): Fresh content, more origin load, higher latency for users
  • Long TTL (days): Better performance, stale content risk, need purge mechanism
  • Best: Immutable URLs (hash in filename) with infinite cache + new URL on deploy. Zero staleness, maximum caching.

Production Example

Netflix Open Connect: Custom CDN with servers physically inside ISP networks. Caches popular content locally — 95% of traffic served from ISP-local boxes. Reduces Netflix bandwidth from crossing the internet backbone, saving ISPs money and improving user experience (lower latency, fewer buffering events).

Key Tip

Mention CDN early in any design serving global users: "Static assets (JS, CSS, images) served via CDN with long cache TTL. API responses for popular queries cached at CDN edge with short TTL (30s) — reduces origin load by 80% for read-heavy workloads like news feeds."
