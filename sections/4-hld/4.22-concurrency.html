4.22Concurrency & Locking Patterns

4.22.1 Optimistic vs Pessimistic Locking

Critical
// OPTIMISTIC: Assume no conflict, detect at commit time
// Implementation: version column. UPDATE ... WHERE version = expected_version
@Entity
public class Product {
    @Version
    private Long version; // JPA increments automatically
    private int stockCount;
}
// If two transactions read version=5, first to commit sets version=6
// Second gets OptimisticLockException -> retry with fresh data
// Best when: conflicts are rare (< 5% of transactions)

// PESSIMISTIC: Lock row immediately, block others
// SELECT * FROM seats WHERE id = ? FOR UPDATE
// Other transactions wait until lock released
// Best when: conflicts are frequent, must not fail (booking systems)

Production Example

GitHub: Pull request merges use optimistic locking. When you click "Merge," GitHub checks if the base branch has changed since you last loaded the page (version check). If it has, the merge is rejected and you must re-fetch — prevents silent overwrites in concurrent merge scenarios without holding locks during code review.

Trade-off

Optimistic (check version at commit, abort on conflict): high throughput when conflicts are rare (read-heavy), but wasted work on abort. Pessimistic (lock before read): guaranteed success but blocks concurrent access (lower throughput). Rule of thumb: optimistic for web apps (low contention on any single row), pessimistic for inventory/seat booking (high contention on popular items).

Key Tip

Match locking strategy to contention level: 'For user profile updates (low contention — only the user themselves edits): optimistic locking with version column. For concert seat booking (high contention — thousands competing for same seat): pessimistic lock (SELECT FOR UPDATE). Optimistic would cause too many retries under high contention.'

4.22.2 Distributed Lock with Fencing

Critical
// Problem: Lock holder pauses (GC), lock expires, new holder gets lock
// Both think they have the lock -> data corruption

// Solution: Fencing tokens - monotonically increasing counter
// Lock acquisition returns token (e.g., 33). Client passes to storage.
// Storage rejects writes with token less than highest seen.

// Client A gets lock (token=33), pauses (GC)
// Lock expires, Client B gets lock (token=34), writes with token=34
// Client A wakes up, tries to write with token=33 -> REJECTED (34 > 33)

Production Example

Shopify: Uses Redis distributed locks for inventory reservation during flash sales. When 100K users click "Buy" simultaneously, each checkout acquires a lock on the specific SKU, decrements inventory, and releases. Lock TTL of 5s prevents deadlocks if a checkout process crashes mid-operation.

Trade-off: Locking Strategies

StrategyThroughputConflict HandlingBest For
Optimistic (version)High (no blocking)Retry on conflictRead-heavy, rare conflicts
Pessimistic (FOR UPDATE)Lower (blocking)Wait for lockWrite-heavy, frequent conflicts
Distributed (Redis)MediumFail-fast or waitCross-service coordination
Lock-free (CAS)HighestSpin retryHot paths, counters, queues

Key Tip

Always mention the fencing token: 'A distributed lock alone isn't safe — if the holder pauses (GC, network delay) and the lock expires, two processes can act as leader simultaneously. The fencing token (monotonic counter from the lock service) ensures storage rejects stale writes. Process with token=34 can't overwrite data written by process with token=35.'

4.22.3 MVCC Deep Dive (PostgreSQL)

Deep Dive
// MVCC: each row has xmin (created by txn) and xmax (deleted/updated by txn)
// Readers see snapshot: rows where xmin < my_txn AND (xmax > my_txn OR xmax=0)
// Writers create new row version (old version invisible to new txns after vacuum)

// Isolation Levels (PostgreSQL):
// READ COMMITTED (default): each statement sees committed data at statement start
// REPEATABLE READ: entire transaction sees snapshot from transaction start
// SERIALIZABLE: transactions behave as if executed one-at-a-time

// Production gotcha: long-running transactions prevent vacuum from cleaning old rows
// -> Table bloat -> Slower queries. Monitor: pg_stat_user_tables (n_dead_tup)

Trade-off

MVCC eliminates read-write contention (readers see snapshots, never block) but: old versions accumulate as dead tuples (bloat), long-running transactions prevent cleanup (hold snapshots open), and VACUUM CPU/IO cost. Alternatives: lock-based (simple, blocks readers) or optimistic without MVCC (recheck at commit). PostgreSQL chose MVCC — optimized for read-heavy OLTP workloads.

Production Example

PostgreSQL: VACUUM ANALYZE runs automatically (autovacuum) to reclaim dead tuples. But: a single long-running query (analytics report on OLTP DB) can pin the snapshot horizon — preventing VACUUM from cleaning up tuples that newer transactions created and deleted. Solution: statement_timeout for long queries, separate OLAP replica for analytics.

Key Tip

Explain MVCC's benefit concretely: 'In PostgreSQL, a SELECT never blocks an UPDATE and vice versa. The reader sees the row version that existed at query start, while the writer creates a new version. This enables high concurrency for read-heavy apps without explicit locking. The cost: tuples accumulate and VACUUM is needed — monitor bloat in production.'

4.22.4 Lock-free Patterns (CAS)

Deep Dive
// Compare-And-Swap: atomic read-modify-write without locks
// AtomicInteger.compareAndSet(expected, new) -> true if current == expected

// Lock-free stack (Treiber Stack):
public class LockFreeStack<E> {
    private final AtomicReference<Node<E>> top = new AtomicReference<>();

    public void push(E item) {
        Node<E> newNode = new Node<>(item);
        Node<E> current;
        do {
            current = top.get();
            newNode.next = current;
        } while (!top.compareAndSet(current, newNode)); // Retry until success
    }
}
// No locks! Multiple threads can push simultaneously.
// Trade-off: under high contention, CAS retries waste CPU (spinning)
// Solution: exponential backoff or LongAdder (striped counters)

Trade-off

Lock-free (CAS loops): no blocking, no deadlocks, better throughput under low-to-medium contention. But: high contention → excessive retries (spinning wastes CPU), complex to implement correctly (ABA problem), and harder to reason about. Lock-based: simpler to understand, fair (FIFO ordering), but risk of priority inversion and deadlocks. Use lock-free for hot counters/queues; lock-based for complex critical sections.

Production Example

Java ConcurrentHashMap: Uses CAS for individual bucket updates — multiple threads can write to different buckets concurrently without locks. Under the hood: compare-and-swap on the bucket's head node. If CAS fails (another thread modified it), retry with the new value. Achieves near-linear scalability on multi-core CPUs for non-conflicting keys.

Key Tip

Know when to suggest CAS: 'For the distributed counter (likes, views), I'd use AtomicLong (CAS-based) rather than synchronized — no thread blocking, scales better on multi-core. For complex operations (transfer money between two accounts), I'd use locks — CAS on multiple variables simultaneously isn't atomic without hardware support (DCAS doesn't exist on commodity hardware).'

4.22.5 Deadlock Prevention in Production

Deep Dive
// Common deadlock: Txn A locks row 1 then row 2. Txn B locks row 2 then row 1.
// Prevention strategies:
// 1. Lock ordering: always acquire locks in consistent order (sorted by key)
// 2. Lock timeout: acquire with timeout, release all and retry if timeout
// 3. Deadlock detection: DB detects cycles, kills one transaction (PostgreSQL)

// Spring/JPA: set lock timeout
@QueryHints({@QueryHint(name = "javax.persistence.lock.timeout", value = "3000")})
@Lock(LockModeType.PESSIMISTIC_WRITE)
Optional<Seat> findByIdForUpdate(String id);

// Production monitoring: track deadlock frequency, slow lock waits
// PostgreSQL: pg_stat_activity (wait_event_type = 'Lock')

Trade-off: When to Use Each Concurrency Strategy

  • synchronized/ReentrantLock: Simple, when contention is moderate. JVM optimizes (biased locking). Default choice.
  • Optimistic locking (@Version): Read-heavy, conflicts < 5%. Retry on failure. No blocking. JPA/Hibernate.
  • Pessimistic locking (FOR UPDATE): Write-heavy, conflicts > 20%. Critical sections (booking). Blocking.
  • Lock-free (CAS/Atomic): Ultra-hot paths (counters, queues). No blocking or deadlock. Complex to implement correctly.
  • Distributed locks (Redis/ZK): Cross-process coordination. Keep TTL short, always use fencing tokens, handle failures gracefully.

Production Example

PostgreSQL: Detects deadlocks automatically by checking the wait-for graph every deadlock_timeout (default 1s). When detected, one transaction is aborted (victim chosen by least work done). Application must catch the deadlock error and retry. Prevention: always acquire locks in consistent order (e.g., by account_id ascending) — guarantees no circular wait.

Key Tip

For any system with multiple locks: 'Deadlock prevention via lock ordering — all services acquire locks in a globally consistent order (e.g., alphabetical by resource ID). For database transactions: keep them short (minimize lock holding time), access tables in the same order across all queries, and use SELECT FOR UPDATE NOWAIT to fail fast rather than block indefinitely.'
