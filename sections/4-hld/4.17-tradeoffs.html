4.17Design Trade-offs

4.17.1 The Four Fundamental Trade-offs

Decision Framework
Trade-offLeft ChoiceRight ChoiceWhen LeftWhen Right
Consistency vs AvailabilityStrong consistency (CP)High availability (AP)Banking, inventory, bookingSocial feeds, DNS, caching
Latency vs AccuracyLow latency (approx)High accuracy (slower)Recommendations, search rankingFinancial calculations, billing
Cost vs PerformanceOptimize cost (eventual, batch)Optimize perf (real-time, premium)Logs, analytics, backupsUser-facing APIs, checkout
Simplicity vs FlexibilitySimple (monolith, fewer services)Flexible (microservices, plugins)Small team, MVP, startupLarge org, diverse requirements

4.17.2 Interview Framework for Trade-off Discussions

Decision Framework
// When interviewer asks "why this choice?", use this framework:
// 1. State the requirements/constraints that drive the decision
// 2. Present 2-3 alternatives you considered
// 3. Explain WHY chosen option fits requirements better
// 4. Acknowledge what you're giving up (shows maturity)
// 5. Mention mitigation for the downside

// Example: "Why Cassandra over PostgreSQL for this chat system?"
// 1. Requirements: 500K writes/sec, multi-region, eventual consistency OK for messages
// 2. Alternatives: PostgreSQL (sharded), DynamoDB, Cassandra
// 3. Cassandra: linear write scalability, multi-DC replication native, tunable consistency
// 4. Giving up: no joins, no complex queries, eventual consistency (messages may appear slightly out of order)
// 5. Mitigation: client-side ordering by timestamp, separate read model for search

Trade-off

No Perfect System: Every design decision has costs. Optimizing for one quality attribute (availability) necessarily compromises another (consistency or cost). The key skill is making these trade-offs EXPLICIT and tying them to business requirements — not choosing one quality blindly.

Production Example

Amazon: Explicitly documents trade-off decisions in design documents (6-page memos). Each design section includes a 'Trade-offs' paragraph: what was gained, what was sacrificed, and why. This creates institutional memory — when requirements change years later, teams can revisit decisions with full context of why they were made.

Key Tip

Never say "X is better than Y." Always say "X is better than Y for THIS use case because..." The best candidates discuss trade-offs unprompted. If you finish a design without mentioning trade-offs, the interviewer will ask - show you already thought about them.
