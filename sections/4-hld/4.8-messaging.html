4.8Messaging & Asynchronous Systems

4.8.1 Kafka Architecture

Critical

Production Example

Uber: Kafka processes 7 trillion messages/day across 6000+ microservices. Every ride generates events (request, match, start, GPS updates, end, payment) flowing through Kafka topics. Consumer groups allow independent processing — fraud, analytics, billing, and mapping services each consume at their own pace.

Trade-off: Kafka vs RabbitMQ vs SQS

Kafka: high throughput, replay capability, ordering within partition, but complex operations. RabbitMQ: flexible routing, priority queues, simpler, but lower throughput and no replay. SQS: fully managed, infinite scale, but no ordering guarantee (FIFO queues limited to 300 msg/s). Choose Kafka for event streaming/replay; RabbitMQ for complex routing; SQS for simple serverless workloads.

Key Tip

Default to Kafka in system design interviews: "Events published to Kafka topics, partitioned by entity_id for ordering. Consumer groups allow independent consumption — analytics, billing, and notification services each read at their own pace. Kafka's retention (7 days) enables replay if a consumer needs reprocessing."

4.8.2 Delivery Semantics

Critical
SemanticGuaranteeImplementationUse Case
At-most-onceMay lose messagesNo ack, fire-and-forgetMetrics, logs (acceptable loss)
At-least-onceNo loss, may duplicateAck after process + retryMost systems (with idempotency)
Exactly-onceNo loss, no duplicatesTransactional producer + idempotent consumerFinancial, critical counters
// Kafka Producer - Exactly-once semantics
Properties props = new Properties();
props.put("enable.idempotence", "true");          // Dedup on broker side
props.put("transactional.id", "order-producer-1"); // Enables transactions
props.put("acks", "all");                          // All replicas must ack
props.put("retries", Integer.MAX_VALUE);           // Retry until success
props.put("max.in.flight.requests.per.connection", "5"); // Idempotent allows 5

// Idempotent consumer pattern (at-least-once + dedup = exactly-once)
@KafkaListener(topics = "payments")
public void processPayment(ConsumerRecord<String, PaymentEvent> record) {
    String idempotencyKey = record.key() + "-" + record.offset();
    if (processedKeys.contains(idempotencyKey)) return; // Already processed

    paymentService.process(record.value());
    processedKeys.add(idempotencyKey);  // Store in Redis with TTL
    // Alternative: use DB unique constraint on event_id
}

Production Example

Stripe: Payment webhooks use at-least-once delivery with idempotency keys. Merchants may receive the same "payment.succeeded" webhook multiple times — they must deduplicate using the event ID. Stripe chose reliability over exactly-once because exactly-once across network boundaries is impossible without client cooperation.

Trade-off: At-Least-Once vs Exactly-Once

At-least-once: simple (retry until ack), may duplicate — consumer must be idempotent. Exactly-once: complex (Kafka transactions + idempotent producer + consumer offset commit in same transaction), higher latency. Most production systems use at-least-once + idempotent consumers — simpler and good enough.

Key Tip

Always mention idempotency with messaging: "We use at-least-once delivery with idempotent consumers. Each message has a unique event_id — consumer checks Redis before processing. If event_id exists, skip (duplicate). This gives us effectively-once semantics without the complexity of distributed transactions."

4.8.3 Dead Letter Queue (DLQ)

Critical
// DLQ: messages that fail after max retries go here for investigation
@Bean
public DefaultErrorHandler errorHandler(KafkaTemplate<String, Object> template) {
    DeadLetterPublishingRecoverer recoverer =
        new DeadLetterPublishingRecoverer(template,
            (record, ex) -> new TopicPartition(record.topic() + ".DLT", -1));

    return new DefaultErrorHandler(recoverer,
        new FixedBackOff(1000L, 3));  // 3 retries, 1s apart, then DLQ
}

Trade-off: Kafka vs RabbitMQ vs SQS

FeatureKafkaRabbitMQAWS SQS
ModelLog (append-only)Queue (AMQP)Queue (managed)
OrderingPer partitionPer queueFIFO queues only
ReplayYes (retention-based)No (consumed = gone)No
ThroughputMillions/sec~50K/sec~3000/sec standard
Best forEvent streaming, log aggregationTask queues, RPCSimple queuing, serverless

Production Example

AWS SQS + Lambda: Messages that fail processing 3 times are automatically moved to a DLQ. Ops team monitors DLQ depth via CloudWatch alarm. Common causes: malformed JSON, missing reference data, transient dependency failures. DLQ messages are manually inspected, fixed, and replayed — no data loss.

Key Tip

Always include DLQ in message-based designs: "Messages that fail after 3 retries with exponential backoff move to a Dead Letter Queue. We set up alerting on DLQ depth — if it grows, it indicates a systemic issue (schema change, downstream outage). DLQ preserves failed messages for manual investigation and replay."

4.8.4 Backpressure Strategies

Critical

What it is: When a producer generates data faster than the consumer can process it, backpressure is the mechanism to prevent the system from being overwhelmed. Without it: queues grow unbounded, memory exhausts, and the system crashes.

StrategyHow It WorksTrade-offExample
DropDiscard excess messages when buffer is fullData loss but system stays aliveMetrics/telemetry (some loss OK)
BufferQueue messages in memory or disk until consumer catches upDelays but no loss — risk OOM if too slowKafka (disk-backed buffer)
Throttle ProducerSlow down or block the producer when consumer is behindProducer latency increases but no lossTCP flow control, Reactive Streams
SampleProcess only every Nth message or latest valueData loss but bounded resource usageUI updates, sensor readings
Scale ConsumerAuto-scale consumers to match producer rateCost increases but no data lossKafka consumer groups, SQS + Lambda
// Reactive Streams backpressure (Java - Project Reactor):
// Consumer requests only what it can handle
Flux.range(1, 1_000_000)
    .onBackpressureBuffer(1000)       // buffer up to 1000, then...
    .onBackpressureDrop(dropped ->    // drop if buffer full
        log.warn("Dropped: {}", dropped))
    .subscribe(new BaseSubscriber<Integer>() {
        @Override
        protected void hookOnSubscribe(Subscription sub) {
            request(100);  // Request only 100 items initially
        }
        @Override
        protected void hookOnNext(Integer value) {
            process(value);
            request(1);    // Request next one after processing
        }
    });

// Kafka consumer backpressure:
// - Consumer controls its own pull rate (poll-based, not push-based)
// - max.poll.records=500 limits batch size per poll
// - If processing is slow, consumer stops polling → no new messages
// - Consumer group rebalance if consumer falls too far behind (max.poll.interval.ms)

// gRPC flow control:
// - Built-in window-based flow control (similar to TCP)
// - Server can signal "I'm busy" via flow control frames
// - Client backs off automatically

Key Tip

When designing any producer-consumer system, always mention backpressure. Ask yourself: "What happens when the producer is 10x faster than the consumer?" The answer reveals your design's failure mode. For Kafka specifically, note that it handles backpressure naturally because consumers pull — they only fetch what they can process. For push-based systems (WebSocket, gRPC streaming), you need explicit flow control.

Trade-off: Drop vs Buffer vs Throttle

Drop: no memory growth but data loss (acceptable for metrics/telemetry). Buffer: no data loss but OOM risk if consumer is permanently slower than producer. Throttle producer: safest but producer latency increases (may cascade upstream). Best: bounded buffer + alert when >80% full + scale consumers automatically.

Production Example

Netflix: Uses reactive streams (RxJava) throughout their microservices. When a downstream service is slow, backpressure propagates upstream automatically — the API gateway slows down acceptance of new requests rather than queuing unboundedly. Prevents cascading OOM failures across the entire call chain.
