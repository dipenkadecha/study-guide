4.15.2 Object Storage (S3) Patterns

Infrastructure
// S3 Design Patterns for System Design:

// 1. Data Lake Architecture
// Raw Zone -> Processed Zone -> Curated Zone
// s3://data-lake/raw/events/2024/01/15/hour=14/*.parquet
// s3://data-lake/processed/user-sessions/*.parquet
// s3://data-lake/curated/analytics-ready/*.parquet

// 2. S3 Performance Optimization
// - Prefix partitioning: distribute across prefixes for >3,500 PUT/s or >5,500 GET/s
// - Multipart upload: files > 100MB (parallel upload parts)
// - S3 Transfer Acceleration: CloudFront edge locations for global uploads
// - S3 Select: query subset of object (push-down filtering, save bandwidth)

// 3. Storage Classes (cost optimization)
// Standard: frequent access ($0.023/GB)
// Intelligent-Tiering: auto-moves between tiers
// Glacier: archive, minutes-to-hours retrieval ($0.004/GB)
// Glacier Deep Archive: rarely accessed ($0.00099/GB)

Production Example

Dropbox: Stores 2+ exabytes in a custom object storage system (Magic Pocket). Originally used S3 but built their own to save $75M+ annually at scale. Objects stored with Reed-Solomon erasure coding (1.5x storage overhead vs 3x for replication) across multiple data centers.

Trade-off

Consistency vs Performance in Object Storage: S3 is now strongly consistent for all operations (since Dec 2020) — no more eventual consistency issues. However: single-object write throughput is limited (5GB single PUT, 5TB multipart). For small files (<1MB), S3 latency (50-100ms first byte) is too slow for hot path — front with CloudFront CDN or ElastiCache.

Key Tip

Use S3 for all blob storage in system design: 'User uploads go to S3 (11 nines durability, lifecycle policies move to Glacier after 90 days). Pre-signed URLs for direct upload/download — our servers never touch the binary data. S3 event notification triggers Lambda for thumbnail generation.' Shows complete data flow without hand-waving.

4.15.4 Distributed File Systems (HDFS)

Infrastructure
// HDFS Architecture:
// NameNode (metadata): file->block mapping, replication factor
// DataNode (data): stores actual blocks (default 128MB each)
// Replication: default 3 copies across racks for fault tolerance

// When to use HDFS vs Object Storage:
// HDFS: Hadoop ecosystem, MapReduce/Spark jobs, need data locality
// Object Storage: Cloud-native, serverless, REST APIs, lower cost
// Modern trend: Object storage (S3) + Spark/Presto = HDFS replacement

Trade-off: Storage System Selection

  • Object Storage: Cheapest at scale, infinite capacity, but higher latency and no in-place updates. Best for write-once-read-many patterns.
  • Block Storage: Fastest I/O, supports databases, but expensive and limited to single-AZ attachment. Best for transactional workloads.
  • File Storage: Shared access across instances, familiar POSIX interface, but lower performance than block. Best for shared state between containers.

Production Example

Hadoop/HDFS: Each data block (128MB) replicated 3x across different racks. NameNode tracks all block locations in memory — 1 billion files requires ~300GB RAM for metadata. Facebook's HDFS cluster: 600+ PB across 4,000+ nodes. Trade-off: great for large sequential reads (analytics), terrible for small random reads (use HBase/Cassandra instead).

Key Tip

Mention HDFS for batch analytics: 'Raw event data (clickstream, logs) is stored in HDFS/S3 as Parquet files partitioned by date. Spark jobs process daily partitions for analytics. The large block size (128MB-256MB) optimizes for sequential scan throughput rather than random access — appropriate for our batch processing pattern.'
