4.11Reliability & Resilience

4.11.1 Resilience Patterns Overview

Resilience
// Retry with Exponential Backoff + Jitter (prevent thundering herd)
public <T> T retryWithBackoff(Supplier<T> operation, int maxRetries) {
    int attempt = 0;
    while (true) {
        try {
            return operation.get();
        } catch (TransientException e) {
            if (++attempt >= maxRetries) throw e;
            long baseDelay = (long) Math.pow(2, attempt) * 100; // 200, 400, 800ms
            long jitter = ThreadLocalRandom.current().nextLong(baseDelay / 2);
            Thread.sleep(baseDelay + jitter); // Jitter prevents synchronized retries
        }
    }
}

// Bulkhead: isolate thread pools per dependency
// If PaymentService is slow, it doesn't exhaust threads for OrderService
@Bean("paymentExecutor")
public Executor paymentExecutor() {
    return new ThreadPoolExecutor(5, 10, 60, TimeUnit.SECONDS,
        new LinkedBlockingQueue<>(100)); // Max 10 threads for payment calls
}

@Bean("shippingExecutor")
public Executor shippingExecutor() {
    return new ThreadPoolExecutor(5, 10, 60, TimeUnit.SECONDS,
        new LinkedBlockingQueue<>(100)); // Separate pool for shipping
}

Production Example

Netflix: Circuit breakers (Hystrix/Resilience4j) protect against cascading failures. When the recommendation service latency exceeds 500ms, circuit opens — returns cached/default recommendations instantly. Fallback serves 95th-percentile quality while preventing the slow service from taking down the entire page load for 200M+ users.

Trade-off: Resilience vs Complexity

Each resilience pattern (circuit breaker, bulkhead, retry) adds code, configuration, and failure modes. Over-engineering with all patterns on every call creates complexity that's harder to debug than the failures it prevents. Apply resilience patterns at service boundaries (external calls), not internal method calls.

Interview Tip

Layer resilience patterns together: "External service calls get: timeout (2s) → retry (3x with exponential backoff + jitter) → circuit breaker (opens after 5 failures in 30s) → fallback (cached/default response). This ensures graceful degradation without cascading failures." Show the full chain, not just one pattern.

Resilience
// Principles of Chaos Engineering (Netflix):
// 1. Define steady state (normal metrics baseline)
// 2. Hypothesize: "system continues working when X fails"
// 3. Inject failure in production (controlled blast radius)
// 4. Observe: does it match hypothesis?

// Tools:
// - Chaos Monkey: randomly kills instances
// - Litmus Chaos: K8s-native chaos experiments
// - Toxiproxy: inject network latency/failures

// Example experiment: "What happens when Redis cache is unavailable?"
// Expected: fallback to DB (higher latency, no outage)
// Inject: block Redis port for 5 minutes in one AZ
// Observe: latency increases, no errors, circuit breaker trips after 30s

Production Example

Netflix Chaos Monkey: Randomly terminates production EC2 instances during business hours. Forces engineers to build systems that tolerate failure by default. Extended to Chaos Kong (simulate entire region failure) and Chaos Automation Platform (game days). LinkedIn and Amazon have adopted similar practices — "if you haven't tested failure, you haven't tested."

Trade-off: Timeout Values

  • Too short: False failures, premature circuit breaks, good requests discarded under load
  • Too long: Thread pool exhaustion, cascading failures, users waiting forever
  • Best practice: Set timeout slightly above p99 latency of dependency. If payment p99=800ms, set timeout=1200ms. Monitor and adjust based on SLOs.

Interview Tip

Mention chaos engineering to show production readiness: "We'd run game days quarterly — inject network latency, kill random pods, simulate region failure. This validates that our circuit breakers, fallbacks, and failover actually work under real conditions rather than just in theory." Shows you think beyond happy-path design.

4.11.3 Failover Patterns

Critical

What it does: Automatically redirect traffic to a standby system when the primary fails. The goal is minimal downtime and data loss.

PatternHow It WorksFailover TimeCostData Loss Risk
Active-Passive (Hot Standby)Standby receives replicated data but serves no traffic. On failure, DNS/LB switches to standby.Seconds to minutes2x infrastructure (idle standby)Minimal (replication lag)
Active-ActiveBoth nodes serve traffic simultaneously. On failure, surviving node absorbs all traffic.Near-zero (already serving)2x infrastructure (both active)None (both are current)
Warm StandbyStandby runs but at reduced capacity. Scaled up on failover.Minutes (scaling time)1.5x infrastructureMinimal
Pilot LightMinimal version of environment always running (DB replicas only). Full infra spun up on failure.10-30 minutesLow (minimal running)Depends on replication
// Active-Passive failover with DNS:
// Normal: app.example.com → Primary (us-east-1)
// Failure detected (health check fails 3 times)
// DNS update: app.example.com → Standby (us-west-2)
// Problem: DNS TTL means some clients still hit dead primary for minutes

// Active-Active with load balancer:
// app.example.com → Load Balancer → [Node A, Node B]
// Node A dies → LB removes from pool → all traffic to Node B
// Failover time: ~10 seconds (next health check interval)
// Challenge: both nodes must handle full write load (no split-brain writes)

Trade-off: Active-Active vs Active-Passive

Active-Active: near-zero failover time, both regions serve traffic, but requires conflict resolution for writes (multi-leader replication) and 2x cost. Active-Passive: simpler (single writer), cheaper (standby mostly idle), but failover takes seconds-to-minutes (DNS propagation, connection draining). Choose based on RTO requirements and budget.

Production Example

Netflix: Runs active-active across 3 AWS regions (us-east-1, us-west-2, eu-west-1). Zuul routes users to nearest region. If us-east-1 fails, traffic automatically shifts to remaining regions within 30 seconds via DNS + health checks. Each region can handle 100% of global traffic (over-provisioned for failover).

Interview Tip

For high-availability questions: "Active-active across 2 regions with stateless services and eventually-consistent data replication. If Region A fails, DNS health checks detect it within 30s and shift traffic to Region B. The key challenge is data: we use async replication with conflict resolution (last-writer-wins) for non-critical data, and synchronous replication for payment state."

4.11.4 Disaster Recovery (RPO / RTO)

Critical

Key Metrics:

MetricDefinitionQuestion It AnswersExample
RPO (Recovery Point Objective)Maximum acceptable data loss measured in time"How much data can we afford to lose?"RPO = 1 hour → backups every hour; lose at most 1 hour of data
RTO (Recovery Time Objective)Maximum acceptable downtime before recovery"How fast must we be back online?"RTO = 15 min → system must recover within 15 minutes
// DR strategies mapped to RPO/RTO:
//
// Strategy          | RPO          | RTO           | Cost
// ─────────────────────────────────────────────────────────
// Backup & Restore | Hours        | Hours         | $     (cheapest)
// Pilot Light      | Minutes      | 10-30 min     | $$
// Warm Standby     | Seconds      | Minutes       | $$$
// Active-Active    | Zero (sync)  | Near-zero     | $$$$ (most expensive)
//
// Choosing: RPO/RTO depends on business impact of downtime
// - Payment system: RPO=0, RTO=30s (active-active, sync replication)
// - Blog platform: RPO=24h, RTO=4h (daily backups, manual restore)
// - E-commerce: RPO=5min, RTO=15min (warm standby, async replication)

Interview Tip

When discussing high availability in system design, always quantify with RPO/RTO. Say: "For this payment system, we need RPO near zero (no lost transactions) and RTO under 30 seconds. This means active-active with synchronous replication across regions." This shows you think in business requirements, not just technical patterns. Then map: stricter RPO/RTO = more expensive = discuss trade-offs with the interviewer.

Trade-off: Cost vs Recovery Speed

Stricter RPO/RTO = exponentially more expensive. RPO=0 requires synchronous cross-region replication (doubles write latency + network costs). RTO=0 requires active-active (2x infrastructure). Most systems accept RPO=5min (async replication) and RTO=15min (warm standby) as a cost-effective middle ground.

Production Example

AWS Multi-AZ RDS: Synchronous replication across Availability Zones (RPO=0 within region). Automatic failover in 60-120 seconds (RTO≈2min). Cross-region read replicas provide RPO=seconds for regional disaster (async replication). Full region failover (Route 53 health check → promote cross-region replica) achieves RTO≈5-10min.
