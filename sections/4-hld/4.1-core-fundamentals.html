4.1Core Fundamentals

4.1.1 Key Metrics

Foundation
PropertyDefinitionMeasurement
ScalabilityHandle growing load by adding resourcesRequests/sec, concurrent users
AvailabilitySystem uptime percentage99.9% = 8.76h downtime/year
ReliabilityCorrect results even under failuresError rate, data loss incidents
DurabilityData survives failuresRPO (Recovery Point Objective)
LatencyTime to first byte / responsep50, p95, p99 percentiles
ThroughputOperations completed per unit timeQPS, TPS, MB/s

Key Tip

Open every system design answer by clarifying which metrics matter most: "Before diving in, let me confirm — are we optimizing for availability (99.99%) or consistency? What's the expected QPS and acceptable p99 latency?" This frames the entire design and shows structured thinking.

4.1.2 CAP Theorem

Foundation

In a distributed system experiencing a network partition (P), you must choose between Consistency (C) and Availability (A). You cannot have both.

ChoiceBehavior During PartitionSystems
CP (Consistency)Reject writes to maintain consistencyZooKeeper, HBase, MongoDB (default)
AP (Availability)Accept writes, resolve conflicts laterCassandra, DynamoDB, CouchDB

Production Example

DynamoDB (AP): During the 2021 us-east-1 event, DynamoDB continued accepting writes on both sides of the partition, resolving conflicts via last-writer-wins. Google Spanner (CP): Uses TrueTime (atomic clocks + GPS) to maintain global strong consistency, accepting ~7ms higher write latency as the cost.

Trade-off: CP vs AP

CP systems (HBase, ZooKeeper) reject writes during partitions — safe but unavailable. AP systems (Cassandra, DynamoDB) accept writes during partitions — available but may serve stale/conflicting data. Most real systems are "tunable" — Cassandra lets you choose per-query (ONE=AP, QUORUM=closer to CP).

Key Tip

Never say "we need all three of CAP." Instead say: "Since network partitions are inevitable in distributed systems, we choose [AP for this feed service because stale reads are acceptable] or [CP for this payment service because incorrect balances are unacceptable]." Always tie the choice to a business requirement.

4.1.3 PACELC Theorem (Extension of CAP)

Foundation

If Partition (P): choose A or C. Else (E): choose Latency (L) or Consistency (C).

// Examples:
// DynamoDB: PA/EL - Available during partition, low latency otherwise (eventual consistency)
// PostgreSQL: PC/EC - Consistent always (single leader), higher latency for consistency
// Cassandra: PA/EL - Available + low latency (tunable consistency per query)
// MongoDB: PC/EC - Consistent (minority partition can't elect a primary, so writes are
//          rejected); reads from primary by default. Matches its CP classification above.

Production Example

Cassandra (PA/EL): LinkedIn uses Cassandra for activity feeds — during partitions it stays available (stale reads OK), and in normal operation it prioritizes low latency over per-write consistency. Tunable consistency per query (ONE for reads, QUORUM for writes).

Trade-off: Latency vs Consistency in Normal Operation

Even without partitions, you must choose: synchronous replication (consistent, slower) or async replication (fast, eventually consistent). This is the "EL vs EC" choice. Most systems choose EL (latency) for reads and EC (consistency) for critical writes.

Key Tip

PACELC is more useful than CAP in interviews because it covers the common case (no partition). Say: "CAP only matters during failures, but PACELC tells us the everyday trade-off. For this system I'd choose PA/EL like DynamoDB — optimize for speed, accept eventual consistency."

4.1.4 Stateless vs Stateful Services

Foundation
// STATELESS: No server-side session. All state in request (JWT, query params)
// Advantages: horizontal scaling trivial, any instance handles any request
// Example: REST API servers behind load balancer

// STATEFUL: Server holds session/connection state
// Examples: WebSocket servers, database connections, in-memory caches
// Challenge: sticky sessions or state externalization needed

// Pattern: Externalize state to make services "stateless"
// Before: session in memory -> sticky sessions, can't scale
// After: session in Redis -> any instance can serve any user
@Bean
public ReactiveSessionRepository sessionRepository(ReactiveRedisConnectionFactory factory) {
    return new ReactiveRedisSessionRepository(factory); // Externalized state
}

Trade-off: Consistency Models

ModelGuaranteeLatencyUse Case
Strong (Linearizable)Read always sees latest writeHighestFinancial, inventory counts
SequentialAll see same order of operationsHighDistributed locks
CausalCausally related ops seen in orderMediumSocial feeds, messaging
EventualAll replicas converge eventuallyLowestDNS, CDN, product catalog

Production Example

Netflix Zuul/Spring Cloud Gateway: All API gateway instances are stateless — JWT tokens carry session state. Any instance can handle any request. Session data externalized to Redis, enabling horizontal scaling to 1000+ gateway instances during peak (Sunday night streaming).

Key Tip

When designing any web service, always state: "The API layer is stateless — session/auth state lives in JWT tokens or external Redis. This lets us scale horizontally behind a load balancer with zero sticky sessions." It's a foundational assumption that unlocks everything else in your design.
