4.19Data Consistency Patterns

4.19.1 Two-Phase Commit (2PC)

Critical
// 2PC: Coordinator asks all participants to prepare, then commit
// Phase 1 (Prepare): Coordinator -> "Can you commit?" -> All vote YES/NO
// Phase 2 (Commit): If all YES -> Coordinator -> "Commit" -> All commit
// Problem: Blocking protocol. If coordinator crashes after prepare, participants stuck.
// Use case: Single database with XA transactions. Avoid across microservices.

Trade-off

2PC guarantees atomicity across services but is blocking — if coordinator crashes after prepare, participants hold locks indefinitely. Also slow (2 round-trips). Alternative: Saga pattern (non-blocking, compensating transactions) for most distributed scenarios. Use 2PC only within a single database (XA transactions) or when atomicity is absolutely required and latency is acceptable.

Production Example

Spanner (Google): Uses 2PC internally for cross-shard transactions, but with TrueTime clock to avoid the blocking problem. For mortals: MySQL XA transactions coordinate across two MySQL instances within the same datacenter. Outside Google, most companies avoid 2PC for cross-service transactions — they use Saga/Outbox instead.

Interview Tip

Know when NOT to use 2PC: "For cross-service transactions, 2PC is fragile (coordinator SPOF, blocking on failure). I'd use the Saga pattern instead — each step has a compensating action. For cross-shard within one database (e.g., transfer money between two MySQL shards), 2PC is acceptable because the coordinator is the application itself with known recovery behavior."

4.19.2 Saga Pattern

Critical
// Saga: sequence of local transactions with compensating actions on failure
// Orchestration: Central coordinator directs each step
// Choreography: Each service publishes events, next service reacts

// Example: Order Saga (Orchestration)
// 1. CreateOrder -> 2. ReserveInventory -> 3. ProcessPayment -> 4. ShipOrder
// If step 3 fails: compensate step 2 (ReleaseInventory) + step 1 (CancelOrder)

@Service
public class OrderSagaOrchestrator {
    public void execute(OrderRequest req) {
        try {
            Order order = orderService.create(req);              // Step 1
            inventoryService.reserve(order.items());              // Step 2
            paymentService.charge(order.total());                 // Step 3
        } catch (PaymentException e) {
            inventoryService.release(order.items());              // Compensate 2
            orderService.cancel(order.id());                      // Compensate 1
            throw new SagaFailedException(e);
        }
    }
}

Production Example

Uber: Ride completion uses orchestrated saga: (1) charge rider → (2) pay driver → (3) update trip record → (4) emit analytics event. If payment fails at step 1, compensating action cancels the ride charge. Cadence/Temporal workflow engine manages saga state and retries — handles 100K+ concurrent sagas.

Trade-off

Choreography vs Orchestration: Choreography (events between services, no central controller) is more decoupled but harder to reason about (implicit workflow, hard to debug). Orchestration (central coordinator directs steps) is easier to understand and monitor but coordinator is a coupling point. Use orchestration for complex flows (>4 steps) and choreography for simple 2-3 step flows.

Interview Tip

Always mention compensation: "Order saga: reserve inventory → charge payment → confirm order. If payment fails, compensation runs: release inventory. Each step is idempotent, each compensation is idempotent. We use an orchestrator (Temporal/Cadence) that tracks saga state and triggers compensations on failure — no manual intervention needed."

4.19.3 Outbox Pattern

Critical
// Problem: DB write + event publish must be atomic (dual write problem)
// If DB saves but Kafka publish fails -> inconsistency

// Solution: Write event to "outbox" table in same DB transaction
// Separate process (CDC/poller) reads outbox and publishes to Kafka
@Transactional
public void createOrder(OrderRequest req) {
    Order order = orderRepo.save(Order.from(req));
    outboxRepo.save(new OutboxEvent("OrderCreated", order.id(), serialize(order)));
    // Both in same transaction - atomic!
}

// Debezium CDC connector reads outbox table changes -> publishes to Kafka
// After successful publish, marks outbox entry as processed

Production Example

Zalando: Uses Transactional Outbox with Debezium CDC. Order service writes order + outbox event in same PostgreSQL transaction. Debezium tails the WAL, picks up outbox entries, publishes to Kafka. Guarantees no lost events even if Kafka is temporarily down — the outbox acts as a durable buffer.

Trade-off

Outbox Pattern solves dual-write (DB + message broker atomicity) but adds complexity: outbox table, CDC/polling job, dedup on consumer side. Alternative: "Listen to yourself" — service publishes event first, then consumes its own event to update DB. Simpler but loses DB transaction guarantee. Outbox is the industry standard for financial/critical operations.

Interview Tip

Explain the dual-write problem first: "If I update the DB and publish to Kafka separately, either can fail — leading to inconsistency. Solution: write the event to an outbox table in the SAME DB transaction as the business data. A CDC connector (Debezium) tails the outbox table and publishes to Kafka. Guarantees: if DB commits, event will be published. Consumer must be idempotent (possible duplicates)."

4.19.4 Change Data Capture (CDC)

Critical
// CDC: Capture row-level changes from DB transaction log (WAL/binlog)
// Tools: Debezium (open-source), AWS DMS, Oracle GoldenGate
// Use cases: sync databases, feed search index, event sourcing, cache invalidation

// Debezium reads PostgreSQL WAL -> publishes change events to Kafka
// Each table change becomes: {before: {...}, after: {...}, op: "u", ts_ms: ...}

Trade-off: 2PC vs Saga vs Outbox

PatternConsistencyPerformanceComplexityUse When
2PCStrong (ACID)Slow (blocking)LowSingle DB, XA transactions
SagaEventualGood (async)High (compensations)Cross-service workflows
Outbox + CDCEventual (reliable)GoodMediumReliable event publishing

Production Example

LinkedIn: Uses Databus (internal CDC) to stream database changes to search indexes, caches, and analytics. When a profile is updated in MySQL, the change propagates to Elasticsearch (search), Espresso cache (profile views), and Hadoop (analytics) — all within seconds. Single source of truth (MySQL), multiple derived views kept in sync via CDC.

Interview Tip

Mention CDC for any "keep systems in sync" requirement: "Rather than dual-writing to both the database and search index, I'd use CDC (Debezium) to capture every committed change from PostgreSQL's WAL and stream it to Kafka. Consumers update Elasticsearch, Redis cache, and analytics warehouse independently. This ensures all derived systems are consistent with the primary database."

4.19.6 CRDTs (Conflict-free Replicated Data Types)

Advanced
// CRDTs: data structures that can be replicated across nodes and merged
// without coordination - mathematically guaranteed to converge
// No consensus needed! Each node updates independently, merges resolve automatically.

// Types:
// G-Counter: grow-only counter (each node has own counter, sum = total)
// PN-Counter: positive-negative counter (two G-Counters)
// G-Set: grow-only set (union for merge)
// OR-Set: observed-remove set (add wins over remove)
// LWW-Register: last-writer-wins by timestamp

// Use cases: collaborative editing (Google Docs), shopping cart (Amazon),
// distributed counters (like counts), offline-first apps

// Example: G-Counter (each node increments its own slot)
// Node A: {A:3, B:0, C:0} -> value = 3
// Node B: {A:0, B:5, C:0} -> value = 5
// Merge:  {A:3, B:5, C:0} -> value = 8 (take max per node)
// No conflicts possible! Commutative, associative, idempotent.

Production Example

Figma: Uses CRDTs for real-time collaborative design. Multiple designers edit the same canvas simultaneously — each change is a CRDT operation that auto-merges without conflicts. No central server needed for conflict resolution. Also used by Apple Notes, Redis (CRDB), and Riak for multi-region replication.

Trade-off

CRDTs guarantee eventual convergence without coordination but: limited data types (counters, sets, registers — not arbitrary business logic), higher storage overhead (metadata for conflict resolution), and semantic conflicts (counter can go negative in some implementations). Use for: collaborative editing, distributed counters, shopping carts. Don't use for: complex business rules requiring invariants.

Interview Tip

Use CRDTs when offline/multi-region writes are needed: "Shopping cart is a OR-Set CRDT — items added in any region merge without conflicts. Likes counter is a G-Counter CRDT — each replica counts independently, sum gives total. For the collaborative whiteboard, we'd use an RGA (Replicated Growable Array) CRDT for the element list — supports insert/delete without central coordination."

4.19.7 Quorum Reads/Writes

Advanced
// Quorum: R + W > N ensures overlap between read and write sets
// N=3 replicas, W=2 (write to 2), R=2 (read from 2)
// At least 1 node has latest write -> client picks highest version

// Tuning:
// W=1, R=3: Fast writes, slow reads (write-heavy: IoT sensors)
// W=3, R=1: Slow writes, fast reads (read-heavy: product catalog)
// W=2, R=2: Balanced (most common)

// Sloppy Quorum: during partition, write to ANY available nodes (not just replicas)
// Hinted Handoff: temporarily store writes for unavailable nodes, forward when back
// Read Repair: on read, if replicas disagree, update stale ones

Trade-off: Strong vs Eventual Consistency

  • Strong (linearizable): Simple programming model, higher latency (cross-node coordination), lower availability during partitions. Use: banking, inventory counts, seat booking.
  • Eventual (CRDTs/quorum): High availability, low latency, complex conflict resolution. Use: social feeds, shopping carts, counters, collaborative docs.
  • Key question in interview: "What happens if two users edit simultaneously?" If answer is "one wins" -> LWW. If "both preserved" -> CRDTs/merge.

Production Example

Cassandra: Default replication_factor=3 with QUORUM consistency = write to 2/3 replicas, read from 2/3 replicas. Guarantees overlap (at least one node has latest value). For cross-DC: LOCAL_QUORUM = quorum within local datacenter only (fast) + async replication to remote DC. Tunable per-query: ONE for analytics reads (fast, possibly stale), QUORUM for user-facing (consistent).

Interview Tip

Explain the math: "With N=3 replicas, W=2 writes, R=2 reads: W+R > N guarantees we read at least one up-to-date replica. Trade-off: higher W = more durable writes but slower. Higher R = more consistent reads but slower. Common configs: (W=2, R=2) for balanced, (W=3, R=1) for read-heavy workloads that need consistency without read overhead."
