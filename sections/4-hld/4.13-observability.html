4.13Observability & Monitoring

4.13.1 Three Pillars of Observability

Operations
// 1. Structured Logging
log.atInfo()
    .addKeyValue("orderId", orderId)
    .addKeyValue("customerId", customerId)
    .addKeyValue("duration_ms", duration)
    .addKeyValue("traceId", MDC.get("traceId"))
    .log("Order processed successfully");
// Output: {"timestamp":"...","level":"INFO","orderId":"123","traceId":"abc",...}

// 2. Metrics (RED Method for services)
// Rate: requests/second
// Errors: error rate (4xx, 5xx)
// Duration: latency percentiles (p50, p95, p99)
@Timed(value = "order.create", percentiles = {0.5, 0.95, 0.99})
public Order createOrder(OrderRequest req) { ... }

// 3. Distributed Tracing (OpenTelemetry)
// Trace -> Spans -> parent/child relationship across services
// Propagation: W3C TraceContext header (traceparent: 00-traceId-spanId-01)
// Tools: Jaeger, Zipkin, AWS X-Ray, Datadog APM

Production Example

Uber: Jaeger (distributed tracing, which Uber created) traces a single ride request across 50+ microservices. Each service adds a span. When a rider reports "ride took 3 minutes to match," engineers search traces by ride_id to find exactly which service added latency — was it matching algorithm, driver notification, or fraud check?

Trade-off

Logs vs Metrics vs Traces — Logs: rich detail but expensive at scale (storage, indexing). Metrics: cheap (numbers, aggregatable) but no context on individual requests. Traces: show full request path but sampling needed at scale (100% tracing is prohibitively expensive). Use all three together: metrics detect, logs diagnose, traces pinpoint.

Key Tip

When asked about observability: 'Every service emits structured logs (JSON, correlation_id), RED metrics (Rate, Errors, Duration), and distributed traces (OpenTelemetry). Grafana dashboard shows service health. Alerts fire on p99 latency > 500ms or error rate > 1%. On-call uses traces to identify which downstream service caused the spike.'

4.13.2 Monitoring Methodologies

Operations
MethodMetricsBest For
REDRate, Errors, DurationRequest-driven services (APIs, microservices)
USEUtilization, Saturation, ErrorsInfrastructure resources (CPU, disk, network)
Golden SignalsLatency, Traffic, Errors, SaturationGoogle SRE approach (comprehensive)

Key Tip

Know USE (Utilization, Saturation, Errors) for infrastructure and RED (Rate, Errors, Duration) for services. In interviews: 'I monitor infrastructure with USE method — is the CPU saturated? Is the disk queue full? For microservices I use RED — what's the request rate, error rate, and latency distribution?'

4.13.3 SLI / SLO / SLA

Operations
TermDefinitionExample
SLI (Indicator)Metric measuring service qualityp99 latency, error rate, availability
SLO (Objective)Target value for SLIp99 latency < 200ms, availability 99.95%
SLA (Agreement)Contract with consequences if SLO breached99.9% uptime or credits issued
Error BudgetAllowed unreliability (100% - SLO)99.9% SLO = 43.8 min/month budget

Key Tip

State your SLO concretely: 'Our SLO is 99.9% of requests complete successfully within 200ms, measured over a 30-day rolling window. Error budget = 0.1% = ~43 minutes of downtime/month. When budget is exhausted, we freeze deployments and focus on reliability.' Shows you can translate reliability into business terms.

4.13.4 Alerting Best Practices

Operations
// Alert on symptoms, not causes
// BAD: Alert on CPU > 80% (cause)
// GOOD: Alert on p99 latency > SLO (symptom users feel)

// Runbook-linked alerts:
// Every alert should have: what it means, impact, investigation steps, remediation
// Severity levels: P1 (revenue impact, page immediately), P2 (degraded, page in hours), P3 (investigate next business day)

// Error Budget Burn Rate alerting:
// Fast burn: consuming 2% of monthly budget in 1 hour -> page
// Slow burn: consuming 5% of monthly budget in 6 hours -> ticket

Production Insight

Set SLOs tighter than SLAs. If SLA promises 99.9%, set internal SLO at 99.95%. The gap is your safety margin. When error budget is exhausted, freeze feature work and focus on reliability.

Trade-off

Sensitive alerts (low threshold) catch real issues faster but generate alert fatigue from false positives. Conservative alerts (high threshold) reduce noise but miss slow degradations. Best practice: multi-window alerting — short window (5min) for severe issues, long window (1h) for gradual degradation. Separate paging alerts (SLO-burning) from informational ones.

Key Tip

Describe your alerting philosophy: 'We alert on symptoms, not causes. Page on: error rate > 1% for 5 minutes (user-facing impact), p99 > 2s (SLO breach), error budget burn rate > 10x (will exhaust budget in 3 days). We DON'T page on: high CPU (a cause, not a symptom) unless it correlates with user impact.'
