4.3Networking & Communication

4.3.1 DNS Resolution Flow

Networking

Production Example

Netflix: Uses Route 53 weighted DNS for gradual region failover. DNS TTL set to 60s — allows shifting traffic from us-east-1 to eu-west-1 within minutes during regional outages without client changes.

Interview Tip

Always start HLD diagrams with DNS. "Client hits DNS which resolves to our load balancer IP. We use weighted DNS for gradual region failover and low TTL (60s) to enable fast failover." Shows you understand the full request path from user to server.

4.3.2 HTTP Protocol Comparison

Networking
VersionKey FeatureConnectionUse Case
HTTP/1.1Keep-alive, pipeliningMultiple TCP (6 per domain)Legacy systems
HTTP/2Multiplexing, header compression, server pushSingle TCP, binary framesMost web traffic today
HTTP/3 (QUIC)UDP-based, 0-RTT, no head-of-line blockingUDP + QUICMobile, unreliable networks

Interview Tip

Mention HTTP/2 as the baseline for modern services (multiplexing eliminates head-of-line blocking at HTTP level). For mobile-first or global apps, mention HTTP/3 (QUIC): "Zero-RTT connection resumption reduces latency for mobile users switching between Wi-Fi and cellular."

4.3.3 API Communication Comparison

Networking
ProtocolFormatStreamingUse Case
RESTJSON/XMLNo (SSE possible)Public APIs, CRUD, browser clients
gRPCProtobuf (binary)BidirectionalInternal service-to-service, low latency
GraphQLJSONSubscriptionsClient-specific data needs, BFF
WebSocketAnyFull-duplexReal-time: chat, gaming, live data

Production Example

Dropbox: Migrated from REST to gRPC for internal service communication — reduced latency by 50% and payload size by 70% due to Protobuf binary encoding. Saved millions in bandwidth costs at their scale.

Trade-off: REST vs gRPC vs GraphQL

  • REST: Universal, cacheable (HTTP caching), browser-native. Overhead: over-fetching, multiple round trips, no typing.
  • gRPC: 10x faster (Protobuf), strong typing, streaming, code-gen. Overhead: browser support limited (needs proxy), not human-readable.
  • GraphQL: Client controls data shape, single endpoint, no over-fetching. Overhead: N+1 problem (use DataLoader), complex caching, no HTTP caching.
  • Decision: Public API = REST. Internal high-throughput = gRPC. BFF with diverse clients = GraphQL.

Interview Tip

Default to REST for external APIs and gRPC for internal microservice communication in your designs. Say: "External clients use REST (universal, cacheable via CDN), while internal services use gRPC for lower latency and strong typing via Protobuf contracts."

4.3.4 REST API Maturity (Richardson Model)

Essential
  • Level 0: Single endpoint, RPC-style (POST /api)
  • Level 1: Resources (/users, /orders)
  • Level 2: HTTP verbs (GET, POST, PUT, DELETE) + status codes
  • Level 3: HATEOAS - hypermedia links in responses

Interview Tip

When designing REST APIs in interviews, aim for Level 2 (resources + proper HTTP verbs + status codes). Mention HATEOAS awareness but note it's rarely implemented in practice. What matters: consistent resource naming, proper status codes (201 Created, 404, 409 Conflict), and idempotent PUT/DELETE.

4.3.5 Idempotency

Essential
// Idempotency Key - safe retries for non-idempotent operations
POST /api/payments
Headers: Idempotency-Key: "uuid-12345-unique-per-request"
// Server: check if key processed before. If yes, return the cached response.
// Implementation: store key + response in Redis with TTL (24h).
// GET/PUT/DELETE are naturally idempotent; POST is not -> needs a key.
→ Pagination is covered in 4.3.8 Cursor-Based Pagination and API versioning in 4.3.9 API Versioning (both below).

Interview Tip

Mention idempotency keys for payment/order APIs — it signals production experience: "We put idempotency keys on POST /payments so a network retry doesn't double-charge; the server stores the key + response in Redis for 24h and replays the cached response on a repeat."

4.3.7 Webhooks

Critical

What it does: Instead of the client polling for updates, the server pushes events to a client-provided callback URL when something happens. "When this order ships, POST the tracking info to this URL."

When to use:

  • Third-party integrations (Stripe payment events, GitHub push events, Slack notifications)
  • Any system where the server knows when something happens and clients need to be notified
  • Replacing expensive polling (checking every 5 seconds vs getting pushed instantly)

Trade-offs:

AdvantageDisadvantage
Real-time notification without pollingClient must expose a publicly accessible URL
Reduces server load (no repeated poll requests)Delivery not guaranteed (client may be down)
Simple HTTP POST — no special protocolRequires retry logic, idempotency keys, signature verification
// Webhook best practices:
// 1. Include a signature header for verification (HMAC-SHA256)
// 2. Implement retry with exponential backoff (1s, 5s, 30s, 5min, 1hr)
// 3. Include an idempotency key so receivers can deduplicate
// 4. Use a dead-letter queue for permanently failed deliveries
// 5. Allow receivers to return 2xx quickly, process async

// Example webhook payload from a payment system:
// POST https://merchant.com/webhooks/payments
// Headers:
//   X-Signature: sha256=abc123...
//   X-Idempotency-Key: evt_12345
//   Content-Type: application/json
// Body:
// {
//   "event": "payment.completed",
//   "data": { "orderId": "ord_789", "amount": 99.99, "currency": "USD" },
//   "timestamp": "2024-01-15T10:30:00Z"
// }

Interview Tip

When designing any event notification system, mention webhook best practices: HMAC signature verification, exponential backoff retries, idempotency keys, and a dead-letter queue for failed deliveries. This shows you've built production integrations, not just consumed APIs.

4.3.8 Cursor-Based Pagination

Critical

What it does: Return results in pages using an opaque cursor (typically an encoded pointer to the last item) rather than offset/limit. The client passes the cursor to get the next page.

ApproachHow It WorksProsCons
Offset-Based?page=5&limit=20 → OFFSET 80 LIMIT 20Simple, can jump to any pageSlow for deep pages (DB scans skipped rows), inconsistent with live inserts
Cursor-Based?after=eyJpZCI6MTAwfQ&limit=20 → WHERE id > 100 LIMIT 20Fast (indexed seek), stable under inserts/deletesCan't jump to arbitrary page, cursor is opaque
Keyset-Based?after_id=100&limit=20 (non-encoded cursor)Same as cursor but transparentExposes internal IDs, multi-column keys are complex
// Cursor pagination implementation:
// 1. Client requests: GET /api/posts?limit=20
// 2. Server returns: { items: [...20 posts...], nextCursor: "eyJpZCI6MTIzfQ==" }
// 3. Client requests: GET /api/posts?limit=20&after=eyJpZCI6MTIzfQ==
// 4. Server decodes cursor → { id: 123 } → WHERE id > 123 ORDER BY id LIMIT 20

// Why cursor is better for infinite scroll / timelines:
// - Offset page 1000: SELECT * FROM posts OFFSET 20000 LIMIT 20 (scans 20000 rows!)
// - Cursor page 1000: SELECT * FROM posts WHERE id > 12345 LIMIT 20 (index seek!)

Interview Tip

When an interviewer asks about pagination, immediately say "cursor-based" and explain why: "Offset pagination degrades at scale — OFFSET 10000 scans and discards 10K rows. Cursor pagination uses an indexed WHERE clause, giving O(1) seek regardless of page depth. Essential for feeds and timelines."

4.3.9 API Versioning

Critical

What it does: Maintain multiple versions of an API simultaneously so existing clients continue working when the API evolves.

StrategyExampleProsCons
URL Path/api/v1/users, /api/v2/usersExplicit, easy to route, cache-friendlyURL pollution, clients must change URLs to upgrade
HeaderAccept: application/vnd.api+json;version=2Clean URLs, follows HTTP semanticsHidden, harder to test in browser, easy to forget
Query Param/api/users?version=2Easy to test, explicitMessy URLs, not RESTful
Content NegotiationAccept: application/vnd.company.v2+jsonMost RESTful approachComplex to implement and document

Interview Tip

In system design interviews, when designing APIs mention: (1) versioning strategy (URL path is most common in practice), (2) pagination approach (cursor for feeds/timelines, offset for admin dashboards with page jumps), and (3) rate limiting. These show you think about API evolution and real-world usage, not just the happy path.

4.3.10 Long Polling

Critical

What it does: Client sends a request, server holds it open until new data is available (or timeout). Once data arrives, server responds. Client immediately sends a new request. Simulates real-time push over standard HTTP.

// Long Polling flow:
// 1. Client: GET /api/messages?since=lastMessageId
// 2. Server: holds connection open (no response yet)
// 3. ... 30 seconds pass, new message arrives ...
// 4. Server: responds with new message data
// 5. Client: immediately sends new GET /api/messages?since=newLastId
// 6. Repeat

// Server-side (Spring Boot):
@GetMapping("/messages")
public DeferredResult<List<Message>> pollMessages(@RequestParam long since) {
    DeferredResult<List<Message>> result = new DeferredResult<>(30000L); // 30s timeout
    
    // Check if messages already available
    List<Message> pending = messageService.getAfter(since);
    if (!pending.isEmpty()) {
        result.setResult(pending);
    } else {
        // Register listener - will complete when new message arrives
        messageService.addListener(since, result::setResult);
    }
    
    result.onTimeout(() -> result.setResult(Collections.emptyList()));
    return result;
}

When to use:

  • Simple real-time needs without WebSocket infrastructure
  • Works through firewalls/proxies that block WebSocket upgrades
  • Low-frequency updates (notifications, status changes)

When NOT to use:

  • High-frequency bidirectional communication (use WebSockets)
  • Streaming large data (use SSE or WebSockets)
  • Many concurrent users — each holds a connection open

Production Example

Slack (early days): Used long polling for real-time messages before migrating to WebSockets. The client held an open HTTP request; when a new message arrived in the channel, the server responded immediately. Scaled to millions of connections but eventually migrated to WebSockets for lower overhead.

Interview Tip

Mention long polling as a fallback when WebSockets aren't feasible (corporate firewalls, proxy servers that strip upgrade headers). "Our primary real-time channel is WebSocket, with long polling as a fallback for restricted network environments."

4.3.11 Server-Sent Events (SSE)

Critical

What it does: Server pushes events to the client over a single long-lived HTTP connection. Unidirectional (server to client only). Client opens a connection, server sends events as they occur. Built on standard HTTP — no protocol upgrade needed.

// SSE vs WebSocket vs Long Polling comparison:
// SSE:         Server → Client only, auto-reconnect, HTTP-native, text only
// WebSocket:   Bidirectional, manual reconnect, protocol upgrade, binary + text
// Long Polling: Simulated push, new connection per message, high overhead

// Server-side SSE (Spring Boot):
@GetMapping(value = "/stream/orders", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<ServerSentEvent<Order>> streamOrders() {
    return orderService.getOrderStream()
        .map(order -> ServerSentEvent.<Order>builder()
            .id(order.getId())
            .event("order-update")
            .data(order)
            .retry(Duration.ofSeconds(5))  // client auto-reconnects after 5s
            .build());
}

// Client-side JavaScript:
// const eventSource = new EventSource('/stream/orders');
// eventSource.addEventListener('order-update', (e) => {
//     const order = JSON.parse(e.data);
//     updateUI(order);
// });
// eventSource.onerror = () => { /* auto-reconnects by default */ };

When to use:

  • Live dashboards, stock tickers, sports scores (server pushes updates)
  • Notification feeds (one-way: server to browser)
  • Progress updates for long-running operations
  • When you need auto-reconnect (SSE has it built-in, WebSocket doesn't)

Trade-off: Choosing the Right Real-Time Pattern

PatternDirectionConnectionReconnectProtocolBest For
Short PollingClient → ServerNew per requestN/AHTTPSimple, infrequent checks
Long PollingServer → Client (simulated)Held open, renewedManualHTTPCompatibility, low-frequency
SSEServer → ClientSingle persistentAutomaticHTTPLive feeds, notifications
WebSocketBidirectionalSingle persistentManualWS (upgrade)Chat, gaming, collaboration
gRPC StreamBidirectionalSingle persistent (HTTP/2)ManualgRPCService-to-service streaming

Interview Tip

When designing notification systems or live feeds, mention SSE first — it's simpler, HTTP-native, auto-reconnects, and works through CDNs. Reserve WebSockets for truly bidirectional needs (chat, multiplayer). Interviewers appreciate when you pick the simpler tool that fits rather than defaulting to WebSockets for everything.

4.3.12 WebSockets & Scaling

Critical
// WebSocket: upgrade an HTTP connection to a full-duplex TCP channel.
// Pro: lowest latency, bidirectional, binary + text frames.
// Con: stateful (connection state per client), not CDN-cacheable, more infra.
// Use: chat, multiplayer gaming, collaborative editing, trading platforms.

// Scaling WebSockets (the part interviewers probe):
// Problem: user is connected to Server A, but a message for them arrives at Server B.
// Solution: a pub/sub backbone (Redis Pub/Sub or Kafka) between the servers.
//   1. Server B receives the message -> publishes to channel "user:{id}"
//   2. Server A (where the user's socket lives) is subscribed -> pushes over the socket
// Also need: sticky sessions (or a connection registry mapping userId -> server),
//   heartbeats/ping-pong to detect dead connections, and backpressure on slow clients.

Trade-off

WebSocket gives the lowest latency and true bidirectionality, but every connection is stateful — horizontal scaling needs sticky routing plus a pub/sub fanout so any server can reach any client. SSE avoids all of that (stateless-ish, auto-reconnect, CDN-friendly) but is one-way. Default to SSE unless the client must also push.

Production Example

Slack: Uses WebSocket for real-time messaging (bidirectional — typing indicators need client→server too), falling back to long polling on restrictive corporate proxies. Presence (online/away) rides the same socket via lightweight heartbeats, so presence adds no extra connections.

Interview Tip

Lead with the scaling answer: "WebSocket connections are stateful, so I'd put a connection registry (userId → server) and a Redis/Kafka pub/sub layer between socket servers — a message arriving anywhere is published and delivered to whichever server holds that user's socket. Sticky sessions at the LB keep a client pinned to its server." See the comparison table in 4.3.11 for choosing between polling / SSE / WebSocket.
