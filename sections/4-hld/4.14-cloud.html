4.14Cloud & Infrastructure

4.14.1 Kubernetes Scaling

DevOps
# HPA: scale pods based on CPU/memory/custom metrics
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: order-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: order-service
  minReplicas: 3
  maxReplicas: 50
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Pods          # Custom metric (Kafka consumer lag)
    pods:
      metric:
        name: kafka_consumer_lag
      target:
        type: AverageValue
        averageValue: "100"
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 30  # Prevent flapping
    scaleDown:
      stabilizationWindowSeconds: 300  # Slow scale-down (5 min)

Production Example

Spotify: Runs 2000+ microservices on GKE (Google Kubernetes Engine). Each team owns their service's Helm chart. HPA scales based on custom Datadog metrics (p99 latency). A new service goes from code to production in under 15 minutes via fully automated CI/CD pipeline triggered by PR merge.

Trade-off

Managed K8s (EKS/GKE) vs Self-Managed: Managed removes control plane burden ($0.10/hr premium) but limits customization. Self-managed gives full control (custom schedulers, CNI plugins) but requires dedicated platform team (3-5 engineers). For 99% of companies: use managed K8s. Self-manage only if you're at Google/Netflix scale or have unique requirements.

Interview Tip

Mention K8s scaling naturally: 'Services deployed as K8s Deployments with HPA (target 70% CPU). Each pod has resource requests/limits to prevent noisy neighbors. Liveness and readiness probes ensure traffic only routes to healthy pods. Rolling deployments with maxSurge=25% for zero-downtime releases.'

4.14.2 Compute Options: EKS vs ECS vs Lambda

Cloud

Choosing where your containers/functions run is a core cloud decision — it trades operational control against managed simplicity and cost model.

→ CI/CD pipeline stages and deployment strategies (blue-green / canary / rolling, Argo Rollouts) are covered on the DevOps & Deployment page (4.23.4 and 4.23.5).

Trade-off: EKS vs ECS vs Lambda

ServiceComplexityControlCold StartCost Model
EKS (Kubernetes)High (k8s ops)FullNone (always running)Per node-hour
ECS (Fargate)MediumModerateNonePer vCPU-second
LambdaLowLimited100ms-10sPer invocation + duration
  • EKS: Full ecosystem (service mesh, operators, KEDA). Use when team has K8s expertise and needs advanced orchestration.
  • ECS Fargate: Simpler K8s. Use when you want containers without managing nodes.
  • Lambda: Event-driven, sporadic traffic, glue logic. Avoid for: steady high traffic (expensive), long-running tasks (>15min), large payloads.

Interview Tip

Default to managed containers, reserve serverless for spiky work: "Steady traffic → ECS Fargate or EKS (no cold starts, predictable cost). Spiky/event-driven → Lambda. We keep Lambda off the hot request path to avoid cold-start tail latency, using it for async jobs and glue."

4.14.3 IaaS / PaaS / SaaS

Foundation

What they are: Three layers of cloud abstraction that determine what you manage vs what the provider manages.

LayerYou ManageProvider ManagesExamplesUse Case
IaaS (Infrastructure as a Service)OS, Runtime, App, DataServers, Storage, Networking, VirtualizationAWS EC2, Azure VMs, GCP Compute EngineFull control needed, custom OS/config, lift-and-shift migration
PaaS (Platform as a Service)App, DataOS, Runtime, Middleware, ServersHeroku, AWS Elastic Beanstalk, Google App Engine, Azure App ServiceFocus on code, not infra. Web apps, APIs
SaaS (Software as a Service)Nothing (just use it)EverythingGmail, Slack, Salesforce, DatadogReady-made tools, no development needed
// Modern additions to the spectrum:
// FaaS (Function as a Service) — Serverless:
//   You manage: individual functions (code snippets)
//   Provider manages: everything including scaling, invocation
//   Examples: AWS Lambda, Azure Functions, Google Cloud Functions
//   Use case: Event-driven, sporadic workloads, pay-per-invocation

// CaaS (Container as a Service):
//   You manage: containers, app code
//   Provider manages: orchestration, OS, infra
//   Examples: AWS ECS/EKS, Google GKE, Azure AKS
//   Use case: Microservices, portable deployments

Trade-off

IaaS (EC2): maximum control, maximum responsibility (you patch the OS). PaaS (Heroku, App Runner): focus on code but limited customization. SaaS (Stripe, Auth0): zero development effort for solved problems but vendor lock-in and less control. Choose: IaaS for core differentiating services, PaaS for standard web services, SaaS for commoditized functions (auth, payments, email).

Production Example

Shopify: Runs core commerce platform on IaaS (custom K8s on GCP). Uses SaaS for non-core: Cloudflare (CDN/WAF), Datadog (observability), PagerDuty (alerting), Vault (secrets). Philosophy: build what differentiates you, buy everything else.

Interview Tip

Show pragmatism in your design: 'For authentication, I'd use Auth0 (SaaS) rather than building custom auth — it's a solved problem. For our core recommendation engine (business differentiator), we need full control so I'd run it on K8s (IaaS). This lets the team focus engineering effort on what makes our product unique.'

4.14.4 Containers & Docker

Foundation

What it does: Package an application with ALL its dependencies (runtime, libraries, config) into a portable, isolated unit that runs identically everywhere — dev laptop, CI server, production.

AspectVirtual MachineContainer
IsolationFull OS (hypervisor-level)Process-level (shared kernel)
SizeGBs (includes full OS)MBs (just app + libs)
StartupMinutesSeconds
Resource overheadHigh (each VM runs full OS)Low (shared kernel)
Density~10 VMs per host~100 containers per host
Security isolationStrong (hardware boundary)Weaker (kernel shared, namespace isolation)
Use caseMulti-tenant, different OS needsMicroservices, CI/CD, dev environments
// Dockerfile example (multi-stage build for Java Spring Boot):
// Stage 1: Build
FROM maven:3.9-eclipse-temurin-21 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline          // Cache dependencies
COPY src ./src
RUN mvn package -DskipTests            // Build JAR

// Stage 2: Run (slim image)
FROM eclipse-temurin:21-jre-alpine
COPY --from=builder /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]

// Result: ~150MB image (not 800MB with full JDK + Maven)

// Key Docker concepts for interviews:
// - Image: immutable template (built from Dockerfile)
// - Container: running instance of an image
// - Layer caching: unchanged layers reused → fast rebuilds
// - Multi-stage builds: separate build and runtime → smaller images
// - Docker Compose: multi-container local development
// - Registry: Docker Hub, ECR, GCR (store/distribute images)

Interview Tip

When discussing deployment in system design, mention containers as the default packaging mechanism: "Services are containerized with Docker, deployed to Kubernetes for orchestration." If asked about the difference from VMs, highlight: containers share the host kernel (faster, lighter) while VMs provide stronger isolation. For multi-tenant SaaS, you might use VMs for tenant isolation with containers inside each VM for service deployment.

Trade-off

Containers vs VMs vs Serverless: VMs provide strong isolation (separate kernels) but slow boot (minutes) and resource-heavy. Containers share kernel (faster boot in seconds, efficient) but weaker isolation. Serverless has zero management but cold starts and execution limits. Default to containers for general workloads; VMs for strict isolation (multi-tenant); serverless for event-driven, sporadic loads.

Production Example

Docker Hub: 13+ billion image pulls per month. Most enterprises standardize on 3-5 base images (company-approved, security-scanned) that teams extend. Multi-stage builds reduce production image size by 70-90% (build stage has compilers, production stage has only runtime). Distroless images (Google) eliminate shell/package managers — reduces attack surface.
