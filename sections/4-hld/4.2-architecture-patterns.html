4.2.1 Key Architecture Patterns

// CQRS: Separate write model (commands) from read model (queries)
// Write side: validates, applies business rules, publishes events
@Service
public class OrderCommandService {
    public void placeOrder(PlaceOrderCommand cmd) {
        Order order = Order.create(cmd);
        orderRepo.save(order);
        eventBus.publish(new OrderPlacedEvent(order));  // Async update read model
    }
}

// Read side: denormalized, optimized for queries
@Service
public class OrderQueryService {
    // Reads from materialized view (Redis/Elasticsearch) - fast!
    public OrderSummaryView getOrderSummary(String orderId) {
        return orderViewRepo.findById(orderId);  // Pre-computed, denormalized
    }
}
// Trade-off: Read model is eventually consistent with write model

Trade-off: When to Use Microservices

  • Start with Monolith when: Small team (<10), unclear domain boundaries, speed-to-market priority, startup phase
  • Move to Microservices when: Teams > 2 pizza size, deployment conflicts, different scaling needs per component, clear bounded contexts
  • Cost of Microservices: Network latency, distributed transactions, operational complexity (K8s, service mesh), data consistency challenges

Production Example

Uber CQRS: Write path ingests 1M+ trip GPS events/sec into Kafka → stored in Cassandra. Read path materializes into Elasticsearch for rider-facing ETA and driver-facing navigation. Different models optimized for their access patterns.

Key Tip

Don't just name a pattern — explain WHY you'd choose it for THIS system. "I'm choosing CQRS here because our read traffic is 100x write traffic, and reads need denormalized views that would be expensive to compute on every query." Pattern + justification = senior-level answer.

4.2.2 Service-Oriented Architecture (SOA)

Architecture

What it is: An architectural style where services communicate over a network via well-defined interfaces. Predates microservices — services are larger, share a common data bus (Enterprise Service Bus / ESB), and typically share databases.

AspectSOAMicroservices
Service sizeLarge, enterprise-level servicesSmall, single-responsibility
CommunicationESB (Enterprise Service Bus), SOAP/XMLREST/gRPC, lightweight messaging
DataOften shared databasesDatabase-per-service
GovernanceCentralized (ESB orchestrates)Decentralized (teams own services)
DeploymentCoordinated releasesIndependent deployment
Era2000s enterprise (IBM, Oracle)2010s cloud-native (Netflix, Uber)

When to use SOA: Legacy enterprise integration, connecting disparate systems (ERP, CRM, billing) that can't be rewritten.

When to choose Microservices instead: Greenfield projects, cloud-native, team autonomy needed, independent scaling required.

Trade-off: SOA vs Microservices

SOA with ESB is simpler to coordinate but creates a central bottleneck (ESB is SPOF). Microservices are more resilient but require each team to handle cross-cutting concerns (auth, logging, tracing) independently or via a service mesh.

Production Example

Banks & Insurance: Most large banks still run SOA — legacy COBOL systems wrapped in SOAP/XML services connected via IBM MQ or TIBCO ESB. Migration to microservices happens slowly (5-10 year programs) because replacing the ESB means rewiring thousands of integrations.

Key Tip

If asked "monolith vs microservices," mention SOA as the middle ground that enterprises used in the 2000s. Show awareness that architecture is a spectrum: Monolith → Modular Monolith → SOA → Microservices → Serverless. Each step adds operational complexity but gains team autonomy and independent scaling.

4.2.3 Event-Driven Architecture (EDA)

Architecture

What it is: Components communicate by producing and consuming events rather than direct calls. An event represents something that happened ("OrderPlaced", "PaymentReceived"). Producers don't know or care who consumes the event.

// Event-Driven vs Request-Driven comparison:

// REQUEST-DRIVEN (synchronous, coupled):
// OrderService → calls PaymentService → calls InventoryService → calls NotificationService
// Problem: if NotificationService is down, entire order fails!

// EVENT-DRIVEN (asynchronous, decoupled):
// OrderService publishes: "OrderPlaced" event
// PaymentService subscribes → processes payment → publishes "PaymentCompleted"
// InventoryService subscribes → reserves stock
// NotificationService subscribes → sends confirmation email
// Benefit: if NotificationService is down, order still succeeds!

// Spring Boot event example:
@Service
public class OrderService {
    private final ApplicationEventPublisher publisher;
    
    public Order placeOrder(OrderRequest req) {
        Order order = orderRepo.save(new Order(req));
        publisher.publishEvent(new OrderPlacedEvent(order)); // Fire and forget
        return order;
    }
}

@EventListener
public class InventoryHandler {
    @Async
    public void onOrderPlaced(OrderPlacedEvent event) {
        inventoryService.reserve(event.getItems());
    }
}
EDA PatternDescriptionUse Case
Event NotificationLightweight event, consumer fetches details"OrderPlaced {id: 123}" → consumer calls back for full data
Event-Carried State TransferEvent contains full data, no callback needed"OrderPlaced {id, items, total, address}" → consumer has everything
Event SourcingStore events as source of truth, derive stateAudit trails, undo, temporal queries

Production Example

Uber: When a rider requests a ride, the event "RideRequested" flows through Kafka to dozens of services independently — matching, pricing, driver notification, fraud detection, surge calculation, ETA estimation. No single orchestrator. Each service acts on the event autonomously. If fraud detection is slow, the ride still proceeds for most cases.

Trade-off: Event-Driven vs Request-Driven

Events decouple producers from consumers (add new consumers without changing producer). But: debugging is harder (no call stack, events flow through queues), ordering isn't guaranteed across topics, and eventual consistency means UI may show stale state briefly.

Key Tip

Use EDA whenever the interviewer says "multiple downstream services need to know about X." Say: "Rather than the order service calling 5 services synchronously, it publishes an OrderPlaced event. Each consumer processes independently — if notification service is slow, it doesn't block payment processing."

4.2.4 Event Sourcing

Pattern

What it is: Instead of storing current state (overwriting), store every state change as an immutable event. Current state is derived by replaying events from the beginning (or from a snapshot).

// Traditional CRUD (state-based):
// UPDATE accounts SET balance = 750 WHERE id = 'A'
// Problem: WHY is the balance 750? No history!

// Event Sourcing:
// Event 1: AccountCreated { id: 'A', balance: 1000 }
// Event 2: MoneyWithdrawn { id: 'A', amount: 200 }
// Event 3: MoneyDeposited { id: 'A', amount: 50 }
// Event 4: MoneyWithdrawn { id: 'A', amount: 100 }
// Current state: replay all → balance = 1000 - 200 + 50 - 100 = 750
// Benefit: FULL audit trail, can answer "what was balance at Event 2?" → 800

// Snapshot optimization (don't replay from beginning every time):
// After every 100 events, save snapshot: { balance: 750, version: 100 }
// New reads: load snapshot + replay only events after version 100
AdvantageDisadvantage
Complete audit trail (compliance, debugging)Storage grows unbounded (need snapshots/compaction)
Temporal queries ("balance at time T")Complex reads (need projections/CQRS)
Easy to add new projections/viewsEvent schema evolution is hard (versioning)
Natural fit for distributed systems (event = WAL)Eventual consistency for read models

Production Example

Banking/Finance: Every transaction is an event. Regulators require full audit trails. LMAX Exchange processes 6 million financial events/second using event sourcing. Git is event sourcing — each commit is an event; current file state = replay all commits.

Trade-off: Event Sourcing vs CRUD

Event Sourcing gives complete audit trail and temporal queries but makes simple reads complex (must replay or maintain projections). CRUD is simpler for basic apps but loses history (you only see current state). Use Event Sourcing when audit/compliance matters or you need "time travel" queries.

Key Tip

Mention Event Sourcing for: payment systems (audit trail), collaborative editors (undo/redo), and any system where "why did this happen?" matters. Pair it with CQRS: "Events are the write model, projections are the read model — optimized for their respective access patterns."

4.2.5 Serverless Architecture

Architecture

What it is: Deploy individual functions (not servers). The cloud provider handles all infrastructure, scaling, and invocation. You pay per execution, not per hour. Scale to zero when idle.

AspectServerless (FaaS)Containers (K8s)VMs (EC2)
Scale unitSingle function invocationPod (1+ containers)Entire VM
ScalingAutomatic, per-requestHPA/VPA (seconds-minutes)Manual or ASG (minutes)
Cold start100ms-10s (language dependent)Seconds (pod scheduling)Minutes
Cost modelPay per invocation + durationPay for provisioned podsPay for running VMs
Max runtime15 min (Lambda limit)UnlimitedUnlimited
StateStateless (ephemeral)Stateless or statefulAny

When to use: Event-driven workloads (S3 upload triggers processing), APIs with variable traffic, scheduled tasks (cron), webhook handlers, image/video processing.

When NOT to use: Long-running processes (>15min), latency-sensitive (cold starts), high-throughput steady-state (cheaper to run containers), stateful applications.

Production Example

Netflix: Uses Lambda for encoding validation — when a video is uploaded to S3, a Lambda triggers to check format, resolution, and metadata before expensive transcoding begins. Idle most of the time, bursts to thousands of concurrent invocations during content drops.

Trade-off: Serverless vs Containers

Serverless: zero ops, auto-scales to zero (cost-effective for sporadic traffic), but cold starts add latency and max execution time is limited (15min Lambda). Containers: more control, no cold starts, unlimited runtime, but you manage scaling and pay for idle capacity.

Key Tip

Use serverless in designs for: webhook handlers, image processing triggers, scheduled cleanup jobs, and low-traffic APIs. Say: "For the thumbnail generation service, I'd use Lambda triggered by S3 upload — it's event-driven, stateless, and we pay nothing when no uploads are happening."

4.2.6 Layered Architecture (N-Tier)

Architecture

What it is: Organize code into horizontal layers where each layer only depends on the layer directly below it. The most common architecture for traditional web applications.

// Classic 3-Tier:
// ┌─────────────────────────┐
// │   Presentation Layer     │  (Controllers, REST endpoints, UI)
// ├─────────────────────────┤
// │   Business Logic Layer   │  (Services, domain logic, validation)
// ├─────────────────────────┤
// │   Data Access Layer      │  (Repositories, SQL, ORM)
// └─────────────────────────┘

// Spring Boot maps naturally to 3-tier:
// @RestController → Presentation (handles HTTP, serialization)
// @Service → Business Logic (rules, orchestration)
// @Repository → Data Access (JPA, SQL queries)

// Rule: Each layer only calls the layer below
// Controller → Service → Repository ✓
// Controller → Repository ✗ (skip layer = coupling)
// Repository → Service ✗ (upward dependency = circular)
AdvantageDisadvantage
Simple mental model, easy onboardingChanges often cascade through all layers
Clear separation of concernsCan lead to "anemic domain model" (logic in services, not domain)
Each layer independently testableHorizontal slicing makes feature changes touch many layers
Industry standard, well-understoodDoesn't prevent tight coupling within layers

When to use: CRUD applications, simple web apps, team learning curve concerns, monolith applications.

When to consider alternatives: Complex domain logic (→ Hexagonal/DDD), microservices (→ vertical slicing per service), event-heavy systems (→ EDA).

Trade-off: Layered vs Hexagonal vs Vertical Slicing

Layered is simple but changes cascade vertically. Hexagonal isolates business logic from infrastructure (better testability) but adds abstraction overhead. Vertical slicing (feature-based) keeps related code together but may duplicate infrastructure patterns across slices.

Production Example

Most Spring Boot apps: The default Spring Boot project structure IS layered architecture — @Controller → @Service → @Repository. Works well for CRUD apps up to medium complexity. Companies like Shopify started layered and migrated to modular monolith (vertical slices) as complexity grew.

Key Tip

For LLD questions, default to layered architecture (it's expected). For HLD, layered applies within each microservice. Say: "Each service internally follows a 3-layer structure (API → Domain → Persistence), while the overall system is microservices communicating via events."
