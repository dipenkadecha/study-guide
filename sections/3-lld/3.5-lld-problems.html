3.5LLD Problems

3.5.1 Algorithms Comparison

Classic
// Token Bucket - most common in production (AWS, Stripe use this)
public class TokenBucketRateLimiter {
    private final int maxTokens;
    private final int refillRate; // tokens per second
    private double availableTokens;
    private long lastRefillTimestamp;

    public synchronized boolean allowRequest() {
        refill();
        if (availableTokens >= 1) {
            availableTokens--;
            return true;
        }
        return false;
    }

    private void refill() {
        long now = System.nanoTime();
        double elapsed = (now - lastRefillTimestamp) / 1_000_000_000.0;
        availableTokens = Math.min(maxTokens, availableTokens + elapsed * refillRate);
        lastRefillTimestamp = now;
    }
}

// Sliding Window Log - precise but memory-heavy
public class SlidingWindowRateLimiter {
    private final TreeMap<Long, Integer> requestLog = new TreeMap<>();
    private final int maxRequests;
    private final Duration window;

    public synchronized boolean allowRequest() {
        long now = System.currentTimeMillis();
        long windowStart = now - window.toMillis();
        requestLog.headMap(windowStart).clear();  // Evict old entries

        int currentCount = requestLog.values().stream().mapToInt(i -> i).sum();
        if (currentCount < maxRequests) {
            requestLog.merge(now, 1, Integer::sum);
            return true;
        }
        return false;
    }
}

Production Insight

// LRU Cache - O(1) get and put using HashMap + Doubly Linked List
public class LRUCache<K, V> {
    private final int capacity;
    private final Map<K, Node<K, V>> map;
    private final Node<K, V> head, tail; // Sentinel nodes

    public LRUCache(int capacity) {
        this.capacity = capacity;
        this.map = new HashMap<>((int)(capacity / 0.75) + 1);
        head = new Node<>(null, null);
        tail = new Node<>(null, null);
        head.next = tail;
        tail.prev = head;
    }

    public V get(K key) {
        Node<K, V> node = map.get(key);
        if (node == null) return null;
        moveToHead(node);  // Mark as recently used
        return node.value;
    }

    public void put(K key, V value) {
        Node<K, V> existing = map.get(key);
        if (existing != null) {
            existing.value = value;
            moveToHead(existing);
        } else {
            Node<K, V> newNode = new Node<>(key, value);
            map.put(key, newNode);
            addToHead(newNode);
            if (map.size() > capacity) {
                Node<K, V> evicted = removeTail();
                map.remove(evicted.key);
            }
        }
    }
    // Thread-safe version: use ConcurrentHashMap + ReadWriteLock on list operations
}

In production, use Caffeine (Java's fastest caching library) over manual LRU. It uses W-TinyLFU eviction which outperforms LRU by ~30% hit rate. Interview wants you to implement LRU manually; production wants you to know Caffeine exists.

3.5.2 Walkthrough: Design a Parking Lot

LLD Problem

The classic object-modeling round. The interviewer scores how you get to the classes, not the code volume. Drive it in four steps.

1. Clarify requirements & scope

  • Functional: multiple levels; spot types (compact / large / motorcycle / EV); a vehicle takes the smallest spot that fits; issue a ticket on entry; compute fee on exit; show free-spot count per level.
  • Non-functional: thread-safe spot assignment (two cars must not get the same spot); pluggable pricing; extensible to new vehicle/spot types.
  • Out of scope (state it): payments gateway, reservations, license-plate OCR.

2. Identify entities (nouns → classes)

enum VehicleType { MOTORCYCLE, CAR, TRUCK, EV }
enum SpotType    { MOTORCYCLE, COMPACT, LARGE, EV }

abstract class Vehicle { final String plate; final VehicleType type; }

class ParkingSpot {
    final String id; final SpotType type;
    private Vehicle vehicle;                 // null = free
    boolean isFree() { return vehicle == null; }
    boolean fits(Vehicle v) { return isFree() && SpotFit.allows(type, v.type); }
}

class Level {                                // one floor
    final int floor;
    final List<ParkingSpot> spots;
    Optional<ParkingSpot> findSpot(Vehicle v) { ... }
}

class Ticket {                               // issued at entry
    final String id; final String plate;
    final String spotId; final Instant entryTime;
}

3. Behavior & concurrency

class ParkingLot {
    private final List<Level> levels;
    private final PricingStrategy pricing;   // Strategy: hourly, flat, tiered
    private final Map<String, Ticket> active = new ConcurrentHashMap<>();

    // Synchronize the find+occupy so two threads can't grab one spot.
    synchronized Ticket parkVehicle(Vehicle v) {
        for (Level lvl : levels) {
            Optional<ParkingSpot> s = lvl.findSpot(v);
            if (s.isPresent()) {
                s.get().assign(v);
                Ticket t = new Ticket(UUID.randomUUID().toString(),
                                      v.plate, s.get().id, Instant.now());
                active.put(t.id, t);
                return t;
            }
        }
        throw new ParkingFullException(v.type);
    }

    Money exit(String ticketId) {
        Ticket t = active.remove(ticketId);
        if (t == null) throw new InvalidTicketException(ticketId);
        freeSpot(t.spotId);
        return pricing.price(t.entryTime, Instant.now(), t /* spot type */);
    }
}

4. Call out the design choices

What scores points

  • Strategy for PricingStrategy so fee rules change without touching the lot — name it.
  • Concurrency: the find-then-occupy must be atomic; coarse synchronized is fine to start, then offer per-level locks or an optimistic CAS on the spot to reduce contention as a scaling follow-up.
  • Open/Closed: adding an EV charging spot = new enum value + a SpotFit rule, no changes to parkVehicle.
  • End with extension hooks: multiple entrances, a display board (Observer on free-count), reservations.
