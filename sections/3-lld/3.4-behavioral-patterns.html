3.4Behavioral Design Patterns

3.4.1 Strategy - Interchangeable Algorithms

Patterns
// Strategy: select algorithm at runtime
// Use case: pricing engines, compression, serialization, sorting strategies
public interface PricingStrategy {
    BigDecimal calculatePrice(Product product, Customer customer);
}

@Component("premium")
public class PremiumPricingStrategy implements PricingStrategy {
    public BigDecimal calculatePrice(Product p, Customer c) {
        return p.basePrice().multiply(new BigDecimal("0.85")); // 15% discount
    }
}

@Component("bulk")
public class BulkPricingStrategy implements PricingStrategy {
    public BigDecimal calculatePrice(Product p, Customer c) {
        int qty = c.orderQuantity();
        BigDecimal discount = qty > 100 ? new BigDecimal("0.70") : new BigDecimal("0.90");
        return p.basePrice().multiply(discount);
    }
}

3.4.2 Observer / Event-Driven (Spring Events)

Patterns
// Spring ApplicationEvent - decoupled communication
public record OrderCompletedEvent(String orderId, String customerId, BigDecimal total) {}

@Component
public class OrderService {
    private final ApplicationEventPublisher publisher;

    @Transactional
    public void completeOrder(String orderId) {
        Order order = updateStatus(orderId, COMPLETED);
        publisher.publishEvent(new OrderCompletedEvent(
            order.id(), order.customerId(), order.total()));
    }
}

@Component
public class LoyaltyPointsListener {
    @EventListener
    @Async  // Non-blocking - runs in separate thread
    public void onOrderCompleted(OrderCompletedEvent event) {
        loyaltyService.addPoints(event.customerId(), calculatePoints(event.total()));
    }
}
// Multiple listeners react independently: email, analytics, loyalty, inventory

3.4.3 Chain of Responsibility

Patterns
// Chain: pass request along handlers until one handles it
// Production use: Spring Security Filter Chain, validation pipelines
public interface ValidationHandler {
    ValidationResult validate(Order order);
    void setNext(ValidationHandler next);
}

public class StockValidationHandler implements ValidationHandler {
    private ValidationHandler next;
    public ValidationResult validate(Order order) {
        if (!inventoryService.hasStock(order.items()))
            return ValidationResult.failure("Insufficient stock");
        return next != null ? next.validate(order) : ValidationResult.success();
    }
}
// Chain: Stock -> FraudCheck -> CreditLimit -> AddressValidation

3.4.4 State - Behavior Driven by Internal State

Pattern
// Intent: an object changes its behavior when its state changes - replaces
// sprawling if/switch on a status field. Classic LLD asks: vending machine,
// order lifecycle, TCP connection, document workflow.
public interface OrderState {
    OrderState pay(OrderContext ctx);
    OrderState ship(OrderContext ctx);
    OrderState cancel(OrderContext ctx);
}

public class CreatedState implements OrderState {
    public OrderState pay(OrderContext c)    { c.capturePayment(); return new PaidState(); }
    public OrderState ship(OrderContext c)   { throw new IllegalStateException("not paid"); }
    public OrderState cancel(OrderContext c) { return new CancelledState(); }
}
public class PaidState implements OrderState {
    public OrderState pay(OrderContext c)    { throw new IllegalStateException("already paid"); }
    public OrderState ship(OrderContext c)   { c.createShipment(); return new ShippedState(); }
    public OrderState cancel(OrderContext c) { c.refund(); return new CancelledState(); }
}
// Context delegates to the current state; each state knows its legal transitions.
public class OrderContext {
    private OrderState state = new CreatedState();
    public void pay()    { state = state.pay(this); }
    public void ship()   { state = state.ship(this); }
    public void cancel() { state = state.cancel(this); }
}

State vs Strategy

Same structure (delegate to a pluggable object), different intent. Strategy is chosen by the client and is usually stateless/independent. State transitions itself — each state decides the next state — and the object's behavior changes over its lifetime. Mention that illegal transitions become explicit exceptions instead of silent bugs.

3.4.5 Command - Encapsulate a Request as an Object

Pattern
// Intent: wrap an action + its data in an object so you can queue, log,
// schedule, or undo it. Use cases: undo/redo, task queues, transactional
// operations, GUI actions, the Memento+Command undo stack.
public interface Command {
    void execute();
    void undo();
}

public class AddTextCommand implements Command {
    private final Document doc;
    private final String text;
    public AddTextCommand(Document doc, String text) { this.doc = doc; this.text = text; }
    public void execute() { doc.append(text); }
    public void undo()    { doc.deleteLast(text.length()); }
}

// Invoker keeps history; doesn't know what each command actually does.
public class CommandInvoker {
    private final Deque<Command> history = new ArrayDeque<>();
    public void run(Command c) { c.execute(); history.push(c); }
    public void undo()         { if (!history.isEmpty()) history.pop().undo(); }
}

Key Tip

Command decouples the invoker (button, scheduler, queue) from the receiver (the object doing the work). The undo stack falls out naturally. If asked for redo, keep a second stack. This is the go-to answer for "design undo/redo" or "design a task scheduler".
